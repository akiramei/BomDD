# BomDD 解説書 設計稿 — 第1工程から第4工程

> **この文書の位置づけ**  
> これは解説書本文ではない。想定読者の認識を、既存の「AIに実装を頼み、コードとテストをレビューする」という開発観から、BomDD の「入力・製造・検査・保守を分離し、AIによる補完を測る」という開発観へ移すための設計稿である。第5工程の本文と第6工程の理解テストは、まだ書かない。

## この設計稿で使う証拠ラベル

本文で事実と構想を混同しないため、以下のラベルを使う。

- **[定義]**: BomDD の正規文書が定めている方法上の規則。定義されていることは、その効果があらゆる案件で証明済みという意味ではない。
- **[観測]**: リポジトリに実験記録または再現手順がある結果。観測した題材、工場、オラクル被覆の範囲を越えて一般化しない。
- **[比較設計]**: 読者理解のために構成した通常AI開発側の反実仮想。実験で通常手法と無作為比較した結果ではない。
- **[仮説]**: 観測を踏まえた作業仮説、候補規則、または一般化が未完了の処方。
- **[適用範囲外]**: 現在の証拠からは答えを出さない事項。

主要な根拠は、[README](../README.md)、[WHITEPAPER](../WHITEPAPER.md)、[concept](../docs/concept.md)、[terminology](../docs/terminology.md)、[FINDINGS](../FINDINGS.md)、[BomDD Method v1](../method/bomdd-method-v1.md)、[BomDD Playbook v1](../method/bomdd-playbook-v1.md)、各 [Loop 報告](../loops/loop-01-report.md) である。

# 第1工程：読者分析

## 1. 想定読者の出発点

想定読者は、要求、設計書、C#/.NET、単体テスト、Git、CI/CD、DDD を、相互に接続はするが別々の活動として理解している。AIについては、おそらく次のどちらかの経験を持つ。

1. AIを実装を速めるペアプログラマとして使い、人間がプロンプト、差分、テスト結果をレビューする。
2. 仕様書やissueをAIに渡し、リポジトリ全体を読ませて変更させる。

この読者に製造業の語彙を先に示すと、未知語の定義を覚える一方で、既存の開発観は変わらない。したがって説明は「BOMとは何か」ではなく、「コードもテストも緑なのに、AI開発の再現性を評価できない失敗」から始める必要がある。

## 3. 「既存手法の言い換えではないか」の分析

この疑問は退けるべき反発ではなく、解説書が正面から答えるべき中心質問である。実際、BomDDの構成要素の多くは既存である。

- BOM、FMEA、PLM、ECOは製造業・品質管理に既存である。
- 要求と設計・テストの双方向トレースは要求工学に既存である。
- ドメインモデルの明確化はDDDで扱える。
- 仕様を正本にすることはドキュメント駆動でも扱える。
- モデルから実装を生成することはMDDでも扱える。
- テスト先行、CI、provenance、依存監視にも既存の実践がある。

したがって「BOMという語を使うこと」や「文書を細分化すること」を新規性として説明してはいけない。[WHITEPAPER](../WHITEPAPER.md) が置く識別点は、次の四つを一つの製造・検査ループとして結合することにある。

1. 設計対話や正解を見せない**隔離AI工場**
2. 合否を変えない**固定オラクル**と、未規定面を観測する**探索プローブ**の分離
3. BOMにない暗黙の補完を申告する**ずる報告**
4. 複数工場の差を、**BOM欠落・工場能力・意図的未規定面**へ帰属すること

ただし、ここでも過大主張を避ける。既存手法を十分に組み合わせれば、同等の統制を設計することは原理上可能である。「BomDDでしか実現不能」なのではなく、これらの統制を共通語彙、成果物、停止条件、測定値を持つ反復可能な工程として束ねたものを、この研究ではBomDDと呼ぶ。従って比較で問うべきは「各部品が新しいか」ではなく、次である。

- AIが何を根拠に補完したかを、工程上観測できるか。
- 合格と探索を混ぜず、同じ物差しで改善前後を比較できるか。
- 修正が一回限りの会話やソース差分ではなく、次のfreshな工場へ転移する入力に宿ったか。
- 稼働後の変更で、影響ありだけでなく影響なし予測も反証できるか。

この整理はP5で明示する。ただし読者が四つの装置をまだ理解していない冒頭で提示すると単なる機能一覧になるため、P0〜P4の失敗事例と工程を先に置く。

## 4. 読者が抱くべき健全な疑問

解説書は、次の疑問を“誤解”として消してはならない。P7〜P8で、証拠の限界として残す。

- 隔離しても、基盤モデルが事前学習したC#/.NET慣習までは隔離できないのではないか。
- ずる報告に現れない補完をどう検出するのか。
- 固定オラクル自体が誤っていたら、誤りを工業的に複製するだけではないか。
- E/M/S/K-BOMの保守コストが、回避できた手戻りを上回らないか。
- 大規模な既存システムで、論理部品から実装ファイルへの写像を維持できるか。
- 人間チーム、他ベンダー、長期運用、規制産業でも同じ結果になるか。
- DDD、要求トレース、テスト戦略を成熟運用している組織に追加導入する限界利益はどれほどか。

# 第2工程：主張の明確化

## 1. 中心的な主張を一文で定義する

> **BomDDとは、設計意図、製造に必要な外部知識、製造・受入条件、稼働後の保守影響を別のBOMとして管理し、その入力だけを渡された隔離AI工場による製造を、固定オラクル、探索プローブ、ずる報告、工場間差分で検査し、変更をECOから再び同じループへ通す、AI支援開発の製造・検査・保守方法である。** [定義]

短く言い直すなら、**「AIにうまく書かせる方法」ではなく、「AIが何を根拠に作り、どこを勝手に補い、同じ管理入力から別の工場でも受入可能な品を作れるかを測る方法」**である。

## 2. 解決しようとする問題を、AI開発の具体的な失敗として示す

### 失敗像

図書貸出APIに「standard会員は3冊、premium会員は5冊まで借りられる」という変更を加える。既存の会員テーブルには区分列がない。依頼者はAIにissueとリポジトリを渡す。

AIは、列挙型、上限判定、API入力、DB列、テストを追加する。新規DBでテストは緑、レビュー上もコードは自然である。しかし本番相当の旧DBでは、次のいずれかが起こり得る。

- 既存行の会員区分がnullになり、読み込み時に例外となる。
- AIが暗黙に既存会員をpremiumと解釈し、既存挙動を拡張してしまう。
- DBを作り直せば動くが、既存データを保持した自動移行がない。
- 変更と無関係なAPIエラー形式まで“整理”され、既存クライアントが壊れる。
- AIは同じ依頼への次回変更で、以前の会話にしかない判断を再現できない。

問題は単にテスト不足ではない。どの判断が要求由来で、どれがSQLite/.NETの慣習知識由来で、何を不変とし、旧個体で何を失敗させて検査器を較正し、どの範囲だけを変えるかが、製造前に別々の管理対象になっていないことである。

BomDDは、この失敗を次の因果で扱う。

1. 「既存会員はstandard」「移行は自動」「既存貸出・API契約は不変」を要求と仕様にする。
2. 要求から導ける会員区分・上限規則をE-BOMに、SQLite移行知識をK-BOMに、改修・受入単位をM-BOMに分ける。
3. 変更前個体、既存オラクル、旧DB fixtureを凍結する。
4. ECOで影響ありと影響なしを製造前に宣言し、新規受入と移行オラクルを先に作る。
5. 設計会話とオラクル詳細を見ないfresh工場に、改訂BOMとwork orderを渡す。
6. 既存回帰、新規上限、旧DB移行、影響外diffを別々に採点する。
7. 結果をAs-BuiltとS-BOMへ戻し、次の変更の基準点にする。

## 3. BomDD固有の解決部分と、既存手法でも解決可能な部分

### 既存手法でも解決可能な部分

| 問題 | 既存手法で可能な対策 |
|---|---|
| 会員区分と貸出上限の業務ルールを整理する | DDD、要求定義、業務ルール表 |
| 旧DBを移行する | DBマイグレーション設計、expand/contract、移行テスト |
| 回帰を防ぐ | 単体・統合・E2Eテスト、CI |
| 要求からテストまで辿る | 要求トレーサビリティ |
| 変更理由と差分を残す | issue、ADR、PR、Git履歴 |
| 依存脆弱性を監視する | 一般的なSBOM、Dependabot等の依存監視 |
| モデルからコードを生成する | MDD、DSL、コードジェネレータ |

### BomDDとして識別される解決部分

個別能力ではなく、次の結合が識別点である。[定義]

- 製造者から設計対話・正解・オラクル詳細を隠し、**管理入力の自己完結性を試験可能にする**。
- AIが管理入力以外から埋めた判断を**ずる**として記録し、要求、K-BOM、工程、検査、S-BOMのどこで管理するかを決める。
- 同じBOMを複数工場へ渡し、出力差を単なる“AIの精度差”ではなく、**未規定残渣、仕様取りこぼし、探索的未規定面**へ帰属する。
- 合否の物差しを固定し、未規定面の探索を別にすることで、BOM補正の効果を同じ基準で比較する。
- 変更時に影響ありだけでなく**影響なし予測**を凍結し、実diffでunder/over-inclusionを採点する。
- 一度のソース修正で終えず、改訂された管理入力からfresh工場が同じ修正を再現できることで、**修正が会話でなくBOMに宿ったか**を試す。

厳密には、既存手法を組み合わせて上記を全て行うことも可能である。本資料には「BomDDだけがこの問題を解ける」という比較実験上の根拠はない。BomDDでなければ解決できないのは能力そのものではなく、**この一式を省略せずBomDDとして適合したと呼ぶこと**である。これは名称の循環定義にならないよう、第5工程では観測可能な最低条件を示す必要がある。

## 4. BomDDは何であり、何ではないか

| BomDDは何であるか | BomDDは何ではないか |
|---|---|
| AI製造を測定可能にする工程 | よいプロンプトの書き方集 |
| 要求→E/K/M-BOM→製造→受入→S-BOM→ECOの閉ループ | BOMファイルを作れば完了する文書管理 |
| 決定性と正しさを別々に測る方法 | 複数AIの多数決 |
| 暗黙の補完を失敗データとして扱う方法 | ずるを完全にゼロにする方法 |
| 核と表面で知識源・検査方法を分ける作業仮説 | すべてを単体テスト可能にする主張 |
| AI出力を交換・再製造可能な単位として扱う方法 | 同じソースをCIで再ビルドするだけの再現性 |
| 人間の裁定と承認を工程上明示する方法 | 無人・無責任な自動開発 |
| 一般的SBOMを含み得るService BOM | NuGet/OSS依存一覧そのもの |
| 変更をECOとして再入口へ戻す継続的な方法 | 変更を嫌う一括前工程、またはウォーターフォールの別名 |
| DDD、テスト、Git、CI/CDと併用する上位の製造統制 | それらの代替 |

## 5. 四つのBOMを分ける中心理由

分離理由は帳票の都合ではなく、**異なる問い、権威主体、変化速度、合否条件を混ぜないため**である。

| 層 | 答える問い | 変更案件で記録するもの | 混ぜた場合の典型的な誤り |
|---|---|---|---|
| **E-BOM**（Engineering/設計BOM） | 何が、なぜ存在するか | 会員区分、貸出上限規則、既存会員の既定区分、部品間の依存・consumer | SQLiteの都合を業務要求と誤認する。実装交換時に要求まで変わったように見える |
| **K-BOM**（Knowledge BOM） | 製造にどの外部・設計知識が必要か | SQLiteの安全な列追加、既定値、`PRAGMA user_version`等の採用知識と版 | AIが知っている慣習を要求の一部と誤認し、モデルや外部仕様変更で再現不能になる |
| **M-BOM**（Manufacturing/製造BOM） | どの単位を、どの契約と受入条件で作るか | Core規則unit、永続化unit、API配線unit、Control Plan、FMEA、routing | 論理部品とファイル構成を同一視し、部分再製造も単独受入もできない |
| **S-BOM**（Service/保守BOM） | 稼働個体の何が何に依存し、変化時に何を再検査・交換するか | as-built参照、SQLite/AIモデル/外部知識の版、再検査するCP、逆引きキー | 依存更新時に全体再テストするか何も見ないかの二択になり、影響を絞れない |

ECOは第五のBOMではない。変更の入口であり、変更前baseline、要求、影響あり/なし予測、各BOMの改訂、先行受入、部分再製造、結果記録を束ねる伝票である。

## 6. 主張の証拠レベル

### [観測] 現在の資料から比較的強く言えること

- MoviePadの純粋ロジック、ffmpeg外部連携、UI知覚の3領域で、要求から導出しやすい核と、外部・設計知識を要する表面の違いが観測された。初期7ループで記録した12件のずるは表面側に現れた。
- Loop 1から1.5では、受入基準の完全性・無矛盾性等の補正後、製造装置のギャップ報告が16件から10件、発散を生んだギャップが2件から0件、従来手法介入が1回から0回になった。ただし残る暗黙知はゼロではない。
- ffmpeg連携では文字列一致が妥当な様式差を誤検出し、意味の違いを見逃した。L2メタデータより深いL3信号比較が音声位置バグを捕捉した。
- 同一BOMで複数工場が一致しても、正しさは保証されない。決定性と受入上の正しさは別軸である。
- Web/APIと分散Sagaを含む3題材で、仕様化済み契約と未規定面を分けて観測する方法が再現された。ただし“全ソフトウェアの一般法則”ではない。
- 図書貸出APIのECO-001では、freshな2工場とも既存回帰23/23、変更2/2、移行4/4、影響外diff 0だった。これは会員区分変更が、観測した1題材・1変更・当該オラクル被覆で転移した証拠である。
- 非自明な期限規則ECOでは、「API層diffゼロ」という影響なし予測が配線1行で反証された。予測を先に凍結したため、影響分析の取りこぼしを1行単位で測定できた。
- 部品数十規模の単一実題材を遡及採点すると、実装のハブunitへの写像漏れが系統的に観測された。BOMグラフだけで影響分析が完全になるわけではない。

### [仮説] まだ一般化してはいけないこと

- 核/表面の分離が、組込・リアルタイム・機械学習・大規模分散システムを含む全ソフトウェアで同じように有効であること。
- 最適なBOM粒度。細かすぎれば実装の転記になり、粗すぎれば部分再製造と影響分析が効かない。
- multi-factoryの決定性と最終品質の一般的な相関。
- 長期のAIモデル更新、外部仕様更新、担当者交代に対するPLM運用コストと効果。
- フォワード・モードの結果が、異なる組織・人間チーム・大規模案件でも同程度に再現すること。
- 規模別テーラリングとリスク等級の最適なしきい値。

### [適用範囲外] 現時点で答えないこと

- 顧客の本当の要求を正しく引き出せること。図書貸出APIの検証ではPO役を設計者が代行した。
- 事業価値、UXの良さ、法的適合性、セキュリティリスク受容をAIが自動決定できること。
- BomDDの導入ROIが、全組織で正になること。
- 安全性・認証・規制上、BomDD成果物だけで説明責任を満たせること。
- “現固定オラクル被覆で差分ゼロ”を、欠陥ゼロや完全仕様と同一視すること。

# 第3工程：比較事例の設計

## 1. 全工程を通して使う変更案件

既に稼働しているC#/.NET図書貸出APIへ、次の変更を行う。

> **変更要求**: 会員を `standard` と `premium` に分け、同時貸出上限をそれぞれ3冊と5冊にする。既存会員は全員 `standard` とみなす。稼働中のSQLiteデータを失わず、停止時のDB再作成を要求せず、自動移行する。従来の貸出、返却、エラー契約は、この変更に関係しない限り維持する。

この案件を選ぶ理由は、単なる条件分岐追加では結果差が出ないからである。業務規則、永続スキーマ、既存データの効力、API配線、回帰範囲が同時に関わり、AIが“新規環境では動く”実装を作っても“稼働個体を保守できる”とは限らない。また、BomDDのforward-01.5で同型案件の観測値がある。

## 2. A. 一般的なAI支援開発での進行

以下は理解用の **[比較設計]** であり、BomDDと通常開発を統制比較した実験結果ではない。

1. 開発者がissueに変更要求を書く。
2. AIにリポジトリ全体とissueを読ませ、「実装してテストを追加して」と依頼する。
3. AIは既存コードから実装慣習を推測し、`MemberType`、上限判定、DB列、API DTO、テストを変更する。
4. AIまたは開発者が、新規作成したDBで単体・統合テストを実行する。
5. 緑ならPRを作り、人間がdiffをレビューする。
6. レビュー指摘があれば同じ会話でAIへ修正を頼む。
7. CIが通ればマージし、デプロイ時にマイグレーションを実行する。

この進行でも、熟練チームなら旧DB fixture、契約テスト、移行テスト、ADR、影響分析を追加できる。問題は“通常開発は必ず失敗する”ことではない。標準進行のままでは、次の問いへの答えが工程上必須ではないことである。

- 既存会員をstandardとする判断は要求か、AIの推測か。
- SQLite移行方式は業務仕様か、交換可能な製造知識か。
- 既存APIのどの挙動が不変か。
- 新しいテストが本当に変更を検出できるか。変更前個体では赤になるか。
- AIが依頼にないファイルを“整理”した場合、それを不要改変として数えるか。
- 修正が会話履歴に宿ったのか、次のfresh AIへ渡せる管理入力に宿ったのか。

## 3. B. BomDDでの進行

### B1. ECOを起票し、変更前個体を凍結する

- ECOに変更要求、発生契機、変更前tag/commit、As-Maintained個体を記録する。
- 既存固定オラクルS01〜S23を不変の回帰基準として凍結する。
- 実v0.2相当のSQLite DBと期待値manifestを移行fixtureにする。

ここでECOはissueの装飾ではなく、後続する予測と検査の基準点になる。

### B2. 要求とE-BOMを改訂する

- REQに会員区分、上限3/5、既存会員の既定値standard、自動移行、既存契約不変を記録する。
- E-BOMに、たとえば次の論理部品を追加・改訂する。
  - `E-MEMBER-TYPE`: 会員区分。存在理由は新REQ。
  - `E-LOAN-LIMIT-POLICY`: 区分別上限規則。`E-MEMBER-TYPE`をconsumerとして参照。
  - `E-MEMBER-PERSISTENCE`: 会員区分を保持する責務。既存データの意味を維持する不変条件を持つ。
- 「standardをどうDBへ追加するか」はE-BOMへ書き込まない。それは要求の意味ではなく製造知識だからである。

### B3. K-BOMを改訂する

- SQLiteで既存テーブルへ非null列を安全に追加する知識を管理する。
- 例として、既定値付きの列追加、`PRAGMA user_version`を使う版管理、DROP/CREATEで既存データを失わないことを記録する。
- その知識の出所、版、暗黙のままにした場合のリスクを持たせる。

AIが同じ知識を既に持っていてもK-BOMは不要にならない。目的はAIに教えることだけでなく、どの外部知識に製造結果が依存したかを保守可能にすることだからである。

### B4. M-BOM、FMEA、Control Planを改訂する

- 製造・受入単位を、Coreの上限規則、永続化移行、API配線などに分ける。
- 各unitに公開契約、成果物範囲、受入基準を割り当てる。
- FMEAに少なくとも次を置く。
  - 既存会員がnullまたはpremiumになる。
  - 旧DBの貸出履歴が消える。
  - standardの4冊目、premiumの6冊目を拒否できない。
  - 新規則の導入で既存返却やエラー契約が変わる。
- Control Planに、上限境界のunit/black-box検査、旧DB移行のL2/L3相当検査、既存契約の回帰、承認者を割り付ける。

### B5. 影響ありと影響なしを製造前に凍結する

- 影響あり: 上限規則、会員永続化、会員作成/読込配線、追加オラクル、移行オラクル。
- 影響なし: 返却、既存エラー語彙、変更と無関係なAPI応答、非対象ファイル群。各項目に根拠を書く。
- As-Maintained実物で参照サイトを検索し、E-BOM品目からファイルへの写像漏れを点検する。

「影響なし」は期待ではなく、後で実diffにより反証される予測である。

### B6. オラクルを先に追加し、較正する

- 既存S01〜S23は変更しない。
- 変更受入S24〜S25として、standard 3冊、premium 5冊の境界を追加する。
- 移行専用M01〜M04として、旧DBが開けること、既存行が保持されstandardになること、既存貸出が残ること、再実行可能であることを追加する。
- 変更前個体に対し、既存行はPASS、新規行はFAILになることを確認する。これにより検査器が変更を弁別することを較正する。

### B7. fresh工場で部分再製造する

- 設計対話、オラクル詳細、他工場成果物は渡さない。
- 改訂BOM、ECO、work order、変更前ソース複製を渡す。
- 影響分析内だけを変更し、自己受入が赤なら納品せず停止・報告するよう指示する。
- 複数工場を使う場合、互いの成果を見せない。

### B8. 五分類で受入し、保守記録へ戻す

- 既存行失敗 = regression
- 新規行失敗 = change miss
- 移行失敗 = data-preservation miss
- 影響外diff = unnecessary modification
- 自己受入赤で停止 = manufacturing nonconformance

合格後、As-Builtへ入力版、工場、artifact hash、検査結果、ずるを追記し、S-BOMへ稼働個体と再検査経路を反映する。

## 4. C. 両者で同じ部分

| 共通する活動 | 説明 |
|---|---|
| 要求を確認する | 上限3/5、既存会員の扱いを決める必要はどちらにもある |
| ドメインモデルを変更する | `MemberType`や貸出可否規則の設計は必要 |
| DBマイグレーションを書く | 既存データを扱うなら手法名に関係なく必要 |
| 自動テストを実行する | 単体、統合、回帰テストはBomDDでも置換されない |
| Gitで差分とbaselineを管理する | BomDDはGit履歴を利用し、代替しない |
| CIで検査を反復する | オラクルやPLM検査をCIへ載せ得る |
| 人間が業務・リスクを裁定する | 上限値、既存会員の意味、受容可能な移行停止等は人間責任のまま |

## 5. D. 両者で異なる部分

| 観点 | 一般的なAI支援開発 [比較設計] | BomDD |
|---|---|---|
| AIの位置づけ | リポジトリを読む実装協力者 | 管理入力だけで作る隔離製造装置 |
| 仕様外の補完 | レビューで問題がなければ黙認されやすい | ずるとして申告・帰属する |
| 知識 | AI/開発者の技能として暗黙 | K-BOMで版と依存先を管理 |
| テスト | 実装と一緒に追加され得る | 設計者側で製造前に固定し、変更前個体で較正する |
| 未規定事項 | 実装時に妥当そうな値を選ぶ | specified / exploratory / out-of-scopeへ裁定する |
| 複数AI | 代替案やレビューに使う | 同じ入力の決定性と差分帰属を測る工場として使う |
| 変更範囲 | PR diffを事後レビュー | 影響あり/なしを事前予測し、diffで採点する |
| 修正の完了 | 現ブランチのコードとテストが緑 | 改訂入力からfresh工場が受入可能品を再現する |
| 稼働後 | issue、依存更新、障害対応へ分散 | S-BOMの劣化イベントからECOへ同じ再入口を使う |

## 6. E. 品質、再現性、保守性の差が現れる箇所

### 品質

差が最も現れるのは、happy pathではなく、**旧DB移行、境界値、変更と無関係な既存契約、検査器の弁別力**である。BomDDは欠陥ゼロを保証しないが、「何の失敗か」を分けて記録する。通常側でも同じテスト群を設計すれば品質は同等になり得るため、品質差をBomDD固有の必然としては主張しない。

### 再現性

通常側の再現性は、同じソースを再ビルドできることや、同じ会話で修正できることを指しやすい。BomDDで問う再現性は、**正解ソースや設計会話を知らないfresh工場が、改訂された管理入力から受入可能な成果を手修正なしで作れるか**である。ここが最も本質的な差になる。

### 保守性

差は次の変更時に現れる。E/K/M/S-BOMとECOが保守されていれば、要求変更、外部知識更新、製造単位変更、稼働個体の劣化を別々に追える。保守されていなければ、BomDD成果物は古い文書の山になり、通常開発より悪化する。

### この事例に対応する観測値

[FINDINGS](../FINDINGS.md) §7.3のforward-01.5では、同型の会員区分変更についてfresh 2工場の双方が、既存回帰23/23、変更2/2、移行4/4、不要改変0を記録した。影響なし予測もunder/over-inclusion 0だった。[観測]

ただし、この値は「通常AI開発なら失敗した」という証拠ではない。またN=1題材・当時1変更・小規模であり、一般的優越性の証明ではない。比較事例で示す結果差は、**通常側では任意である統制が、BomDDでは工程上の完了条件になる**という差である。

# 第4工程：説明順序の設計

以下は章立てではなく、読者の認識が変わる依存順序である。第5工程では、この順序を守り、同じ図書貸出API変更を途切れさせずに使う。

## P0：テストが緑でも、AI変更の根拠は見えない

- **読む前の読者の認識**: AIにissueとリポジトリを渡し、テストとレビューを十分に行えばよい。
- **提示する事実または事例**: 会員区分変更が新規DBでは通る一方、旧DB移行、既存会員の既定区分、影響外API変更が未検査になり得る。Loop 1では14件の暗黙補完が原版と偶然一致し、丸め方式の未指定はテスト緑のまま乖離した。
- **解消する誤解**: 「テスト緑=仕様が十分」「AIが自然に補った値=要求どおり」。
- **読後に得られる認識**: 問題はAIの能力不足だけでなく、入力の沈黙と検査の被覆不足が成功に隠れることである。
- **次の段階へ進むために必要な理解**: AIがどこから判断を得たかを、コード結果とは別に観測する必要がある。

## P1：AIを“同僚”だけでなく“製造装置”として測る

- **読む前の読者の認識**: AIの品質はモデル性能、プロンプト、レビュー能力で決まる。
- **提示する事実または事例**: 正解ソースと設計会話を見せない工場へ管理入力だけを渡す。BOM外の補完はずるとして申告させる。複数工場の一致と固定オラクルの合格を別々に測る。「鋳造できる」は手修正なしで定義済み受入を通るという操作的条件である。
- **解消する誤解**: 「AI=製造装置」は人間排除の比喩、または同一コードを毎回出すという主張。
- **読後に得られる認識**: 製造装置という見方は、入力の自己完結性と出力の受入可能性を試験するための役割分離である。
- **次の段階へ進むために必要な理解**: 製造入力を、異なる責任ごとに分ける必要がある。

## P2：BOMは文書名ではなく、異なる問いへの回答である

- **読む前の読者の認識**: BOMは部品一覧、または依存ライブラリ一覧である。
- **提示する事実または事例**: 同じ変更でも、会員区分が存在する理由、SQLiteでの実現知識、改修・受入unit、稼働個体の再検査経路は別の問いである。E/K/M/S-BOMの四表で対比する。
- **解消する誤解**: 「E/M/S/Kは同じ情報の重複」「S-BOMは一般的SBOM」。
- **読後に得られる認識**: E=何をなぜ、K=何を知って、M=どう作り受け入れ、S=稼働後に何を再検査・交換するか、である。
- **次の段階へ進むために必要な理解**: 四層を分ける価値は、変更時に同じ論理部品を異なる方向から辿れることにある。

## P3：四層分離は変更理由を保存し、交換範囲を絞る

- **読む前の読者の認識**: 一人の開発者が全部分かっていれば、層分離は余分である。
- **提示する事実または事例**: 会員上限3/5は要求変更なのでE-BOMが変わる。SQLite移行方式はK/M-BOMの関心であり、業務要求ではない。将来SQLiteの脆弱性が出た場合はS-BOMから影響部品と再検査を逆引きし、会員上限の要求を変更しない。
- **解消する誤解**: 「実装方式の変更=要求変更」「依存更新なら全機能を同じ深さで再検査」。
- **読後に得られる認識**: 分離は重複を増やすためでなく、独立に変わるものを独立に改訂し、必要箇所だけを再認証するためである。
- **次の段階へ進むために必要な理解**: 静的な四表だけでは効果は出ず、製造・検査ループが要る。

## P4：BomDDの本体はBOMを流す閉ループである

- **読む前の読者の認識**: BOMを完成させてAIへ渡せばBomDDである。
- **提示する事実または事例**: 要求→E/K/M-BOM+FMEA/Control Plan→オラクル先行→隔離製造→固定受入/探索→ずる・分散の帰属→補正→As-Built/S-BOM、という流れを会員区分変更で辿る。固定オラクルと探索プローブを混ぜない理由、一致≠正しさを示す。
- **解消する誤解**: 「文書管理」「巨大プロンプト」「複数AIの多数決」。
- **読後に得られる認識**: BOMは工程への入力と保守の索引であり、価値は隔離・検査・帰属・再入口まで動かしたときに生じる。
- **次の段階へ進むために必要な理解**: 既存手法と活動が重なることを認めた上で、工程上の差を比較できる。

## P5：既存手法との差は“扱う対象”より“完了条件”にある

- **読む前の読者の認識**: DDD、MDD、要求トレーサビリティ、ドキュメント駆動の寄せ集めである。
- **提示する事実または事例**: 次の比較を、会員区分変更へ適用する。
  - DDDは業務概念と境界を設計するが、隔離AI工場の入力自己完結性までは完了条件にしない。
  - MDDは形式モデルから変換・生成することを中心にするが、BomDDは生成器を特定せず、外部知識、受入深さ、工場差、保守再入口を管理する。
  - 要求トレーサビリティはリンクを提供するが、リンク先だけでfresh工場が製造できるか、ずるが何件かを試験しない。
  - ドキュメント駆動は文書を正本にできるが、固定オラクル/探索分離、隔離、multi-factory帰属を必須にしない。
  - Git/CI/CD/テストはBomDDの実装基盤であり、置換対象ではない。
- **解消する誤解**: 「構成要素が既存なら方法全体にも差がない」「BomDDが既存手法を置き換える」。
- **読後に得られる認識**: 個々の構成要素は既存で、差はAI製造の自己完結性・決定性・正しさ・変更転移を観測可能な完了条件として結合した点にある。同等の統制を既存手法の組合せで作ることは可能である。
- **次の段階へ進むために必要な理解**: 名称の違いではなく、具体的変更で何をどこへ記録し、何を測るかを判断する必要がある。

## P6：ECOで“どこを変え、どこを変えないか”を先に宣言する

- **読む前の読者の認識**: ECOはissueやPRの製造業風名称である。
- **提示する事実または事例**: 会員区分変更について、ECOにbaseline、REQ改訂、E/K/M/S-BOMの影響、既存オラクル不変、新規S24〜S25、移行M01〜M04、影響なし予測、fresh部分再製造、五分類の結果を書く。forward-02で「API層diffゼロ」が1行で反証された観測も示す。
- **解消する誤解**: 「影響分析は変更ファイル一覧」「予測が外れたら失敗なので隠す」。
- **読後に得られる認識**: ECOは、変更を再び製造・検査ループへ入れ、影響分析そのものの精度を採点する単位である。
- **次の段階へ進むために必要な理解**: この統制には費用があり、全案件へ同じ重さで適用すべきではない。

## P7：導入判断は規模だけでなく、再製造価値と失敗コストで行う

- **読む前の読者の認識**: 品質手法なら全案件に適用すべき、または重いので全て不採用。
- **提示する事実または事例**: 次の適否基準とテーラリングを示す。
  - **導入候補**: AIによる変更が継続する、担当/モデル交代がある、既存データ移行がある、外部依存が劣化する、複数実装工場を比較したい、影響外変更のコストが高い、監査可能な変更根拠が必要。
  - **特に有効そうな例**: 長期運用する.NET業務APIで、SQLite/SQL Serverスキーマ、外部サービス契約、権限・課金規則を保守し、AI担当が交代する案件。
  - **過剰になりやすい例**: 一度だけ使って捨てる、データを持たない、失敗が容易に見える数十行の社内変換スクリプト。通常のテストとレビューで十分なら、フルS-BOMやmulti-factoryの回収先がない。
  - **小規模でも省略しにくい例**: 認証、課金、不可逆データ移行など、変更量は小さくても高リスクな案件。
  - S規模では文書を統合し、探索プローブやService BOMを条件付き省略できる一方、Control Plan、固定オラクル、隔離、ずる報告など測定器は残すというplaybookの方針。
- **解消する誤解**: 「小さい=低リスク」「テンプレートを全部作らないとBomDDではない」「文書だけ減らせば測定器も省ける」。
- **読後に得られる認識**: 導入価値は、再製造・変更・監査の反復回数と失敗コストが、BOM同期・検査・人間ゲートの費用を上回るかで判断する。
- **次の段階へ進むために必要な理解**: 適否判断には、成功値だけでなく証拠の限界と失敗条件を見る必要がある。

## P8：BomDD自身も検査対象であり、万能性は未証明である

- **読む前の読者の認識**: 実証研究の数値があるなら方法は確立済み、またはNが小さいから全て無価値。
- **提示する事実または事例**: 観測範囲、N、同系統工場、固定オラクル被覆を明示する。残る暗黙知、検査器のバグ、影響写像のハブ漏れ、文書同期コストを示す。「現固定オラクル被覆で未観測差分ゼロ」としか言わない規律を紹介する。
- **解消する誤解**: 「差分ゼロ=完全」「工程に従えば検査器も正しい」「研究限界がある=何も分からない」。
- **読後に得られる認識**: BomDDは、完全性を宣言する方法ではなく、不完全な入力・工場・検査・影響分析を観測し、次のループで管理対象へ変える方法である。適用判断は仮説と証拠を分けて行う。
- **次の段階へ進むために必要な理解**: ここまで理解できれば、第5工程の本文を読み、自分の案件に記録先と停止条件を割り当てられる。
