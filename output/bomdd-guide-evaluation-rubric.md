# BomDD 解説書 評価ルーブリック

## 1. 目的と評価者の立場

このルーブリックは、BomDD を初めて読む C#/.NET 技術者が、文書だけを根拠に正しい説明と適用判断を行えるかを評価するためのものとする。文章の美しさ、著者への好意、用語数の多さは採点しない。

評価者は次の前提を固定する。

- 業務システム開発、DDD、Git、CI/CD、単体テスト、一般的な設計書は知っている。
- 製造業の BOM、ECO、FMEA は詳しくない。
- BomDD の事前知識はなく、新手法の宣伝には懐疑的である。
- 著者の意図、リポジトリ内の別文書、一般的な BomDD 知識で、候補文書の欠落を補わない。
- 候補文書を単独で読んだ後に初めて、リポジトリの一次資料と照合する。

リポジトリ資料は事実確認にのみ使用する。候補文書に書かれていない事実を「知っているはず」として合格させてはならない。

主な照合資料は `README.md`、`WHITEPAPER.md`、`docs/concept.md`、`docs/terminology.md`、`FINDINGS.md`、`method/bomdd-method-v1.md`、`method/bomdd-playbook-v1.md` とする。資料間に時点差や表現差がある場合、候補文書が断定範囲と根拠時点を明記しているかを評価し、評価者が都合のよい記述を選んで補完しない。

## 2. 評価の二段階

### 2.1 第1〜第4工程の中間評価

この段階では本文の完成を要求しない。評価対象は、後の本文が読後の7合格条件へ到達できる情報設計になっているかである。

中間評価の合格条件はすべてを満たすこととする。

1. 総合点が 85/100 以上。
2. 重大欠落が 0 件。
3. 下記 D2、D3、D4、D5 がそれぞれ評点3以上。
4. 読後の7合格条件すべてについて、どの認識段階・事例・判断課題で成立させるかが追跡可能。
5. 第3工程の一貫事例が、第5工程の本文と第6工程の理解テストまで継続利用できる具体性を持つ。

中間評価で「合格」とするのは本文の合格ではなく、「本文執筆へ進める設計である」という意味に限定する。

### 2.2 第5〜第6工程を含む最終評価

最終評価では、候補文書だけを読んだ評価者が、本ルーブリック第7節の10項目へ実際に回答する。最終合格は次のすべてを満たす場合に限る。

1. 理解度分類が「理解できた」である。
2. 評価項目1〜6に、文書から導ける一意で実行可能な回答がある。
3. 評価項目7〜10に、判断を妨げる重大な未決点、複数解釈、既存手法との差の消失、根拠のない中心主張が残らない。
4. 末尾の理解テストが読後の7合格条件をすべて測り、模範解答または採点観点によって誤判断を識別できる。
5. 重大欠落が 0 件。

## 3. 共通採点尺度

各評価軸を 0〜4 で評定し、`評点 ÷ 4 × 配点` を得点とする。

| 評点 | 判定 |
|---:|---|
| 0 | 欠落、または文書内の矛盾により判断不能 |
| 1 | 名称・主張はあるが、関係や判断基準がない |
| 2 | 概要は分かるが、具体案件では複数解釈または補完が必要 |
| 3 | 第三者が概ね正しく説明・判断できる。軽微な不足はある |
| 4 | 第三者が根拠を示して説明・比較・適用でき、反例や限界も扱える |

「書いてある量」ではなく、読者がそこから取れる行動で評点を決める。

## 4. 第1〜第4工程の採点表

### D1 読者の誤解と認識移行の設計 — 10点

確認する証拠:

- 初見時の自然な解釈が列挙されている。少なくとも「BOM形式の文書管理」「設計書を細分化したもの」「DDD/MDD/要求トレーサビリティ/ドキュメント駆動開発の言い換え」「AI向けプロンプト管理」「SBOM」「コード生成手法」を含む。
- 「文書が増えるだけ」「AIに任せるのに工程が重い」「テストやCIで十分」「BOMが正しければ品質も正しいのか」「誰がBOMとオラクルを保守するのか」という反発・疑問を扱う。
- 各誤解に、解消する段階、提示する事実または事例、解消後の認識が対応する。
- 誤解を冒頭の用語定義だけで解消したことにしていない。

減点例: 読者の疑問が著者側の想定問答に留まり、懐疑的な技術者が実際に持つ「既存手法で足りるのでは」という比較軸がない。

### D2 中心主張・問題・因果関係 — 15点

確認する証拠:

- BomDD の中心主張が一文で定義され、BOMという名詞ではなく、AI製造と検査・改善を含む工程として読める。
- 一般的な AI 支援開発の具体的失敗がある。例として、未規定事項をAIが暗黙知で補う、複数AIが一致して誤る、浅い検査が意味的な誤りを見逃す、ソース直接修正で上流成果物と乖離する、外部知識の更新影響を追えない、のいずれかを因果付きで示す。
- 失敗から、投入情報の構造化、製造装置の隔離、ずる報告、固定オラクルと探索プローブの分離、工場間分散の帰属、変更後の再検査へ因果がつながる。
- 「一致」と「正しさ」を別軸にする理由が示される。
- 「鋳造」は比喩だけでなく、BOM/工程のみから生成し、手修正なしで定義済み受入を通るという操作的意味で扱う計画がある。

評点3以上には、単に「AIを製造装置とみなす」だけでなく、それにより日常作業と品質判定がどう変わるかまで設計されている必要がある。

### D3 既存手法との境界と「何ではないか」 — 15点

確認する証拠:

- DDD、MDD、要求トレーサビリティ、ドキュメント駆動開発を個別に比較する。
- 各比較は「相手の主目的」「BomDDと共通する部分」「BomDDで追加される工程または測定」「併用関係」の4点を持つ。
- 既存の BOM、FMEA、PLM、双方向トレース自体を BomDD 独自発明とは主張しない。
- BomDD の識別点を、個別技法ではなく、隔離AI工場、固定オラクル/探索プローブ、ずる報告、複数工場の分散帰属を一つの製造・検査・改善ループに結合する点として検証可能にする。
- 「文書管理」「SBOM管理」「プロンプト集」「DDDの代替」「MDDの別名」「テスト自動化の別名」「人間不要の自動生成」「BOMを書けば正しさが保証される手法」ではないことを明記する計画がある。

比較の最低基準:

| 比較対象 | 文書から判別できるべき境界 |
|---|---|
| DDD | ドメイン知識のモデル化とは重なるが、AI製造装置の隔離、製造単位、検査深さ、来歴、保守影響までをDDD単独は規定しない |
| MDD | モデルを生成入力にする点は似るが、BomDDはAIの未規定補完と出力分散を測り、受入と改善へ戻す。決定的変換器を前提にしない |
| 要求トレーサビリティ | 要求から成果物へのリンクは構成要素だが、製造方法、暗黙知、受入モダリティ、工場差、保守処置を含む全工程ではない |
| ドキュメント駆動開発 | 文書を先に書くこと自体が核ではない。構造化された投入物、独立した合否判定、未規定補完の観測、変更による同期と再検査が核 |

### D4 E-BOM・M-BOM・S-BOM・K-BOM・ECO の操作的分離 — 20点

確認する証拠:

- 4種のBOMを定義するだけでなく、同じ変更案件の情報をどこへ置くか判断できる。
- E-BOMは論理部品、責務、存在理由、要求との関係を扱う。
- M-BOMは実現技術、非機能、調達部品、製造単位、Routing、Control Plan、FMEA、合否基準を扱う。
- K-BOMは要求から導けない外部ツール仕様、設計トークン、フレームワークやライブラリの利用知識などを、変更・版管理可能な知識部品として分離する。
- S-BOMは Service/Maintenance BOM であり、一般的な Software Bill of Materials と同一ではない。稼働個体について、外部知識・調達部品・環境などの劣化イベントから影響部品、必要な再検査深さ、交換または再製造判断へ逆引きする。
- E/M/K/Sを分ける理由が、「目的、変更速度、責任者、判断時点が違い、分けることで必要箇所だけを改訂・再製造・再検査できる」こととして因果的に示される。
- ECOが単なるチケットや変更履歴ではなく、変更理由、効力、影響/影響なし予測、改訂対象、検査追加、移行、実施結果、来歴を貫く変更単位として扱われる。
- 一つの情報を重複転記するのか、参照で結ぶのかが不明瞭になっていない。

具体案件で最低限判定できるべき問い:

1. 業務規則の変更は要求・仕様とE-BOMのどこへ反映するか。
2. API、永続化、採用ライブラリ、非機能、受入基準の変更はM-BOMのどこへ反映するか。
3. 外部ライブラリやツールの版・利用慣行の変更はK-BOMのどこへ反映するか。
4. 稼働個体と外部依存の対応、再検査範囲、交換/再製造判断はS-BOMへ何を残すか。
5. ECOには、変更理由、対象、非対象、効力、移行、オラクル、検査結果をどう結ぶか。

上記のいずれかが文書外の推測を要する場合、評点3にはできない。

### D5 一貫した比較事例 — 15点

確認する証拠:

- 一つの現実的な変更案件に、業務規則、外部表面またはデータ移行、回帰リスクのうち少なくとも二つが含まれる。
- A 一般的AI支援、B BomDD、C 同じ部分、D 異なる部分、E 品質・再現性・保守性の差が、同一条件で並ぶ。
- 一般的AI支援側を「テストしない」「レビューしない」など不当に弱く描かない。要件確認、実装、単体/結合テスト、レビュー、CIは両者に存在し得る共通部分として扱う。
- 結果差は「BomDDだから高品質」という宣言でなく、どの未規定事項、検査深さ、知識依存、影響予測、再製造で差が生じるかに帰属する。
- BomDD側では、少なくとも要求/仕様、E-BOM、M-BOM、K-BOM、S-BOM、ECO、固定オラクル、ずるまたは未規定残渣のうち、案件に必要なものへの記録判断が見える。
- 品質、再現性、保守性を混同しない。複数工場の一致は決定性の証拠であり、正しさは独立した受入で判定する。
- 事例が本文全体と理解テストで再利用できる。

重大な減点: 結果差が「書類を詳しくしたから漏れが減る」で終わり、隔離、受入、分散帰属、変更同期の効果を分離できない。

### D6 認識変化の依存順序 — 10点

確認する証拠:

- 各段階に「読む前の認識」「提示する事実/事例」「解消する誤解」「読後の認識」「次に必要な理解」の5項目がある。
- 冒頭は現実的な失敗で始まり、用語一覧から始まらない。
- 推奨される依存順序が概ね、失敗の観測 → 通常のAI支援では残る原因 → BomDDの中心主張 → BOM分離 → 製造/検査ループ → ECOでの変更 → 既存手法との境界 → 適用判断、となる。
- 新用語を一度に投入せず、導入直後に同じ事例で使う。
- 比喩から正式工程へ移る箇所が明確である。
- 最後に読者が、自分の言葉での説明、比較、案件への記録判断、導入/非導入判断を実施する設計がある。

### D7 実証・仮説・適用範囲の境界 — 5点

確認する証拠:

- 観測事実、そこからの推論、作業仮説、未検証、適用範囲外を区別する。
- 「すべてのソフトウェアで成立」「AI出力が一意なら正しい」「外部仕様の劣化を自動検出できる」「人間承認が不要」など、資料が支えない一般化をしない。
- N、題材、モデル系列、検査被覆、原版あり/なしなど、主張を解釈するために必要な限界を示す計画がある。
- 事例による説明と効果の一般証明を混同しない。

### D8 導入コスト・適用限界・失敗条件 — 10点

確認する証拠:

- 導入コストとして、要求/BOMの構造化、オラクルと治具の作成・較正、K/S-BOM保守、複数工場実行、差分帰属、人間承認、ECO同期、教育/ツール整備の少なくとも主要項目を扱う。
- 有効案件の条件を、長寿命、高変更頻度、高い監査性、複数AI/担当者間の再現性、外部依存、データ移行、障害影響、高い保守コストなどのリスクから説明する。
- 過剰案件の条件を、短命な試作、低リスク、変更/保守予定が乏しい、単純で既存テストだけで十分、BOM/治具保守コストが失敗コストを上回る、といった比較で説明する。
- 失敗条件として、BOMが実装の転記になる、BOM/オラクルが陳腐化する、オラクルが表現に過剰結合する、隔離が破れる、ずる報告が形骸化する、複数工場一致を正しさと誤認する、ECOを飛ばして直接修正する、を扱う。
- 導入可否が宣伝的な「複雑なら有効」ではなく、案件条件から判断できる。

中間成果物には、最低限次の対応が読み取れることを要求する。

| 読後の合格条件 | 第1〜第4工程で必要な設計証拠 | 最終文書での観測方法 |
|---|---|---|
| 1. 自分の言葉で説明 | 一文の中心主張、失敗→対策→工程の因果 | 評価項目1、理解テストの自由記述 |
| 2. 既存手法との差 | 4手法ごとの共通点・相違点・併用関係 | 評価項目2・9、比較問題 |
| 3. 文書管理ではない | 隔離製造、独立受入、ずる、分散帰属、ECOフィードバック | 評価項目2・3・9、誤適用問題 |
| 4. 4種BOMの分離理由 | 同一案件での情報配置と変更伝播 | 評価項目4、配置理由の説明問題 |

| 6. 変更案件へ適用 | ECOと各BOMへの記録判断を含む一貫事例 | 適用問題、模範判断との照合 |
| 7. 導入/過剰を区別 | コスト、リスク、寿命、監査、保守性による境界 | 評価項目5・6、非導入判断問題 |

## 6. 重大欠落の判定条件

次のいずれかがあれば、点数にかかわらず不合格とする。

1. BomDDをBOM文書の作成・保管手法として説明し、AI製造と独立した検査・改善ループがない。
2. DDD、MDD、要求トレーサビリティ、ドキュメント駆動開発のいずれかとの違いがなく、「似ているが違う」という主張だけで終わる。
3. 一般的AI支援開発との比較がなく、または比較相手をテストもレビューもしない非現実的工程にしている。
4. E/M/S/K-BOMのいずれかが欠ける、S-BOMを一般的なSoftware Bill of Materialsと誤認させる、または分離理由が説明できない。
5. ECOが欠け、具体的変更でどのBOMを改訂し、何を再検査するか判断できない。
6. 一貫した変更事例がない、または事例が用語の置換だけで結果差を生まない。
7. 複数工場の一致を正しさの保証として扱い、決定性と正しさを分離しない。
8. 固定オラクルを製造AIへ渡す、あるいは合否用の固定オラクルと未規定面を観測する探索プローブを区別しないため、改善効果を同じ尺度で比較できない。
9. BomDD固有部分と既存手法で可能な部分を区別せず、既存概念を独自発明として主張する。
10. 実証、仮説、未検証、適用範囲外の区別がなく、研究結果を普遍的保証として記述する。
11. 導入コスト、適用限界、失敗条件、非導入案件のいずれかがない。
12. 第4工程が章題の列挙に留まり、各段階の5項目または理解の依存順序がない。
13. 第5工程を開始する前の中間確認であるにもかかわらず、未解決の重大設計判断を隠して本文へ進める。

## 7. 最終文書を読む評価者の必須回答形式

評価者は候補文書だけを読んだ直後、次の10項目へ回答する。回答後に資料照合を行う。

1. **BomDDとは何か**: 2〜4文で自分の言葉に言い換える。BOM名の列挙だけは不可。
2. **通常の開発手法との違い**: 共通工程と、BomDDで追加・変更される工程を分ける。
3. **日常作業の変化**: 変更依頼から完了まで、開発者の作業が「AIへ依頼してコード差分を直す」から何へ変わるかを、入力、製造、検査、記録、保守の観点で述べる。
4. **4種BOMの分離理由**: 各BOMの役割と、統合した場合に失う判断能力を述べる。
5. **有効案件**: 案件を一つ挙げ、リスクとBomDDの工程がどう対応するかを述べる。
6. **過剰案件**: 案件を一つ挙げ、導入コストがどの便益を上回るかを述べる。
7. **判断不能点**: 文書だけでは行動を決められなかった点を、必要な追加情報とともに列挙する。
8. **複数解釈**: 複数の工程、責任者、記録先、合否条件に読めた記述を引用せず要約する。
9. **既存手法の言い換えに見える部分**: どの既存手法のどの部分と同じに見え、BomDD固有の差が文書中で実証されているかを述べる。
10. **根拠不足**: 効果、独自性、一般性について、事例・測定・限定がなく著者の主張だけに見える箇所を挙げる。

回答の合格判定では、評価者がリポジトリ知識で正答できたかではなく、候補文書内の記述から正答できたかを見る。「文書にはないが一般常識で分かる」は不合格要因である。

## 8. 理解度の5分類

最終評価では、次のうち最も重大な状態を一つだけ選ぶ。

| 分類 | 判定条件 |
|---|---|
| 理解できた | 自分の言葉で定義でき、既存手法との差、日常工程、4種BOM、ECOへの記録、適用/非適用を一貫して判断できる |
| 用語は分かったが、工程として理解できない | E/M/S/K等の意味は言えるが、変更依頼から受入・保守までの順序、担当、入出力が説明できない |
| 工程は分かったが、既存手法との差が分からない | 流れは説明できるが、DDD/MDD/トレーサビリティ/文書駆動との差が「詳しい文書化」以上にならない |
| 理念は分かったが、実際の適用方法が分からない | AIを製造装置とみなす意図は分かるが、具体変更をどのBOM/ECOへ記録し、何を検査するか決められない |
| 文書内で矛盾または重大な欠落がある | 定義・工程・事例が食い違う、重大欠落条件に該当する、または複数の正反対な行動が成立する |

複数に該当する場合は、最下段の「文書内で矛盾または重大な欠落がある」を最優先し、次に実適用不能、差別化不能、工程不能の順で選ぶ。

### 9.5 第1〜第4工程への修正要求

修正要求は「詳しく書く」ではなく、欠けている判断能力で記述する。

- Blocker: この欠落により7合格条件のいずれかを測れない。
- Major: 補完しないと複数の工程または記録先に読める。
- Minor: 判断はできるが、用語・例・根拠範囲に軽微な不整合がある。

各指摘には、対象箇所、読者が現在行う誤判断、必要な追加事実または事例、再評価方法を含める。

## 10. 評価時の禁止事項

- 著者の意図を推測して「おそらくこういう意味」と救済しない。
- リポジトリの別資料へのリンクだけを、候補文書中の説明と同等に扱わない。
- 用語が登場しただけで理解可能と判定しない。
- 評価者自身が製造業用語を知っている前提で省略を許さない。
- 文章表現、語調、図の美しさを合否理由にしない。
- BomDDに好意的な結果を正解とせず、導入しない判断も正しく導けるかを見る。
- 根拠のない利点を「将来証明されるだろう」として許容しない。
- 事例の成功を一般法則の証明として扱わない。
