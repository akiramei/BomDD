# BomDD 実証研究 最終総括 — ソフトウェア工業化はどこまで成立するか

> **一文要旨**: 本研究で観測した範囲では、ソフトウェア部品は「**ロジックの核**」と「**外部表面**」に分けて管理できた。核は BOM から鋳造でき(原版非開示の装置が BOM/工程のみから生成し、定義済み受入を手修正なしで通過)、
> 工業化が成立した。表面(外部ツールのコマンド文法・画面の見た目)は要求から導けず、外部/設計知識の転記と
> execution/golden+人間判断による検査を伴って初めて鋳造保証が立った。**観測した12件のずるはすべて表面で出た**(単一題材。一般法則としては未検証)。

題材: 既存アプリ **MoviePad**(Avalonia/.NET/C# + LibVLC + ffmpeg の非破壊動画編集)を BOM 手法で再実装。
方法: アクションリサーチ。①リバース→②E-BOM→③M-BOM→④工程→⑤AI製造(隔離装置)→⑥合否 を回し、
手法から導けず従来手法で埋めた箇所=**「ずる」を観測データ**として記録・分析・手法改善した。

## 3. 中心的発見

### 3.1 構造的法則 — 核と表面(3領域で再現)
どの部品も2層に分かれる:
- **ロジックの核**: ドメイン代数(Split)/判定規則(ExportPlan)/幾何(Layout)。**要求から導出可・unitで検査・BOMから鋳造できる。** 3領域すべてで核は全green。
- **外部表面**: ffmpegコマンド文法 / 画面の画素。**要求から導けない。** 外部仕様(ffmpeg)や設計システム(デザイントークン)の知識を転記する必要があり、検査は execution / golden + 人間判断を要する。

「ずる」は核ではなく**常に表面で**出た。これは製造業の比喩と一致する: 寸法・機構(核)は図面から作れるが、表面処理・嵌合・外観(表面)は加工知識と検査工程を要する。

### 3.2 C2 — BOMが十分に見える理由は「共有された暗黙知」
製造装置は BOM に無い大量の判断を、自分の暗黙知(C#慣習・ffmpeg慣用・配色直感)で埋め、**たまたま原版と一致**した(Loop1.5: 16ギャップ中発散2、Loop2: ffmpeg慣用を自前再現)。
- BOM が十分に見えるのは記述が完全だからでなく、**製造者と設計者が暗黙知を共有するから**。
- 共有知識が**尽きた一点**で、ずるは様式差でなく**機能バグ**になる(Loop2 CHEAT-008 の音ズレ)。
- これが「AIを製造装置とする」構想の核心リスク: **NCデータ(BOM)の精度不足が装置の暗黙知に隠れて見えなくなる。**

### 3.3 受入は「梯子+承認者」 — 「テスト緑 ≠ 仕様適合」
受入モダリティには深さの段階があり、領域ごとに必要十分が違う:
`L0 文字列照合 < L1 存在/終了コード < L2 メタデータ(ffprobe) < L3 内容/信号(PSNR/相関) < 知覚golden+人間承認`
- L0 は過剰結合(様式差を誤検出)かつ被覆不足(意味乖離を見逃す)。Loop2 で実証、Loop2.5 で execution(L2)に上げて解決。
- pixel-exact golden も同病(Loop3)。知覚は **許容差 + 設計承認者** が要る。
- 「テスト緑 ≠ 仕様適合」は丸め(005)・コーデック(009)・配色(011)で繰り返し現れた。

### 3.4 収束は漸近 — 改善ループは効くが完全性は一括達成しない
改善ループは実測で効いた(Loop1.5: 介入1→0、Loop2.5: 沈黙乖離を捕捉)。
だが各ループは「未知の慣習依存」を「明示仕様」へ変換するのは**発見済みの次元に限る**。Loop1.5 で丸めを潰すと、次は Excluded継承が顔を出した。
→ **完全性は漸近**。「BOMで振る舞いを完全記述できるか」は、領域の表面知識を尽くすまで終わらない。

### 3.5 K-BOM と品質の二軸(Loop4)
暗黙知(C2)を「装置が勝手に持つ部品」でなく**管理対象(K-BOM=知識部品表)**にする。3工場(opus/sonnet/haiku)に同一仕様を渡すと、K-BOMなしは約8次元で総当たり分岐、K-BOMありは3工場が文字一致(ばらつき0)・低能力工場も均等化。
→ **BOM品質は二軸**: (1)決定性=工場間ばらつき (2)正しさ=受入。**一致≠正しさ**(揃って間違う「共有暗黙知の罠」, CHEAT-012)。両軸を独立検査する。

### 3.6 受入の梯子は L3 まで実装され、PLM は保守層を閉じた(Loop5,6)
- **L3(信号比較)**: 尺・コーデックが同一(L2では区別不能)でも、音声の座標系バグを内容比較で捕捉(Loop5)。受入の梯子 `L0<L2<L3<golden` が実装で裏付いた。
- **S-BOM/PLM**: 外部知識(K-BOM)更新を「表面の劣化」として逆引きし、影響を6部品中1へ絞込・核は再検査不要・必要な深さ(L3)を導出(Loop6)。**PLMの価値は被覆でなく絞り込み**。治具も劣化対象(知識更新が許容差を上げ得る)。

### 3.7 リバースの弱点は behavior でなく rationale-precision(Loop7)
原版を複数リーダーに読ませると、振る舞い・座標系は全員一致(コードが自己説明的)だが、**根拠の精度**で乖離(全員が「AV同期」と粗く読み、真の「AACサンプル精度」を取りこぼした)。根拠は**受入の許容差**を決めるため、過小読みは粗い受入を招く(Loop6 治具劣化と同根)。
→ リバース工程=双方向トレース+カバレッジ監査。受入深さを決める根拠行は K-BOM で補強、unknown は S-BOM 登録。

## 4. 三層BOM構想への正式フィードバック

実証から、構想([memory: bomdd-core-concept])を次のように更新すべき:

### E-BOM
1. **部品を「核」と「表面」で型分けする。** 核=要求由来(requirement)。表面=外部/設計知識由来。表面部品は出所(外部ツール仕様・デザインシステム)へ**トレース必須**にし、変更は defect でなく**知識更新**として扱う。
2. **存在理由の型を拡張**: `requirement / defect-class / constraint / external-knowledge / design-token`(CHEAT-003/007/010)。
3. **木でなくグラフ**: 横断部品(woven)に owner/consumers を持たせ、変更影響を辿れる(CHEAT-002)。

### M-BOM
4. **受入基準に3点を必須化**: (a) 無矛盾性検査(CHEAT-004)、(b) 完全性=全数値/全フィールドの規定(丸め・継承含む, CHEAT-005)、(c) **検査モダリティ列(L0–L3+golden)と承認者**(CHEAT-009/011)。
5. **識別子・座標系など「関係の不変条件」を明示**(ID採番ポリシー、seekBaseの時刻基準: CHEAT-008)。

### 工程(製造)
6. **製造装置を隔離**し、**「BOMに無く慣習で埋めた箇所」の網羅報告を正式工程に**(C2 を可視化)。本研究で最も効いた測定器。
7. 表面部品には**仕上げ・検査工程**(execution/golden)を必ず付ける。

### S-BOM / PLM
8. 表面の出所(外部ツールのバージョン・デザイントークン)を保守対象に。ffmpeg のEOL・デザイン改定は核を変えずに表面を劣化させる=交換理由。

### 「コードを鋳造品とみなせるか」への最終回答
- **核は鋳造できる**(要求→BOM→AI製造→unit受入。手修正不要の再生成が成立)。
- **表面は“仕上げ・検査工程”を伴って初めて鋳造保証が立つ**(外部知識の固定 + execution/golden + 人間承認)。製造業の表面処理・外観検査と同型。
- よって「ソフトの工業化」は**部分的に成立する**: 核は完全自動化に向かい、表面は半自動+人間検査の協働に落ち着く。

## 5. 到達点と「研究の外周」(将来課題)
**中核は閉じた**: 入口(リバース, Loop7)→ E-BOM(設計)→ M-BOM+工程(製造)→ S-BOM(保守)が一周。ずる12件すべてに手法的対策。
正規方法論は [method/bomdd-method-v1.md]、外部公開版は [WHITEPAPER.md](WHITEPAPER.md)(強い主張/未検証の分離を含む)。

残るのは構想の外周(BomDD v2 research program):
- **別題材での再現性**(最優先): Web/分散/組込・リアルタイムで核/表面法則が再現するか。
- **収束の定量化**: 表面知識が尽きるループ数(題材サイズ依存)。
- **BOM=成果物 崩壊の境界**: 表面の最適BOM化粒度(外部ツール/デザインシステムの部品外部化)。
- **長期PLM**: AIモデル更新を S-BOM 劣化イベントとして扱う再製造差分管理。

各ループ詳細: `loops/loop-0*-*/`。手法: `method/`(bomdd-method-v1 / control-plan / k-bom-ffmpeg / s-bom-template / cheat-taxonomy / improvements)。

## 6. v2 — Web/API での外部妥当性検証(別題材: 予約 API)

§5 の最優先課題「別題材で核/表面の法則が再現するか」に着手。別リポジトリ **BomDD-WebApi-Sample**(会議室予約 API・.NET10 minimal API)で、機械可読 BOM(E/K/M-BOM・Control Plan・Routing・As-Built・Service BOM の YAML)から隔離再製造を回した。核=予約可否/重複/冪等/キャンセル期限、表面=HTTP status code・OpenAPI 契約・API key 認証・UTC 日時・JSON error schema。製造装置は原版非開示のクリーンなサブエージェント、採点は設計者側の**固定ブラックボックス HTTP オラクル**(原版と製造品の 2-way diff、内部C#名でなくHTTP契約で見る)。

> **証拠リポジトリ(公開)**: [akiramei/BomDD-WebApi-Sample](https://github.com/akiramei/BomDD-WebApi-Sample) — 機械可読 BOM(`bomdd/`)・各ループ成果物(`loops/webapi-0*/`)・固定オラクル/探索プローブ・As-Built/metrics。コミット列:
> [`35c3022`](https://github.com/akiramei/BomDD-WebApi-Sample/commit/35c3022) seed → [`7b3a4f2`](https://github.com/akiramei/BomDD-WebApi-Sample/commit/7b3a4f2) webapi-01(2/16)→ [`c46fe8b`](https://github.com/akiramei/BomDD-WebApi-Sample/commit/c46fe8b) 01.5(3/16)→ [`4eed25f`](https://github.com/akiramei/BomDD-WebApi-Sample/commit/4eed25f) 01.6(0/16, tag `webapi-02-input-bom`)→ [`84338b0`](https://github.com/akiramei/BomDD-WebApi-Sample/commit/84338b0) webapi-02 多工場(opus 0 / sonnet 1 / haiku 3)。

### 6.1 単一ファクトリの収束 — BOM補正で鋳造性が漸近的に上がる
| ループ | BOM | 原版との差分 | 何が起きたか |
|---|---|---|---|
| webapi-01 | seed | 2/16 | 冪等 fingerprint 構成未列挙→customerId 次元の機能差(blocker)/ リプレイ status 未規定 |
| webapi-01.5 | 第1次補正 | 3/16 | 上記消失。代わりに別の未規定次元(エラーコード名・非Zオフセット)が露出 |
| webapi-01.6 | 第2次補正 | 0/16 | 明記で解消(現固定オラクル被覆で未観測差分ゼロ) |

各補正は毎回 baseline も前ファクトリも知らない fresh device で一致を出した=**修正はコーチングでなく BOM に宿る**。完全性は漸近(締めると別次元が顔を出す)。MoviePad の核/表面の法則・改善ループ(Loop1.5)が Web で再現した。

### 6.2 マルチファクトリ(opus/sonnet/haiku)— 仕様面は転移・未規定面は分散
締めた BOM を固定(tag 化)し 3 工場へ同一供与(BOM・オラクル不変)。固定オラクル層: **opus 0/16・sonnet 1/16・haiku 3/16**。差分を 3 分類すると:
- **capable factory transfer**: opus 0(締めた BOM は能力ある工場へ完全転移。opus は2回とも 0/16=同ティア再現性)。
- **unspecified BOM residue**: sonnet 1(`not_found` code 名は無指定。単一ティアでは慣習で隠れ、別ティアが露見=共有暗黙知 C2)。
- **specified contract miss**: haiku 3(409=conflict を 400、code=unauthorized 欠落=明記済み契約の取りこぼし=**工場能力**)。

探索プローブ層(合否非混在): ID 形式・ID 一意性・日時表現・応答スキーマが全工場で分散=**ばらつきは BOM が沈黙する場所に集中**。Loop4(K-BOM で工場間ばらつき消失)の Web 多工場版。

### 6.3 v2 で言えること(Web/API 時点 N=2 → §6.4 で分散 Saga を追加し N=3)
1. 核/表面の法則は **MoviePad 固有でなく Web/API でも再現**(2 題材目)。
2. **BOM は差分を観測→補正→再製造で鋳造性を漸近的に上げる**(2→3→0)。
3. 多工場で **仕様化済み契約は転移し、未規定の表面は分散**。差分は単なる「AI 差」でなく **BOM欠落 / 工場能力 / 探索的未規定面**に**分類できる**。

記述規律: 全工場一致を「完全鋳造」と呼ばず「**現固定オラクル被覆で未観測差分ゼロ**」とする。詳細は BomDD-WebApi-Sample の `loops/webapi-0*/report*.md`・`metrics.yaml`。
**Future Work**: 応答スキーマ・ID アルゴリズム・日時形式まで締めると探索層の分散が消えるか(webapi-02.5)。題材 N の拡大(分散/組込・リアルタイム)。

### 6.4 N=3 — 分散 Saga(非同期イベント駆動)でも再現
3 題材目 **BomDD-DistributedSaga-Sample**(注文フルフィルメント Saga・.NET10・in-memory synthetic event bus)で再検証。核=Saga 状態遷移/在庫・支払い・出荷の判定と補償規則、表面=event schema・at-least-once・冪等 consumer・retry/dead-letter・ordering。HTTP の無い in-process 題材なので、観測の黒箱境界を**「固定シナリオを流し正規化イベントログ/dispatch trace を emit する観測契約」**で作り、内部 C# 名でなくイベント契約 + Control Plan の語彙で 2-way diff した。

| ループ | 工場 | 固定オラクル(挙動契約) | 備考 |
|---|---|---|---|
| saga-01 | factory-01(opus) | **0/7** | 練られた seed BOM から初回 0(収束不要)。核を非同期ドメインで鋳造 |
| saga-02 | opus / sonnet / haiku | **0 / 0 / 0** | 仕様化済み契約は全ティアへ転移(webapi-02 で 3/16 だった haiku も 0) |

- **未規定面は分散**(探索プローブ層・合否非混在): eventId 形式(original 含め4通り)・timestamp 精度・sequence 基点・inbox key 区切り(`:`×3 vs `#`)・causationId 方針・events.json の JSON 形状 が工場間で割れた=ばらつきは BOM が沈黙する場所に集中。

### 7.3 変更オーダー(ECO)は BOM に宿り転移する — 「保守できる」の初回データ(forward-01.5)
納品済み個体(v0.2 = factory-04 ビルド)への ECO-001(会員区分 standard=3冊 / premium=5冊 の導入+**v0.2 スキーマの自動移行**)を Phase 7 の型で実施: 影響分析と**影響なし予測を反証可能な形で製造前に凍結**→ 既存オラクル不変+追加 S24–S25+移行専用オラクル M01–M04(fixture=As-Maintained 個体の実 DB)→ fresh 2工場の部分改修(変更前ソース複製を事前コミット=diff 基準点)。

| 測定 | opus | sonnet |
|---|---|---|
| 回帰 S01–S23(既存オラクル不変) | 23/23 | 23/23 |
| 変更受入 S24–S25 | 2/2 | 2/2 |
| 移行 M01–M04(fixture=実 v0.2 DB) | 4/4 | 4/4 |
| 不要改変(影響分析外への diff) | **0** | **0** |

- **影響なし予測が全的中**(under/over-inclusion 0)。S07(既定 standard=上限3)も予測どおり不変=「**既定値設計が既存オラクルを保存する**」が成立。

### 7.4 フォワード固有の構造リスク — 検査器は未検証のまま初回採点に臨む
リバースでは原版が治具の検証相手だった。フォワードには居ない。実測: 検査器側ずる 2件(CHEAT-F01-H001 デシリアライザ暗黙日時変換=v1 採点を一時誤らせた / H002 PS 変数名大小非区別=**セルフテストが凍結前捕捉**)+ 1件(CHEAT-F015-H001 判定順序干渉=**較正が凍結前捕捉**)。v1.2「C2 は測定ハーネスにも現れる」の4–6例目。対策を3段で正式化(playbook §4.4): ①**治具セルフテスト**(静的)②**較正 negative control**(動的: 検査一式を既知個体へ実行し、既存行=PASS・変更行=FAIL の期待プロファイルと突合してから凍結)③等価規則の表現非依存。

### 7.5 言える範囲と限界

### 7.6 影響分析の取りこぼしは1行単位で測定できた — 非自明影響グラフの ECO と効力(forward-02)
7.5 の残課題への応答(2026-07-04)。依存グラフの中心(期限規則)に入る ECO-002(会員区分依存の貸出期間 standard=14日/premium=21日+**効力=貸出作成時点**)を、「依存があるのに影響しない」を切る反証可能な予測つきで実施した — 延滞判定・料金・API 層は**保存済み確定値(dueDateUtc)経由**で結合しており、コード変更は Core の期限計算1点に閉じると予測(影響 E 品目 1/12・**src/Library.Api は diff ゼロ**と宣言)。

| 測定 | opus(採用) | sonnet(対照) |
|---|---|---|
| 回帰 S01–S25 / 移行回帰 M01–M04 | 25/25・4/4 | 25/25・4/4 |
| 変更受入 S26–S28 / 効力 E01–E04 | 3/3・4/4 | 3/3・4/4 |
| 不要改変 | **0** | **0** |

- **主測定 = 影響なし予測の成績: under-inclusion 1 / over-inclusion 0**。「Api 層 diff ゼロ」予測は反証された — As-Maintained 実物は期限計算を Api 側の別呼び出しサイトに置いており、memberType を通す**配線1行**が不可避。**2 fresh 工場が事前申告つきで同一の1行に独立収束(git blob 同一)**=この diff は工場の逸脱でなく影響分析の取りこぼし。**E-BOM 粒度の予測(1/12)は的中**し、外れたのは品目→unit の写像(実装の分布)。影響分析の精度が「BOM の正しさ」と「実物点検の有無」に**分解できる**ことを示す初のデータ。是正は工程化済み: 影響分析に **As-Maintained 実物の呼び出しサイト点検**を追加(playbook §8 手順 2・テンプレ 61 §1.2 —「実物が正」の影響分析版)。予測を「diff ゼロ」級の反証可能な形で書いたからこそ、取りこぼしが**1行単位**で可視化された。
- **効力(effectivity)の初検証**: スキーマ変更なしの規則変更でも既存データ上の振る舞いは fixture でしか検査できない — 移行オラクルの一般化として**効力専用オラクル**(E01–E04: 既存データ保持/確定値の非遡及/新規のみ新規則/派生計算が確定値基準)を新設(テンプレ 62 §5)。両工場とも遡及の故障モード(FMEA 狙い撃ち)を作り込まず、「確定値を引数に取る」v0.1 以来の構造が効力を**構造的に**保証していた。

### 7.8 互換性が壊れる部品交換 — 昇格の決定点は scratch に前倒しされる(forward-04)

forward-03 の残課題「回帰赤→再製造へ昇格」を検証した。イベントは**シナリオ起票**(DEG-002 = 供給網離脱の組織裁定。DEG-001 の実 GHSA が文脈・外部事実の捏造なし)、**非互換性は実在の部品差**(Microsoft.Data.Sqlite → System.Data.SQLite.Core: 名前空間/型名の実 API 差)。

連鎖: 検出 → 逆引き(1/3 SB)→ 交換候補 → **scratch 検証= build 6 エラー(赤)** → 昇格裁定「**再製造必要**」(処置語彙第4項の**初適用**)→ **K-BOM v3+M-BOM procurement のみ改訂**(REQ/仕様/E-BOM/オラクルは無改訂)→ fresh 工場 1 体の部分再製造(影響 1/3 unit・61 §1.2 実物点検で絞り込み)→ 全再認証 **S 28/28・E 4/4・M 4/4・V01 1/1・不要改変 0**。

- **昇格の決定点は「適用後の回帰赤」でなく scratch 検証に前倒しされる**: 決定的な非互換(コンパイル/決定的挙動差)は forward-03 で工程化した scratch 検証が先取りする。適用後に回帰が赤くなる領域は scratch で再現できない非互換(環境依存・非決定的・実データ規模依存)に限られ、これは意図的に作れない(実イベント待ち — watch CI が受信機)。
- **互換性の壁は K/M 層に閉じた**: 挙動契約(REQ/仕様/E-BOM/オラクル)を 1 行も変えずに部品交換+再製造が完結 — BOM 層分離の実証データ。
- **K-BOM が適応の裁量を吸収**: K v3 に型対応・パラメータ形式・機能可用性を明記した結果、工場 cheat **0 件(forward 系列初)** — 部品交換の適応は知識パック次第で裁量ゼロの機械的作業に落ちる。
- **データ互換性次元の初検査**: 移行オラクルが「旧部品が書いた実 DB を新部品で読む」検査を兼ね、合格(out-of-scope 宣言だった沈黙次元の初実測)。

### 7.9 追補: 部品交換の同等性 — 系統誤差は順位等価を素通りする(viewprism2 ECO-054 還元)

adapter 選定(P-08: full-decode → scaled-decode・6.29× 高速化)は速度と順位等価(EQ-RANK)を実測して
採用されたが、**誤差の系統性は測定外**だった。実測: 異フォーマット複製ペア(png×jpg)が経路非対称
(JPEG= DCT 面積平均縮小 vs PNG= 全解像度点サンプル)により系統的 -18 点(96→78)。4 象限切り分けで
誤差の全量が経路差に帰属。是正(一様中間段)後= 78→96(full-decode と同値)・回転複製は 70 台→100
(既存 ECO の変種照合との相互補完)。限界: N=1(ViewPrism2・知覚ハッシュ)。一般命題「同等性契約は
不変量の保存と誤差の系統性の両輪」の初回実証。

### 8.3 追補: 検査器の沈黙 — 陽性対照の欠如は no-op を健全に見せる(viewprism2 還元)

§7.4「検査器は未検証のまま初回採点に臨む」と §8.1-1「セルフテストの盲点」の先に、3 つ目の様式が実測された:
**常設化された検査器が運用中に沈黙する**(2026-07-06・ViewPrism2 ECO-053)。

観測:
- 台帳リストキーの改名 1 行(別 ECO・doc-only)に validator が未追随 → register 検査 3 本(語彙・参照
  整合)が **3 日間 no-op** 化。改名 ECO の検証「error/warn 不変」は、検査停止による不変を健全と誤読した
  (negative-only 検証の罠)。
- 同期間に YAML 重複キー事故(エントリ誤挿入で parsed 値が化ける)が **0 error/0 warning で素通し**
  (PyYAML の警告なし後勝ち)。発見は人手。復活させた検査は既存台帳から実データ違反 4 件を即検出した
  (検査が止まっていた期間の堆積)。
- 対策の 3 点セット: 重複キー検出ローダ(行番号報告)/ キー不在の明示エラー化(「読めなければ空」型
  fail-soft の排除)/ **陽性対照 selftest の pre-commit 常設**(必ず検出されるべき合成フィクスチャを
  毎コミット確認)。

含意: 検査器ずる対策は 4 段になった — ①セルフテスト(静的・比較器の正しさ)②較正(動的・凍結時の
negative control)③等価規則の表現非依存 ④**陽性対照の常設実行(運用中の沈黙検出)**。①〜③は凍結
時点の正しさを守るが、④だけが「その後も動き続けているか」を守る。コードの生死を呼び出し元で判定する
(同還元の撤去規律)のと同型で、**検査の生死は陽性対照で判定する**。N=1 リポ・validator 1 本の観測
につき一般化はしない(2 例目が出たら cheat-taxonomy の様式へ昇格)。

### 10.4 追補: 検査エージェントのモデル感度 — 質の差は「深さと発見の余剰」に出る(model-sensitivity-01)

発端はユーザーの問い: 「この仕組みはモデル差の影響をどの程度受けるか。期待は、人間のエキスパート/中堅/
若手と同様に**質の差は出てよいが、達成不能には転ばない**こと」。同一の合成プロンプト(emit-prompt・
sha256 来歴付き)・同一ステージング・盲検並列で、lint 検査エージェントのモデルだけを
haiku / sonnet / opus / fable に変え、10.1 の凍結オラクル 16 項目で採点した。

| | haiku | sonnet | opus | fable |
|---|---|---|---|---|
| オラクル再現(hit/partial/16) | 1/4 | **8/3** | 7/2 | **8/3** |
| 未規定契約クラス(修正コスト高・6件) | 1h+2p | **4/6** | **4/6** | **4/6** |
| check(治具・書式規律) | fail(番号形式) | **fail(コスト順序×5)** | pass | pass |
| オラクル外の真発見 | 0 | 1 | 2 | 4 |
| 規律違反 | **根拠なき「可」断定** | なし | なし | なし |

- **能力フロアの実像**: 最小モデル(haiku)は「達成不能」ではなく**「完遂するが検査として信頼できない」**
  という形で落ちた。報告書は最後まで出る・体裁も整う — しかし raw IR に痕跡のない操作を「可」と断定した
  (三行原則違反の実例)。**弱いモデルの危険は沈黙ではなく、自信のある充填**である。候補と質問しか
  出させない三行原則+人間裁定が防波堤である理由の直接実証。
- **中堅の粗さは治具が吸収する**: sonnet の検出力は最上位と同数(hit 8。10.1 の実測 miss 由来で還流した
  観点 3 行がモデル横断で機能し、音量を rank 1 で的中)。崩れたのは書式規律(コスト降順×5)で、
  check が機械検出した — 規律の維持をモデルの自制に頼らない設計が、そのままモデル間可搬性になる。
- **silent miss は実在し、書式合格と直交する**: check 合格の opus / sonnet も、機械矛盾の深掘り
  (README の操作記述 vs IR 痕跡なし)を miss した。この型を取ったのは最上位のみ(2 run 連続)。
  さらに別の 1 項目は**全モデルが miss**(過去 run では hit)— **run 間変動もモデル差と同型の silent miss を
  作る**。書式 lint は検出の深さを保証しない。対策は反復とオラクル健診(遡及採点)の運用側にある。
- **モデル差の本体**: 修正コスト高クラスの再現率は sonnet / opus / fable で同率だった。差が出たのは
  (a) 機械矛盾の深掘り、(b) オラクル外の真発見数(4>2>1>0)、(c) 検算の網羅性 — エキスパートの価値は
  「同じチェックリストでより多く拾う」ではなく**「チェックリストの外を見つける」**に現れる。人間の熟練差と同型。
- **結論**: sonnet 級以上で「質の差であって達成不能ではない」が成立する。フロア未満のモデルは検査官には
  使えないが、装置全体は三行原則+裁定+ゲートで誤混入を防ぐ。プロンプト(正典)と治具は素の markdown/python
  でありベンダー可搬 — 固有なのは adapter 層(スキル)のみ。
- 限界: モデルごと N=1(run 間変動と交絡)・採点者と最上位モデルの同型性・課題は lint 1 種のみ
  (refmodel・意味付与・製造工場のモデル感度は未測定)・他ベンダー未測定。
- 証拠: MoviePad branch `bomdd-model-sensitivity-01`(`8425c5a`: protocol / 4 報告書 / scoring)。

### 11.2 追補: ベンダーの壁も越えた — 文化は横断し、様式は横断しない(transfer-03)

第 3 ラウンドは**ベンダー+ハーネスの複合軸**(GPT-5.5 / Codex CLI — 分離不能を protocol で正直申告)。
題材=有給休暇 API・事前予測 4 件を凍結。結果: **転移成立(N=3・初のベンダー横断)— 介入完全ゼロ**
(method_explanation 0・rescue 0・tooling_gap 0・申告なき充填 0)。一周+ECO 2 件。

**中心的発見**:
1. **正直申告文化はベンダーを横断した**: 要求不適合の自己申告(旧既定の明示)・自己受入失敗の記録・
   帰属診断(「仕様変更であり欠陥でない」)・as-built への "prompt_hash: not-computed" の正直記載 —
   **ずる申告は特定ベンダーの訓練特性ではなく、kit の規律として通じる**ことの初証拠。一方、**様式は
   横断しない**(CHEAT-ID 様式・G4 形式節・git 凍結タグは非採用 — 内容は満たすが形が変わる)。
2. **環境制約はテーラリングが吸収した**: 隔離サブエージェント機構の不在は tooling_gap にならず、
   S 規模・1 工場・G2 セルフ監査の**正規の縮退裁定**として現れた — 権限分離規則が環境制約の受け皿
   として機能。代償として検出された穴: 規模判定の担当者間ばらつき(同クラス題材で M vs S)・
   要求改訂時の oracle 行 supersede 手順の不在(in-place 書き換え→次 ECO で自己改善)・凍結規律の
   git 前提(ハーネス起因)。
3. **ゲート運用の 3 様**: fable=自発 / sonnet=従属(ゲートが連れて行く)/ GPT=**圧縮+文書化**

   説明介入ゼロ帯= §11.1 補償機構の一般化。
4. **第 3 のエリシテーション型= 決定の外部化**: 曖昧さ(年度の区切り)を質問せず・既定で埋めず、
   **決定自体を入力契約へ外出し**(fiscalYear 入力・開始月非依存)。質問(t1)/ゲート誘導(t2)/
   外部化(t3)— いずれも申告なき充填 0 を保つ別経路。ただし外部化は運用判断を呼び出し側へ転嫁する
   ため顧客が気づきにくい(限界として記録)。
- 限界: 複合軸(git 不履行・S 縮退がモデル起因かハーネス起因か分離不能)・受入確認は顧客起点(t2 同様)・
  要求シードの一部が担当者の様式により初回で渡らず遅延投入(計画外 ECO を誘発 — 接点数の比較に注意)。
