# BomDD Playbook v1 — フォワード・モード実践手順(prescriptive draft)

> **ステータス**: prescriptive / **単一題材検証済み(single-case validated)・一般化は未検証**。
> 本書は [bomdd-method-v1.md](bomdd-method-v1.md)(N=3 で実証済みの規則)を、**原版が存在しない新規開発**(ブレインストーミング→仕様→BOM→AI製造)へ移植した実践手順である。
> 検証状況: グリーンフィールド1題材(図書貸出 API・SQLite 永続化)で、初回製造(forward-01: opus/sonnet 20/20・未規定残渣 0)→仕様昇格の転移(rev2: fresh 23/23)→変更オーダー(forward-01.5: 回帰 0・移行 4/4・不要改変 0)まで一周した([../FINDINGS.md](../FINDINGS.md) §7、証拠リポ [BomDD-LibraryLending-Sample](https://github.com/akiramei/BomDD-LibraryLending-Sample))。**N=1 につき一般化はしない**(題材横断の再現が次の検証対象)。method-v1(実証された規則だけの薄い正規版)とはまだ統合しない。両者を混同しない。

## 0. リバース→フォワードの置換表(本書の設計原理)

method-v1 の装置はすべて「原版あり」で実証された。フォワードでは原版が無いため、各装置を次のように置換する。

| リバースの装置(実証済み) | フォワードで失われるもの | 置換(本書の処方) |
|---|---|---|
| 原版との 2-way diff(固定オラクル) | 比較対象の原版 | **仕様由来の固定オラクル**を設計者が**製造前に**書く(§4.4 オラクル・ファースト) |
| 原版=完全性の参照点(差分で未規定を発見) | 「何が未規定か」を教えてくれる相手 | ①[silence-checklist.md](silence-checklist.md) による既知の沈黙次元の事前掃討(§4.2)②**マルチファクトリ分散**=残った沈黙の検出器(§6。「ばらつきは BOM が沈黙する場所に集中する」N=3 で再現済み) |
| リバース工程の双方向トレース(原版証拠↔REQ) | 原版証拠列 | **要求根拠(rationale)↔REQ↔部品↔AC** の双方向トレース。原版証拠の代わりに「なぜその要求か」の根拠を、受入の深さ・許容差が決まる精度まで書く(§2。Loop7 昇格命題の転用) |
| マルチリーダー・リバース(複数AIが原版を読む) | — | **マルチリーダー仕様監査**: 複数AIが独立に仕様を読み REQ・不変条件・受入深さを抽出。差分=仕様の曖昧箇所(§3.2) |
| 製造装置の隔離(原版非開示) | 隠すべき原版 | 隔離対象の転換: ①**設計対話の文脈**(ブレスト・仕様策定の会話)を渡さない=「修正は BOM に宿ったか」のテスト ②**固定オラクルのシナリオ詳細**を渡さない=オラクル過適合の防止(§5.1) |
| ずる報告の義務化 | — | そのまま使う。さらにフォワードでは、ずる報告と探索層分散を**ユーザーへの質問リスト**に変換して要求へ還流する(§6.2。要求エリシテーションの装置になる) |

> 例外: 「既存資産を BOM 化してから拡張する」案件は、入口に method-v1 §1.5 のリバース工程をそのまま使い、以降を本書のフォワード手順に接続する。

## 1. 全体フロー

```
Phase 0  チャーター            … 題材・スコープ・予算・工場構成を固定
Phase 1  ブレインストーミング   … 要求台帳(REQ ledger)を作る
Phase 2  仕様化               … 仕様書+仕様完了ゲート(マルチリーダー監査)
Phase 3  BOM・工程設計        … E/K/M-BOM・Control Plan・Routing・固定オラクル・製造パッケージ
Phase 4  製造                 … 隔離ファクトリへパッケージのみ供与・ずる報告
Phase 5  受入・収束            … オラクル合否+分散観測/ユーザー指摘→差分帰属→質問リスト→BOM補正→再製造
Phase 6  引き渡し・保守        … As-Built・Service BOM
Phase 7  変更/是正オーダー(ECO/CAPA) … 仕様変更・欠陥修正→影響分析→部分再製造→回帰(納品後の再入口)
```

各フェーズに**完了ゲート**を置く。ゲートを通らず次へ進んだ箇所は、後段で必ず「ずる」として現れる(N=3 の一貫した観測)。

Phase 0(チャーター)では、**後から動かすとループ測定が濁る項目**(工場構成・収束ループ予算・役割と承認者・黒箱境界の種別)をこの時点で固定する。題材・スコープは `(仮)` 付きで仮置きしてよく、Phase 1 終了時(G1)に確定する。手順は [prompts/phase0-charter.md](prompts/phase0-charter.md)。

成果物の置き場所(推奨): 対象プロジェクトのリポジトリ直下に `bomdd/` を作り、[templates/](templates/) の番号体系(00=チャーター, 10=要求, 20=仕様, 30番台=BOM, 40番台=製造, 50番台=記録, 60番台=変更オーダー)で置く。規模が小さい場合の省略規約は §11 テーラリング。

## 2. Phase 1 — ブレインストーミング → 要求台帳

**目的**: 散漫な発想を、後段の BOM が参照できる **REQ 台帳**([templates/10-requirements.yaml](templates/10-requirements.yaml))へ収束させる。

**手順**:
1. 自由なブレスト(発散)。ここは形式不要。
2. 収束時に、合意した要求を 1 件ずつ REQ にする: `id / statement / rationale / rationale_precision`。
3. **根拠精度の規律**(Loop7 昇格命題の転用): rationale は「受入の深さと許容差を決められる精度」まで掘る。判定質問は一つ:
   > 「この記述だけから、検査の深さ(unit/L1–L3/golden)と許容差を決められるか?」
   - 例: 「AV同期が必要」(×: 許容差が決まらない)→「シーク後も音声は絶対時刻基準で、ズレは ±N ms 以内」(○)。Loop7 では全リーダーが「AV同期」と粗く読み「AACサンプル精度」を取りこぼした——**過小な根拠は粗い受入を招く**。
4. 各 REQ に**核/表面の仮分類**を付ける: その要求は要求文から実装を導出できるか(核)、外部ツール文法・デザイン・プロトコル慣習の転記が要るか(表面)。表面と分類した時点で K-BOM 調達(§4.3)の候補に積む。
5. AI の役割: 聞き手・構造化係に加えて**反例生成器**(「この要求が破られるシナリオを 5 つ挙げよ」)。反例は FMEA(§4.2)の種になる。

**完了ゲート G1**: 全 REQ が rationale_precision の判定質問を通る。通らない REQ は「未精製」マークのまま Phase 2 へ持ち越してよいが、Phase 3 開始までに精製する。

## 3. Phase 2 — 仕様化

**目的**: REQ 台帳を、製造装置と検査器の両方が参照できる**仕様書**([templates/20-spec.md](templates/20-spec.md))にする。

### 3.1 仕様に必ず含めるもの
- **REQ へのトレース**: 仕様の各節がどの REQ を実現するか(双方向: REQ→節 / 節→REQ。宙に浮いた節・どこにも実現されない REQ を検出)。
- **不変条件の節**: 識別子採番・座標系/基準系・冪等性・順序など「関係の不変条件」(CHEAT-008、M-BOM 必須項目の前倒し)。
- **核/表面の宣言**: 各機能がどちらか。表面には出所(外部仕様・デザインシステム)を明記。
- **表示契約(原典ありの UI/帳票 surface)**: 原典に見えている提示フィールド、ファイル名、サムネイル、ラベル、アイコン、状態表示、空/エラー表示を全列挙し、各要素を REQ・E-BOM・Control Plan へトレースする。移植しない要素は理由付きで out-of-scope にする。これは「見た目」ではなく表示パリティ契約であり、仕様に無い表示要素は凍結後の工場・オラクルでは検出できない。
- **HTML モック入口(candidate)**: AI デザインツールや手書き HTML で実行可能 UI モックを先に作った場合は、[ui-ir-ui-bom.md](ui-ir-ui-bom.md) の手順で UI-IR / UI-BOM / trace map を抽出し、表示契約・E-BOM 候補・Control Plan 候補の材料にする。UI-BOM は候補部品表であり、仕様・E-BOM へ昇格する前に未解決事項を確認する。
- **[silence-checklist.md](silence-checklist.md) の第1回掃討**: 仕様レベルで答えられる沈黙次元(日時表現・エラー語彙・丸め等)を `specified / exploratory / out-of-scope` で宣言するか、`deferred-to-phase3`(第1回掃討のみ有効な明示的延期)を付ける。**無言の先送りを残さない**(deferred は第2回掃討で必ず3択に解消)。

### 3.2 仕様完了ゲート G2 — マルチリーダー仕様監査(転用仮説)
Loop7 のマルチリーダー・リバースの転用。原版の代わりに**仕様**を読ませる:
1. 互いを知らない N 体(≥2、推奨3)の AI リーダーに仕様だけを渡し、独立に「REQ 一覧・不変条件・各 REQ の受入深さと許容差」を抽出させる。
2. 抽出結果を突合する。**差分が出た箇所=仕様が一意に読めない箇所**。Loop7 の実証では、振る舞いは全員一致し**根拠の精度**で割れた——仕様でも同じ割れ方を想定する(深さ・許容差の読みが割れた箇所を優先補正)。
3. **指摘は採用前に本文照合で真偽を裁定する**。差分は「仕様が一意に読めない箇所」だけでなく**監査者の誤読**でも出る(plm-v0: 低ティアの再監査が解消済み矛盾を3件再指摘= specified_contract_miss の監査版)。本文照合で誤読と判定した指摘は理由付きで棄却する——裁定なしの全件採用は、仕様を監査者の読み能力の最小公倍数へ劣化させる。
4. 補正して再監査。差分ゼロ(または残差に理由付き)で通過。

### 3.3 測定可能性ゲート G2' — MeasurementCapability 検査
[observation-schema-candidates.md](observation-schema-candidates.md) §4.9 を仕様ゲートに前倒しする:
- 各 REQ について「観測する深さ・治具・承認者が足りるか」を問い、`adequate / unmeasurable / under-specified-oracle / insufficient-depth / human-approval-required` を付ける。
- `unmeasurable` の REQ は**仕様未完成**(知覚系なら golden+許容差+承認者を、in-process なら観測契約をこの時点で計画する)。「測れない仕様は製造に渡せない」が規律。

### 3.4 凍結前原典パリティサインオフ(原典ありの移植・再現案件)
凍結後の製造装置は原典を見ないため、原典との比較でしか見つからない欠落(パリティ欠陥)は G2/G2' までに止める。表示パリティはその代表例。
1. 原典画面/帳票ごとに表示要素集合を列挙する。
2. 各要素を `仕様節 / REQ / E-BOM item または display_contract / Control Plan 特性` へトレースする。
3. 未トレース要素は `out-of-scope` 理由を承認するか、仕様へ追加して再監査する。
4. 未トレース要素ゼロを凍結前のサインオフ条件にする。ここを通らない BOM は製造へ渡さない。

## 4. Phase 3 — BOM・工程設計

**目的**: 仕様から製造パッケージ(§4.6)を組み上げる。method-v1 §2–§4 の実証済み規則をそのまま使い、フォワード固有の手順(K-BOM 事前抽出・オラクル・ファースト・ドライラン)を足す。

### 4.1 E-BOM([templates/30-ebom.yaml](templates/30-ebom.yaml))
- 核/表面で型分け(§0 第一原理)。核は `requirement_refs` 必須。表面は `external_source_ref` 必須。
- UI-IR / UI-BOM を前段に使った場合は `ui_sources` に参照を置き、UI-BOM 品目を `E-BOM item / display contract / Control Plan / K-BOM / traceのみ` のいずれかへ裁定する。HTML タグや装飾 wrapper をそのまま E-BOM 品目に昇格しない。
- **Design System BOM(candidate)**: UI-CAD が要求する Card / CTA / Chip / Badge / IconButton / SectionHeader / HelpBox / DragHandle などを、`35-design-system-bom.yaml` で E-DESIGN-* surface 部品と K-DESIGN-* 知識に分ける。UI-BOM surface item ごとに coverage matrix を埋め、`missing` のまま製造へ進めない。
- 木でなくグラフ: 横断(woven)部品に owner/consumers(CHEAT-002)。
- 各部品の `invariants` に、仕様の不変条件節を割り付ける。
- **粒度規準(candidate)**: M-BOM unit は「**独立に再製造・交換でき、単独で受入できる最小単位**」(Phase 7 の部分再製造単位と同一)。E-BOM 品目はその設計版(存在理由 1 つ・受入観点が閉じる単位)。迷ったら「この部品だけ fresh factory に再製造させられるか?」で判定する。**実測フィードバック(forward-01.5 → plm ECO-001 で改訂)**: 小規模 ECO では unit 粒度の影響集合が 3/3 unit に達し**絞り込み効果が出ない**ことがある(forward-01.5)。ただし plm ECO-001(同程度の小規模: 改修4ファイル)では **8 unit 中 2** に絞れた — 絞り込み効果は**規模でなく変更の局所性**に依存する。局所的な変更なら unit 粒度は小規模でも機能し、横断的な変更は E-BOM 部品粒度+不要改変 diff 測定([templates/63-diff-audit.md](templates/63-diff-audit.md))で代替する。
- **NFR の置き場所(candidate・三分割)**: 目標値=E-BOM の `nfr_targets`(測定可能な数値+条件)/ 実現手段・予算配分=M-BOM / 検証=Control Plan の NFR 特性行(原則 L3 execution measurement+許容差)。「速い」のままの NFR は G2' で unmeasurable=仕様未完成として弾く。

### 4.2 FMEA + 沈黙チェックリスト第2回掃討
M-BOM 確定**前**に(Loop4):
1. 部品ごとに故障モードを列挙(Phase 1 の反例を種に)。受入を狙い撃つ。
2. [silence-checklist.md](silence-checklist.md) を全行なめ、各次元を3択で**宣言**する:
   - `specified` — M-BOM/K-BOM に明記する(どこに書いたか参照を残す)
   - `exploratory` — 今回は固定しない。探索プローブ層で分散として観測する(合否に入れない)
   - `out-of-scope` — 理由付きで対象外
   **未宣言の次元がずるの温床**(N=3 で、分散は常に BOM が沈黙する場所に出た)。
3. 原典ありの UI/帳票 surface は、`表示要素集合・原典表示パリティ` 行を必ず `specified` または理由付き `out-of-scope` に解消する。`exploratory` のまま製造へ渡すと、原典パリティ欠陥を固定オラクルでは検出できない。

### 4.3 K-BOM 調達 — 暗黙知の事前抽出(転用仮説・フォワード固有)
リバースでは K-BOM は「ずるが出てから」整備した(Loop4)。フォワードでは**製造前に抽出**する:
1. 表面部品ごとに、設計用 AI へ問う: 「この領域を BOM 記載なしで実装するとき、**あなたが慣習で埋めるであろう判断をすべて列挙せよ**」(= ずる報告の事前実行)。
2. 列挙された判断を 1 件ずつ裁定: K-BOM に固定する / exploratory に置く / out-of-scope。
3. K-BOM 項目は `managed_knowledge`(箇条書きの判断)+ `risk_if_implicit`(暗黙のままだと何が分岐するか)を持つ([templates/31-kbom.yaml](templates/31-kbom.yaml)。webapi 実例の形式)。
> 根拠(転用元): K-BOM なし=全次元分岐 / あり=3工場ばらつき0(Loop4)。saga では締めた seed BOM が低能力ティアの取りこぼしも防いだ(saga-02 haiku 0/7)。

### 4.4 受入の設計 — オラクル・ファースト(フォワードの最重要規律)
**製造より先に**、設計者側で観測一式を作り凍結する:
- **Control Plan**([templates/33-control-plan.yaml](templates/33-control-plan.yaml)): 特性ごとに `深さ(unit/L1/L2/L3/G)・許容差・治具・承認者`。核=unit(exact, 丸め規定込み)。表面=領域別深さ(外部ツール≥L2、同期/座標系=L3、知覚=golden+許容差+承認者)。L0 文字列照合は既定不採用(method-v1 §5)。完全性が要る特性(丸め・全フィールド規定)には `test_vectors`(境界値・中間値・反例)を必ず列挙する(CHEAT-005 対策。「受入の被覆が捕捉を決める」)。NFR 特性は L3 execution measurement+許容差で行を立てる(§4.1 三分割)。
- **表示パリティ特性(原典ありの UI/帳票 surface)**: pixel-exact ではなく、表示要素集合の全 required elements が存在し意味一致することを Control Plan 行にする。自動で読めるなら L1/L2(アクセシビリティツリー/DOM/スクリーンショット解析)、知覚判断を含むなら G+承認者。
- **Design System coverage 特性(UI-CAD 案件)**: Card / CTA / Chip / Badge / IconButton などの required design parts が実機に存在し、正しい variant/state/application で使われていることを Control Plan 行にする。素の panel/text/button への退化は cosmetic でなく S2 以上の visual gap として扱う。
- **固定オラクル**([templates/41-fixed-oracle.yaml](templates/41-fixed-oracle.yaml)): 仕様化済み契約**だけ**を合否にする固定シナリオ集合。**製造開始前に凍結し(tag/commit)、収束ループ中は不変**(同一ヤードスティック。途中拡張は改善効果測定を無効にする)。
- **探索プローブ**([templates/42-exploratory-probes.yaml](templates/42-exploratory-probes.yaml)): §4.2 で `exploratory` 宣言した次元を観測するプローブ。**合否に混ぜない**(v1.1 受入の2層分離)。
- **等価規則**: オラクルは契約セマンティクスを比較し、付随表現(JSON 形状・キー綴り・ID 具体値)を `ignores` に明記する。**検査器自身にも L0 回避を課す**(v1.2: saga で検査器側 C2 が2度再発した)。
- **観測契約**: ワイヤ境界の無い題材(in-process/ライブラリ)は、固定シナリオを流して正規化ログを emit する I/F を設計者側で定義する(v1.2 (a))。
- **治具セルフテスト(フォワードでは必須)**: 採点治具は凍結前にセルフテストを通す(判定ヘルパ・等価規則の実装・デシリアライザの挙動を合成データで検証)。リバースと違いフォワードには原版が無く、**治具は未検証のまま初回採点に臨む**ため、検査器側ずるが初回採点に直接混入する(forward-01 CHEAT-F01-H001: JSON デシリアライザの暗黙日時変換で誤採点)。
  **治具は前提不成立を検査せよ(plm ECO-002)**: 検査器には「fixture がテストの前提を満たしているか」の precondition 検査を入れ、**治具の前提不成立を製品欠陥と区別して報告**させる(plm ECO-002 H004: suppress fixture の指定ミスを precondition 検査が『前提不成立』と報告し、製品帰属の誤診を防いだ)。
  **セルフテストの盲点(plm-v0)**: セルフテストが検証できるのは**比較器の正しさ**までで、「治具が仕様の**どの断面**を比較すべきか」(全集合か・フィルタ適用後集合か・どの座標系か)は検証できない(CHEAT-PLM-V0-H001: 仕様は「常に全所見」、期待プロファイルはゲート適用後集合——橋渡しフィルタが治具に無く偽 FAIL 5件で S-07 全滅。C2-in-harness の7例目)。処方: ①期待プロファイル側に「どの断面の値か」を宣言し、治具の比較対象と明示的に突合する ②初回採点の FAIL は製品帰属の**前に**治具帰属を必ず検討する(§6.4 `harness_bug` を先に消してから工場能力を疑う)。
- **較正(negative control。既知個体が存在する時点から必須)**: セルフテスト(静的)に加え、凍結前に検査一式を**既知個体**(変更前/旧 rev の As-Maintained 個体)へ動的に実行し、**事前に宣言した期待プロファイル**と突合する — 既存行=PASS(検査器が偽回帰を作らない)/ 新規・昇格行=FAIL(検査器が変更を実際に検出できる)。一致して初めて凍結。初回ループには既知個体が無いためセルフテストまで、rev2 以降と Phase 7 では必須。実証: rev2(S23 の v1 個体 FAIL を確認してから fresh 製造=「修正は BOM に宿る」の証明形式)/ forward-01.5(較正の初回実行が M03 の判定順序干渉 CHEAT-F015-H001 を凍結前に捕捉)。検査器ずる対策はこれで3段: ①治具セルフテスト(静的)②較正(動的)③等価規則の表現非依存(v1.2)。
- フォワード固有の役割: 原版が無いため、**期待値はすべて仕様から導く**。「期待値が書けない」ことが判明したオラクル行は、仕様の穴の検出器として Phase 2 へ差し戻す(オラクル設計自体が仕様検査になる)。

### 4.5 M-BOM([templates/32-mbom.yaml](templates/32-mbom.yaml))・Routing([templates/34-routing.yaml](templates/34-routing.yaml))
- M-BOM: 製造単位ごとに `interface_contract / invariants / acceptance_refs(→Control Plan)`。受入3条件=完全性(全数値・全フィールド・丸め・継承・既定値)/ 無矛盾性 / Control Plan 列(method-v1 §3)。
- Routing: 読む→核製造→表面製造→自己受入→As-Built 記録、の工程列。治具・測定器(受入ハーネス)は製品 BOM と同格に管理(Loop2.5,3,5)。

### 4.6 製造パッケージと完了ゲート G3 — ドライラン(転用仮説)
**製造パッケージ** = 製造装置に渡す全部。これ以外を渡さない:
```
仕様書(20-spec.md)               … 契約と不変条件の典拠
E-BOM / K-BOM / M-BOM / Control Plan / Routing(30〜34)
Work Order(40-work-order.md)     … 製造対象・必須受入・ずる報告義務の明示
観測契約(該当題材のみ)           … emit すべき正規化出力の形
```
**渡さないもの**: ブレスト・仕様策定の会話履歴 / 固定オラクルのシナリオと期待値 / 探索プローブ。

**ゲート G3(BOM 自己完結性のドライラン)**: 設計文脈を持たない fresh AI にパッケージだけを渡し、「製造に着手できるか。着手前に必要な質問をすべて挙げよ」と問う。**返ってきた質問=BOM の穴**。質問ゼロ(または残る質問が exploratory 宣言済み)になるまで補正してから製造へ。安価(製造1回より遥かに安い)で、webapi-01 の「seed 2/16」のような初回 blocker を前倒しで潰す狙い。

## 5. Phase 4 — 製造

### 5.1 隔離ファクトリ
- 製造は**クリーンな文脈のサブエージェント**(設計対話・オラクル・前ファクトリの成果と cheat を一切渡さない)に、製造パッケージのみ供与して行わせる。設計者 AI 自身は製造しない(文脈汚染。v2 運用で確立した方式)。
- **ずる報告の義務化**: 「BOM/K-BOM/Control Plan から導けず慣習で埋めた判断」を、実装を止めずに全件 [cheat-log 形式](cheat-taxonomy.md)で報告させる。Work Order に重点報告次元(silence-checklist の `exploratory` 行)を列挙しておく。
- 自己受入: Routing 記載のビルド+受入ハーネスを工場自身に実行させる(緑が納品条件)。**必須範囲には unit 行の test_vectors に加えて L1 API スモーク(プロセス起動+全エンドポイント正常系1本)を必ず含める** — unit 行だけだと表面の実行時欠陥が自己検査に映らない(forward-01: haiku が自己受入 17/17 緑のまま貸出エンドポイントが実行時全滅)。ただし**自己受入緑≠合格**(受入の被覆が捕捉を決める。webapi-01 で seed 6/6・自己 26/26 緑のまま blocker S7 が通過した)。**自己受入赤=stop/report**: 赤のまま「納品」しない(停止し manufacturing nonconformance として申告。40-work-order に明文化。forward-01 factory-06 が赤のまま納品した実測より)。
- **隔離パッケージの組成(実行環境付き題材)**: 依存物(node_modules 等)は**実体コピーでなくパッケージマネージャで再構築させる**(npm ci 等)。実体コピーは環境によってリンクが実体化し「再ビルドが import に反映されない=ビルド緑・テスト緑・実行は旧コード」の検出困難な偽陰性を生む(plm ECO-001 CHEAT-ECO01-F02-001 の実測。工場が正直に申告して発覚)。
- **中断引継(セッション上限・API エラー)**: 工場個体が完遂不能になった場合、**同一の隔離パッケージ内**で後任モデルへ引き継いでよい(隔離条件は変えない——設計対話・オラクル非開示のまま)。規律は3つ: ①引継 work order(完了範囲・残作業・自己受入の現況)をパッケージ内に残す ②後任が発見した前任コードの実欠陥の修正は blocker として cheat-log に記録する(隠すと差分帰属が壊れる) ③As-Built の `ai_model` に**混成個体として正直に記録**する(実測: plm-v0 factory-03 = opus 中断→sonnet 引継で 25/25 到達。blocker 4件は全て引継修正=前任の実欠陥で、正直記録が収束の帰属を保った)。

### 5.2 工場数の選択(受入経済性)
| ティア | 構成 | 何が得られるか / 何を諦めるか |
|---|---|---|
| 最小 | 1工場+固定オラクル+ずる報告 | 安価。**分散という沈黙検出器を失う**ため、単一ティアの慣習と設計者の暗黙知が一致したままの穴(C2)は通過しうる |
| **推奨** | 初回のみ 2–3 工場(別モデルティア)、収束後は1工場 | 初回分散で BOM の沈黙箇所を一括検出(ばらつきは沈黙箇所に集中する=N=3 再現)。別ティアを混ぜるのが要点——同一ティアは慣習を共有し、穴を揃って隠す(C2) |
| 研究 | 全ループ多工場+metrics 分解 | 決定性/正しさの二軸を完全測定 |

> **工場能力は単一軸でない(plm-v0 実測)**: 同一 BOM・同一パッケージで、parse 系契約は sonnet 優位・family 判定系は opus 優位と、能力差の**向きが領域で逆転**した(52-metrics: f-01 は parse 全滅/f-02 は parse 正解のまま family 判定が壊れる)。多工場の価値はモデルの序列づけ(「一番強いのを選ぶ」)ではなく**分散の観測**にある——「強いモデル1体」運用は、そのモデルが劣位な領域の穴をそのまま通す。ティア選択に迷う場合も、初回は**別ティア混成**を優先する。

## 6. Phase 5 — 受入・収束

### 6.1 測定
1. 固定オラクルを成果物へ実行(原版が無いので 2-way diff でなく**仕様期待値との突合**)。
2. UI-CAD 案件では **Visual Gap Analysis**([templates/43-visual-gap-analysis.md](templates/43-visual-gap-analysis.md))を作り、CAD(HTML mock + UI-IR + UI-BOM + Design System BOM)と実機を突合する。S1(欠落/構造)と S2(設計言語)は cosmetic ではなく blocker として扱う。
3. 探索プローブで未規定次元の値を採取(多工場なら分散、1工場でも値を記録)。
4. **差分帰属**(v1.1 3分類+検査器側+UI-CAD): `unspecified_bom_residue / specified_contract_miss(工場能力)/ exploratory_unspecified_surface / observer_* / design_system_part_missing / design_system_not_applied / display_contract_gap`。
5. metrics は raw 一致率単独で報告しない: `targeted_fix_success / blocker_diffs / new_unspecified_diffs` に分解([templates/52-metrics.yaml](templates/52-metrics.yaml))。

### 6.2 質問リストへの変換(フォワード固有の還流)
ずる報告・探索層分散・`unspecified_bom_residue` を、**ユーザーへの質問リスト**に変換する:
- 各件を「工場は X を慣習で A と埋めた(他工場は B)。これは仕様として固定すべきか?」の形にする。
- ユーザーの裁定は3択: **仕様に昇格**(REQ/仕様/K-BOM へ追記→次ループで `specified`)/ **探索のまま**(どちらでもよいと明文化)/ **対象外**。
- これがフォワードにおける**要求エリシテーションの第2ラウンド**になる——ブレストで思いつけなかった要求は、製造の分散として顔を出す。
- **規律**: 裁定で固定した次元は**次の収束ループの開始時に**オラクルへ昇格させる(study 途中でオラクルに足さない。プローブ→オラクル昇格の禁止)。

### 6.3 収束ループ
補正した BOM を **fresh な工場**(前ループの成果・差分・cheat 非開示)へ渡して再製造する。fresh であることが「修正がコーチングでなく BOM に宿った」ことの証明(webapi 2→3→0 の方式)。
**終了条件**: blocker 差分ゼロ かつ 新規 `unspecified_bom_residue` ゼロ(exploratory 宣言済みの分散は残ってよい)。このとき「**現固定オラクル被覆で未観測差分ゼロ**」と表現する(「完全」とは言わない)。

### 6.4 不具合発覚時の是正経路(直接ソース修正禁止)
受入中、またはユーザー確認で製造物の不具合が指摘された場合、設計者 AI は**ただちにソースを直してはならない**。ソースは最上流の真実ではなく BOM から製造された個体であるため、まず不具合を上流へ帰属する。

1. **不具合を記録**: 観測、再現手順、期待、実際、発見段(固定オラクル/探索/ユーザー確認)を cheat-log または是正ログに残す。
2. **帰属調査**: `仕様 → E-BOM → K-BOM → M-BOM → Control Plan/Oracle → Routing → 製造` を逆引きし、欠陥を分類する。
   - `spec_omission`: 仕様に要求が無い。REQ/仕様を改訂し、E-BOM/M-BOM/Control Plan/Oracle を同期してから再製造する。
   - `bom_sync_gap`: 仕様にはあるが E-BOM/M-BOM/Control Plan に落ちていない。BOM 同期を直して再製造する。
   - `oracle_gap`: 仕様/BOMにはあるが検査が無い、または深さ不足。オラクル・ファーストで検査を追加/較正してから再製造する。
   - `manufacturing_miss`: BOM と検査は十分だが工場が外した。工場能力または製造不適合として扱い、fresh 工場で再製造する。
   - `harness_bug`: 検査器側の誤判定。治具を修正・セルフテスト/較正し、既存測定値を再評価する。
3. **上流改訂を凍結**: 原因が仕様/BOM/検査器なら、該当成果物を改訂し bom_rev と oracle_ref を上げる。既存固定オラクル行はむやみに書き換えず、新規/昇格行として追加する。
4. **工場へ製造指示**: 改訂 BOM と Work Order だけを fresh factory へ渡す。直接ソース修正は、緊急封じ込めとして明示した場合を除き正式是正ではない。封じ込めを行った場合も、後続で必ず上流成果物へ逆流させて再製造し、As-Built に区別して記録する。

## 7. Phase 6 — 引き渡し・保守
- **As-Built**([templates/50-as-built.yaml](templates/50-as-built.yaml)): どの BOM 版を・どのモデルで・どのプロンプトで製造し、何点だったか。成果物ハッシュ・ずる参照込み(SLSA provenance の AI 製造拡張)。
- **Service BOM**([templates/53-service-bom.yaml](templates/53-service-bom.yaml)、概念は [s-bom-template.md](s-bom-template.md)): 表面部品ごとに `依存外部知識+版 / 変更時に再実行する検査(深さ付き) / 交換判断`。劣化イベント(外部ツール更新・デザイン改定・**AI モデル更新**)から影響部品と再検査深さを逆引きする(Loop6: 価値は被覆でなく絞り込み)。
- 納品物 = 成果物 + `bomdd/` 一式(BOM・オラクル・治具・As-Built・cheat-log)。**治具とオラクルは製品と同格**——次の改修時、これがそのまま回帰検査になる。

## 8. Phase 7 — 変更/是正オーダー(ECO/CAPA・candidate / validated ×2)

納品後の仕様変更・機能追加・欠陥是正の再入口。製造業の ECO/ECN と CAPA に相当する。「バグ=劣化→部品交換=再製造」を要求側の変更にも適用する。ユーザー確認で見つかった defect escape も、まず §6.4 の帰属調査を行い、原因が仕様/BOM/検査器なら本 Phase 7 で上流成果物を改訂してから再製造する。**変更の起点が外部劣化(Service BOM の DEG イベント — 調達部品の脆弱性・外部知識の改版)の場合も同じ再入口を使う**: 53 の逆引き→処置裁定→本 Phase 7、の連鎖(forward-03)。処置が調達交換のみ(コード製造なし)なら fresh 工場は使わず**設計者適用+全再認証**で閉じる(語彙と実測は [s-bom-template.md](s-bom-template.md)。調達処置の影響分析は BOM トレースでなく**パッケージグラフの実測**=アドバイザリ本文+scratch 交換検証を一次資料にする)。テンプレ: [templates/60-change-order.md](templates/60-change-order.md)+[61-impact-analysis.md](templates/61-impact-analysis.md)(影響分析)+[62-migration-oracle.md](templates/62-migration-oracle.md)(データ移行)+[63-diff-audit.md](templates/63-diff-audit.md)(不要改変監査)、手順: [prompts/phase7-change-order.md](prompts/phase7-change-order.md)。

> **検証状況(forward-01.5 + forward-02・N=2)**: ①forward-01.5(会員区分+v0.2 スキーマ自動移行): fresh 2工場(opus/sonnet)とも**回帰 23/23・変更 2/2・移行 4/4・不要改変 0**、影響なし予測が全的中。②**forward-02(2026-07-04・非自明影響グラフ)**: 依存グラフの中心(期限規則)への ECO(区分依存期間+効力=作成時点)で、fresh 2工場とも**回帰 25/25・変更 3/3・効力 4/4・移行回帰 4/4・不要改変 0**。影響なし予測は **under-inclusion 1 / over-inclusion 0** — 「Api 層 diff ゼロ」予測が配線1行で反証され、**2工場が事前申告つきで同一の1行に独立収束(blob 同一)=取りこぼしは影響分析に宿る**。E-BOM 粒度の予測(影響 1/12 品目)は的中し、外れたのは品目→unit 写像(実装の分布)——是正は手順 2 の実物点検として工程化済み。③**scale-01(2026-07-04・遡及測定)**: 部品数十の実題材(ViewPrism2: 37k 行・29 unit・実 ECO 34)で宣言影響集合 vs 実 diff を全数採点 — 実 under を持つ ECO **14/16(88%)**・同一ハブ unit(god-VM)を 10/16 で取りこぼし=**写像の外れは規模で系統誤差化する**(是正= 61 §1.2 実物点検+**§1.4 ハブ台帳**)。「unit 単体緑・組立乖離」型欠陥は記録 0 = end-to-end 受入はこの規模で破れず(組立検査は導入見送り — gap-analysis B3 実測済み)。candidate は外さない: 遡及測定は対策 regime 前の成績であり、N=1 題材・数十万行級・対策下プロスペクティブ ECO は未踏(FINDINGS §9)。

0. **変更前 baseline の凍結**: どの個体(**As-Maintained**。どの工場のビルドか)を改修するかを固定する。データを持つ題材では**実データ fixture**(+期待値 manifest)をその個体から採取する。
1. **変更要求の受理**: 変更を REQ の追加/改訂として書く(Phase 1 と同じ根拠精度の規律)。
2. **影響分析(トレース逆引き+影響なし予測の先行凍結)**: REQ→仕様節→E-BOM 部品(graph_edges 含む)→M-BOM unit→Control Plan 特性→オラクル行、を逆引きし**変更影響集合**を出す。**仕様の閉集合宣言を掃討する**: 新設節・新機能が旧本文の「〜の N 種で全て」「〜のみ」型の宣言(カーディナリティ・閉集合)を破っていないか grep する — plm ECO-002 で「X-* 7種で全て」が新設 X-GIT-001 で破れ、影響分析でなく**製造工場が検出**した(トレース逆引きは ID 参照を辿るが、散文の閉集合宣言は辿れない)。**As-Maintained 実物の呼び出しサイト点検を行う**(forward-02): トレース逆引きは BOM 上の影響品目を特定するが、**品目→unit の写像**(実装がその品目をどのファイル・呼び出しサイトに分布させたか)は BOM だけでは辿れない——部分改修では変更対象シンボルの全参照を As-Maintained 実物で確認してから影響集合を凍結する(forward-02 で「Api 層 diff ゼロ」予測が配線1行で反証・2工場が同一1行に独立収束=「実物が正」の影響分析版)。同時に**「影響なし」を反証可能な予測として製造前に凍結**する(回帰結果が予測の採点になる。外れ=under-inclusion として計上)。予測は「**ファイル単位の diff ゼロ**」級の反証可能な形で書く——外れ方が行単位で測定でき、under/over-inclusion が定量化される(forward-02 の主測定はこの書き方が可能にした)。Service BOM(外部知識劣化)と同じ装置を要求変更に使う——価値は被覆でなく絞り込み。粒度は §4.1 実測フィードバック参照(小規模では E-BOM 粒度+diff 測定で代替)。
3. **BOM 改訂**: 影響部品だけを改訂し `bom_rev` を上げる。変更分の受入(新オラクル行・test_vectors)を**再製造の前に**追加する(オラクル・ファーストは変更でも変わらない。既存オラクル行は不変=回帰のヤードスティック)。スキーマ変更を伴う場合は**移行専用オラクル+fixture**を立てる(62。データはコードと違い再鋳造で交換できない)。スキーマ変更が**無くても**、規則変更が既存データ上でどう振る舞うか(**効力=effectivity**: 遡及の有無・確定値と再計算の別)は fixture でしか検査できない——**効力専用オラクル**(62 §5=移行オラクルの一般化。forward-02 E01–E04: 既存データ保持/確定値の非遡及/新規のみ新規則/派生計算が確定値基準)を立て、効力(適用時点)は変更要求の受理時に REQ へ明文化する(例:「値は作成時に確定し遡及しない」)。治具はセルフテスト+**較正**(§4.4: 変更前個体に対し既存行=PASS・新規行=FAIL)を通してから凍結 tag を打つ。
4. **部分再製造**: 変更影響箇所**のみ**を fresh factory(設計対話・オラクル非開示)で改修/再鋳造する。部分**改修**(既存ソースへの変更)の場合は、変更前ソースの複製を設計者が事前コミットして **diff 基準点**にする(63)。非開示リストに**移行オラクル実装・fixture 期待値**を加える。未変更 unit は再利用。**部分再製造できない=粒度設計の失敗**として記録(§4.1 粒度規準へのフィードバック)。
5. **回帰+変更受入**: 既存固定オラクル全行(不変)+新規行+移行オラクルを実行し、**失敗5分類**で帰属する。**diff 監査(63)は機械実行できる**: BomDD-Plm(bomdd-lint)の R-052 を使う場合、register の対象 ECO に `diff_audit: { baseline: <凍結タグ>, allowed_paths: [<影響集合のパスプレフィックス>] }` を宣言し `--eco` で実行 — 実 diff が宣言影響集合の外にあれば R-052(error)が unnecessary modification を機械検出する(plm ECO-002 で自己適用により初稼働・手動 63 と同一結論) — `regression`(既存行の差分)/ `change miss`(新規行の差分)/ `data-preservation miss`(移行の差分)/ `unnecessary modification`(影響分析外への diff。format-noise / test-only / behavior-risk / contract-change に細分)/ `manufacturing nonconformance`(自己受入赤での納品=採点対象外)。
6. **記録**: As-Built 追記(新 unit の来歴・不要改変件数)・Service BOM 更新・cheat-log(変更で露出したずるは C6 系として記録)・metrics に ECO 行(52)。

### 8.1 unit 再編成(挙動保存リファクタ)ECO の規律 — scale-02/ECO-036 系列(全5段)で確立

ハブ unit(god ファイル)の解体など「挙動を変えずに M-BOM unit 構造を変える」ECO の型。
- **切り出し境界の三型**(実測で確立): ①**popup 型**(独立 UI を持つフィーチャ)= UI 契約ごと子へ移送
  ②**ストア型**(独立 UI なし・蓄積+外部受け渡し)= 実体のみ子へ移送し合成 UI プロパティはホスト残置
  (XAML/tests 無改変にできる)③**データフロー座標型**(Recompute 系パイプラインの構成要素)= **切らない**
  (切ると所有の移転でなく配線の複製になり unit 粒度規準に不適合)。切らない残余は**複合 unit として
  全品目を ebom_refs に張る**(多:1 写像の明示化= 「宣言=実装」で影響分析の系統誤差が消える。
  品目→unit 1:1 は理想でなく、多:1 の正直な宣言が正)。
- **移送表の必須列**: メンバ全数(行番号つき)・**消費者一覧**(XAML/tests/他 VM — 参照 grep は
  **tests/ を含む**)・**通知トポロジー**(誰が・どの範囲へ・何回通知するか)。
- **通知トポロジー 3 則**: ①通知不足は視覚欠陥(一括通知の肩代わりが分割で切れる)②過剰も欠陥になり得る
  (重複通知の冪等性は VM の真・view の偽)③正解は「**旧実装と回数・位置を同一**」。既存テストは
  プロパティ直読で通知漏れに盲目= **golden(人間)が通知面の唯一の検査器**。golden 所見は
  **旧版バイナリとの A/B** で回帰/既存バグを判別して初めて閉じる(挙動保存検証は既存バグの検出器を兼ねる)。
- **lineage(M 層)**: 系列完了時に旧 unit へ `status: superseded`+`split_by: [...]`+`superseded_by:` を宣言。
  **superseded unit は artifact.path を所有しない**(後継へ移譲 — 遡及採点の帰属も後継へ)。
  active 参照(FMEA の unit・CP)は再帰属し、履歴記載(register/50)は stale historical mention として残す。
- **工場運用の隔離規律**(工程事故の実測から): ①工場への指示に**入れ子委譲の禁止**を明記する
  ②工場を停止したら**全数停止を確認してから**設計者作業に着手する ③**同一ファイルを設計者と工場が
  同時に触らない**(diff 監査が汚染される)。
- golden 是正の read-across(レイアウト版): 共有コンテナを触る是正は**条件付き IsVisible の裏面**
  (そのとき表示されていない状態)も再検査対象に含める(表示中の面だけの目視は裏面崩壊を見逃す)。

## 10. この playbook 自体の検証(forward-01 ループ設計と実施結果)
本書は処方箋であり、研究としては次で測る:
- **題材**: 原版の存在しない小規模新規。**永続化(軽量 DB+マイグレーション 1 回)を意図的にスコープへ含める**ことを推奨——N=3 が全て in-memory で、データモデルの BOM 表現が最大の未検証領域のため([gap-analysis-v1.md](gap-analysis-v1.md) §A3/§C2)。ブレスト〜納品まで本書通りに実施。
- **測るもの**: ①G3 ドライランが検出した質問数と、製造後に残った `unspecified_bom_residue` 数の比(前倒し検出率)②マルチリーダー仕様監査の差分箇所が実際の分散箇所と一致するか ③K-BOM 事前抽出(§4.3)が事後の cheat 件数を減らすか(リバース時代の件数との比較)④収束ループ数。
- **ずる記録**: 本書の手順から導けず従来手法で埋めた箇所を、これまで通り cheat-log に記録する(playbook 自体が被験者)。

**実施結果(forward-01 / 01.5。2026-06)**: 上記の測定は1題材で完了した。①G3 ドライランが 7問+矛盾2件を製造前に検出し、製造後の `unspecified_bom_residue` は **0**(前倒し検出が機能)②G2 マルチリーダー監査は3体が曖昧 13–17 件を検出、共通指摘6件は全て「期待値が一意に書けない」級(Loop7 の「精度で割れる」が仕様でも再現)③K-BOM 事前抽出(慣習判断 40+ 件)の下で、3工場の cheat 計20件は全件が裁量宣言済み次元(宣言外への手伸ばし 0)④収束ループ数 **0**(opus/sonnet が rev1 で 0 差分)。詳細: [../FINDINGS.md](../FINDINGS.md) §7・証拠リポ [BomDD-LibraryLending-Sample](https://github.com/akiramei/BomDD-LibraryLending-Sample)・再現手順 [../docs/reproduce-forward-01.md](../docs/reproduce-forward-01.md)。

## 11. テーラリング(規模別の省略規約)
