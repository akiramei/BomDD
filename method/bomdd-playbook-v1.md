# BomDD Playbook v1 — フォワード・モード実践手順(prescriptive draft)

> **ステータス**: prescriptive / **単一題材検証済み(single-case validated)・一般化は未検証**。
> 本書は [bomdd-method-v1.md](bomdd-method-v1.md)(N=3 で実証済みの規則)を、**原版が存在しない新規開発**(ブレインストーミング→仕様→BOM→AI製造)へ移植した実践手順である。
> 検証状況: グリーンフィールド1題材(図書貸出 API・SQLite 永続化)で、初回製造(forward-01: opus/sonnet 20/20・未規定残渣 0)→仕様昇格の転移(rev2: fresh 23/23)→変更オーダー(forward-01.5: 回帰 0・移行 4/4・不要改変 0)まで一周した([../FINDINGS.md](../FINDINGS.md) §7、証拠リポ [BomDD-LibraryLending-Sample](https://github.com/akiramei/BomDD-LibraryLending-Sample))。**N=1 につき一般化はしない**(題材横断の再現が次の検証対象)。method-v1(実証された規則だけの薄い正規版)とはまだ統合しない。両者を混同しない。

## 0. リバース→フォワードの置換表(本書の設計原理)

method-v1 の装置はすべて「原版あり」で実証された。フォワードでは原版が無いため、各装置を次のように置換する。

| リバースの装置(実証済み) | フォワードで失われるもの | 置換(本書の処方) |
|---|---|---|
| 原版との 2-way diff(固定オラクル) | 比較対象の原版 | **仕様由来の固定オラクル**を設計者が**製造前に**書く(§4.4 オラクル・ファースト) |
| 原版=完全性の参照点(差分で未規定を発見) | 「何が未規定か」を教えてくれる相手 | ①[silence-checklist.md](silence-checklist.md) による既知の沈黙次元の事前掃討(§4.2)②**マルチファクトリ分散**=残った沈黙の検出器(§6。「ばらつきは BOM が沈黙する場所に集中する」N=3 で再現済み) |
| リバース工程の双方向トレース(原版証拠↔REQ) | 原版証拠列 | **要求根拠(rationale)↔REQ↔部品↔AC** の双方向トレース。原版証拠の代わりに「なぜその要求か」の根拠を、受入の深さ・許容差が決まる精度まで書く(§2。Loop7 昇格命題の転用) |
| マルチリーダー・リバース(複数AIが原版を読む) | — | **マルチリーダー仕様監査**: 複数AIが独立に仕様を読み REQ・不変条件・受入深さを抽出。差分=仕様の曖昧箇所(§3.2) |
| 製造装置の隔離(原版非開示) | 隠すべき原版 | 隔離対象の転換: ①**設計対話の文脈**(ブレスト・仕様策定の会話)を渡さない=「修正は BOM に宿ったか」のテスト ②**固定オラクルのシナリオ詳細**を渡さない=オラクル過適合の防止(§5.1) |
| ずる報告の義務化 | — | そのまま使う。さらにフォワードでは、ずる報告と探索層分散を**ユーザーへの質問リスト**に変換して要求へ還流する(§6.2。要求エリシテーションの装置になる) |

> 例外: 「既存資産を BOM 化してから拡張する」案件は、入口に method-v1 §1.5 のリバース工程をそのまま使い、以降を本書のフォワード手順に接続する。

## 1. 全体フロー

```
Phase 0  チャーター            … 題材・スコープ・予算・工場構成を固定
Phase 1  ブレインストーミング   … 要求台帳(REQ ledger)を作る
Phase 1.5 UI/UXデザイン(GUI案件) … モック自由創作 → 受入検査(refmodel/mock-lint)→ CAD化(UI-IR/UI-BOM+双子出力)。完了条件は治具 exit code
Phase 2  仕様化               … 仕様書+仕様完了ゲート(マルチリーダー監査)
Phase 3  BOM・工程設計        … E/K/M-BOM・Control Plan・Routing・固定オラクル・製造パッケージ
Phase 4  製造                 … 隔離ファクトリへパッケージのみ供与・ずる報告
Phase 5  受入・収束            … オラクル合否+分散観測/ユーザー指摘→差分帰属→質問リスト→BOM補正→再製造
Phase 6  引き渡し・保守        … As-Built・Service BOM
Phase 7  変更/是正オーダー(ECO/CAPA) … 仕様変更・欠陥修正→影響分析→部分再製造→回帰(納品後の再入口)
```

各フェーズに**完了ゲート**を置く。ゲートを通らず次へ進んだ箇所は、後段で必ず「ずる」として現れる(N=3 の一貫した観測)。

Phase 0(チャーター)では、**後から動かすとループ測定が濁る項目**(工場構成・収束ループ予算・役割と承認者・黒箱境界の種別)をこの時点で固定する。題材・スコープは `(仮)` 付きで仮置きしてよく、Phase 1 終了時(G1)に確定する。手順は [prompts/phase0-charter.md](prompts/phase0-charter.md)。

成果物の置き場所(推奨): 対象プロジェクトのリポジトリ直下に `bomdd/` を作り、[templates/](templates/) の番号体系(00=チャーター, 10=要求, 20=仕様, 30番台=BOM, 40番台=製造, 50番台=記録, 60番台=変更オーダー)で置く。規模が小さい場合の省略規約は §11 テーラリング。

## 2. Phase 1 — ブレインストーミング → 要求台帳

**目的**: 散漫な発想を、後段の BOM が参照できる **REQ 台帳**([templates/10-requirements.yaml](templates/10-requirements.yaml))へ収束させる。

**手順**:
1. 自由なブレスト(発散)。ここは形式不要。
2. 収束時に、合意した要求を 1 件ずつ REQ にする: `id / statement / rationale / rationale_precision`。
3. **根拠精度の規律**(Loop7 昇格命題の転用): rationale は「受入の深さと許容差を決められる精度」まで掘る。判定質問は一つ:
   > 「この記述だけから、検査の深さ(unit/L1–L3/golden)と許容差を決められるか?」
   - 例: 「AV同期が必要」(×: 許容差が決まらない)→「シーク後も音声は絶対時刻基準で、ズレは ±N ms 以内」(○)。Loop7 では全リーダーが「AV同期」と粗く読み「AACサンプル精度」を取りこぼした——**過小な根拠は粗い受入を招く**。
4. 各 REQ に**核/表面の仮分類**を付ける: その要求は要求文から実装を導出できるか(核)、外部ツール文法・デザイン・プロトコル慣習の転記が要るか(表面)。表面と分類した時点で K-BOM 調達(§4.3)の候補に積む。
5. **安全性の「実証」と「通知」を別要求に分解する**(viewprism2 ECO-052 還元): 「非破壊であること(実証=L3 検査)」と「非破壊だと表示すること(通知=UI 要求)」を 1 つの REQ に束ねない。woven 化された全体保証を機能ごとに UI 通知する要求はノイズ源になり、surface 置換時に黙って脱落しても誰も困らない(=要求が過剰だった証拠)。通知が要るのは不可逆警告のような機能固有リスクのみ。G2 監査の観点にも含める。
6. AI の役割: 聞き手・構造化係に加えて**反例生成器**(「この要求が破られるシナリオを 5 つ挙げよ」)。反例は FMEA(§4.2)の種になる。

**完了ゲート G1**: 全 REQ が rationale_precision の判定質問を通る。通らない REQ は「未精製」マークのまま Phase 2 へ持ち越してよいが、Phase 3 開始までに精製する。

## 3. Phase 2 — 仕様化

**目的**: REQ 台帳を、製造装置と検査器の両方が参照できる**仕様書**([templates/20-spec.md](templates/20-spec.md))にする。

### 3.1 仕様に必ず含めるもの
- **REQ へのトレース**: 仕様の各節がどの REQ を実現するか(双方向: REQ→節 / 節→REQ。宙に浮いた節・どこにも実現されない REQ を検出)。
- **不変条件の節**: 識別子採番・座標系/基準系・冪等性・順序など「関係の不変条件」(CHEAT-008、M-BOM 必須項目の前倒し)。
- **核/表面の宣言**: 各機能がどちらか。表面には出所(外部仕様・デザインシステム)を明記。
- **表示契約(原典ありの UI/帳票 surface)**: 原典に見えている提示フィールド、ファイル名、サムネイル、ラベル、アイコン、状態表示、空/エラー表示を全列挙し、各要素を REQ・E-BOM・Control Plan へトレースする。移植しない要素は理由付きで out-of-scope にする。これは「見た目」ではなく表示パリティ契約であり、仕様に無い表示要素は凍結後の工場・オラクルでは検出できない。
- **UI モック(Phase 1.5 の産物)**: GUI 案件では Phase 1.5(§1 全体フロー)で UI/UX モックの作成(自由創作)・受入検査(/bomdd-refmodel → /bomdd-mock-lint)・CAD 化(/bomdd-ui-cad — UI-IR / UI-BOM / trace map+[ui-ir-ui-bom.md](ui-ir-ui-bom.md) §12.1 双子出力)を仕様化の前に済ませ、表示契約・E-BOM 候補・Control Plan 候補の材料にする。未実施ならここで実施してから進む。UI-BOM は候補部品表であり、仕様・E-BOM へ昇格する前に未解決事項(裁定台帳の open)を確認する。
- **[silence-checklist.md](silence-checklist.md) の第1回掃討**: 仕様レベルで答えられる沈黙次元(日時表現・エラー語彙・丸め等)を `specified / exploratory / out-of-scope` で宣言するか、`deferred-to-phase3`(第1回掃討のみ有効な明示的延期)を付ける。**無言の先送りを残さない**(deferred は第2回掃討で必ず3択に解消)。

### 3.2 仕様完了ゲート G2 — マルチリーダー仕様監査(転用仮説)
Loop7 のマルチリーダー・リバースの転用。原版の代わりに**仕様**を読ませる:
1. 互いを知らない N 体(≥2、推奨3)の AI リーダーに仕様だけを渡し、独立に「REQ 一覧・不変条件・各 REQ の受入深さと許容差」を抽出させる。
2. 抽出結果を突合する。**差分が出た箇所=仕様が一意に読めない箇所**。Loop7 の実証では、振る舞いは全員一致し**根拠の精度**で割れた——仕様でも同じ割れ方を想定する(深さ・許容差の読みが割れた箇所を優先補正)。
3. **指摘は採用前に本文照合で真偽を裁定する**。差分は「仕様が一意に読めない箇所」だけでなく**監査者の誤読**でも出る(plm-v0: 低ティアの再監査が解消済み矛盾を3件再指摘= specified_contract_miss の監査版)。本文照合で誤読と判定した指摘は理由付きで棄却する——裁定なしの全件採用は、仕様を監査者の読み能力の最小公倍数へ劣化させる。
4. 補正して再監査。差分ゼロ(または残差に理由付き)で通過。

### 3.3 測定可能性ゲート G2' — MeasurementCapability 検査
[observation-schema-candidates.md](observation-schema-candidates.md) §4.9 を仕様ゲートに前倒しする:
- 各 REQ について「観測する深さ・治具・承認者が足りるか」を問い、`adequate / unmeasurable / under-specified-oracle / insufficient-depth / human-approval-required` を付ける。
- `unmeasurable` の REQ は**仕様未完成**(知覚系なら golden+許容差+承認者を、in-process なら観測契約をこの時点で計画する)。「測れない仕様は製造に渡せない」が規律。

### 3.4 凍結前原典パリティサインオフ(原典ありの移植・再現案件)
凍結後の製造装置は原典を見ないため、原典との比較でしか見つからない欠落(パリティ欠陥)は G2/G2' までに止める。表示パリティはその代表例。
1. 原典画面/帳票ごとに表示要素集合を列挙する。
2. 各要素を `仕様節 / REQ / E-BOM item または display_contract / Control Plan 特性` へトレースする。
3. 未トレース要素は `out-of-scope` 理由を承認するか、仕様へ追加して再監査する。
4. 未トレース要素ゼロを凍結前のサインオフ条件にする。ここを通らない BOM は製造へ渡さない。
5. **等価性はユーザー可視挙動を基準に取る — 「コード無し」と「挙動無し」を区別する**(viewprism2
   ECO-049 還元): 原典に明示コードが無くても、実行基盤(ブラウザ既定・OS シェル・ランタイム既定)が
   挙動を暗黙提供していることがある(実例: Chromium の EXIF 自動適用を「原典も未適用」と誤同一視し、
   移植先で横倒し表示が『仕様どおり』として 2 週間存続)。沈黙次元の裁定文には「コード上の事実」か
   「挙動上の事実」かを明記する(silence-checklist §10 の掃討行)。

## 4. Phase 3 — BOM・工程設計

**目的**: 仕様から製造パッケージ(§4.6)を組み上げる。method-v1 §2–§4 の実証済み規則をそのまま使い、フォワード固有の手順(K-BOM 事前抽出・オラクル・ファースト・ドライラン)を足す。

### 4.1 E-BOM([templates/30-ebom.yaml](templates/30-ebom.yaml))
- 核/表面で型分け(§0 第一原理)。核は `requirement_refs` 必須。表面は `external_source_ref` 必須。
- UI-IR / UI-BOM を前段に使った場合は `ui_sources` に参照を置き、UI-BOM 品目を `E-BOM item / display contract / Control Plan / K-BOM / traceのみ` のいずれかへ裁定する。HTML タグや装飾 wrapper をそのまま E-BOM 品目に昇格しない。
- **Design System BOM(candidate)**: UI-CAD が要求する Card / CTA / Chip / Badge / IconButton / SectionHeader / HelpBox / DragHandle などを、`35-design-system-bom.yaml` で E-DESIGN-* surface 部品と K-DESIGN-* 知識に分ける。UI-BOM surface item ごとに coverage matrix を埋め、`missing` のまま製造へ進めない。
- 木でなくグラフ: 横断(woven)部品に owner/consumers(CHEAT-002)。
- 各部品の `invariants` に、仕様の不変条件節を割り付ける。
- **粒度規準(candidate)**: M-BOM unit は「**独立に再製造・交換でき、単独で受入できる最小単位**」(Phase 7 の部分再製造単位と同一)。E-BOM 品目はその設計版(存在理由 1 つ・受入観点が閉じる単位)。迷ったら「この部品だけ fresh factory に再製造させられるか?」で判定する。**実測フィードバック(forward-01.5 → plm ECO-001 で改訂)**: 小規模 ECO では unit 粒度の影響集合が 3/3 unit に達し**絞り込み効果が出ない**ことがある(forward-01.5)。ただし plm ECO-001(同程度の小規模: 改修4ファイル)では **8 unit 中 2** に絞れた — 絞り込み効果は**規模でなく変更の局所性**に依存する。局所的な変更なら unit 粒度は小規模でも機能し、横断的な変更は E-BOM 部品粒度+不要改変 diff 測定([templates/63-diff-audit.md](templates/63-diff-audit.md))で代替する。
- **NFR の置き場所(candidate・三分割)**: 目標値=E-BOM の `nfr_targets`(測定可能な数値+条件)/ 実現手段・予算配分=M-BOM / 検証=Control Plan の NFR 特性行(原則 L3 execution measurement+許容差)。「速い」のままの NFR は G2' で unmeasurable=仕様未完成として弾く。

### 4.2 FMEA + 沈黙チェックリスト第2回掃討
M-BOM 確定**前**に(Loop4):
1. 部品ごとに故障モードを列挙(Phase 1 の反例を種に)。受入を狙い撃つ。
2. [silence-checklist.md](silence-checklist.md) を全行なめ、各次元を3択で**宣言**する:
   - `specified` — M-BOM/K-BOM に明記する(どこに書いたか参照を残す)
   - `exploratory` — 今回は固定しない。探索プローブ層で分散として観測する(合否に入れない)
   - `out-of-scope` — 理由付きで対象外
   **未宣言の次元がずるの温床**(N=3 で、分散は常に BOM が沈黙する場所に出た)。
3. 原典ありの UI/帳票 surface は、`表示要素集合・原典表示パリティ` 行を必ず `specified` または理由付き `out-of-scope` に解消する。`exploratory` のまま製造へ渡すと、原典パリティ欠陥を固定オラクルでは検出できない。

### 4.3 K-BOM 調達 — 暗黙知の事前抽出(転用仮説・フォワード固有)
リバースでは K-BOM は「ずるが出てから」整備した(Loop4)。フォワードでは**製造前に抽出**する:
1. 表面部品ごとに、設計用 AI へ問う: 「この領域を BOM 記載なしで実装するとき、**あなたが慣習で埋めるであろう判断をすべて列挙せよ**」(= ずる報告の事前実行)。
2. 列挙された判断を 1 件ずつ裁定: K-BOM に固定する / exploratory に置く / out-of-scope。
3. K-BOM 項目は `managed_knowledge`(箇条書きの判断)+ `risk_if_implicit`(暗黙のままだと何が分岐するか)を持つ([templates/31-kbom.yaml](templates/31-kbom.yaml)。webapi 実例の形式)。
> 根拠(転用元): K-BOM なし=全次元分岐 / あり=3工場ばらつき0(Loop4)。saga では締めた seed BOM が低能力ティアの取りこぼしも防いだ(saga-02 haiku 0/7)。

### 4.4 受入の設計 — オラクル・ファースト(フォワードの最重要規律)
**製造より先に**、設計者側で観測一式を作り凍結する:
- **Control Plan**([templates/33-control-plan.yaml](templates/33-control-plan.yaml)): 特性ごとに `深さ(unit/L1/L2/L3/G)・許容差・治具・承認者`。核=unit(exact, 丸め規定込み)。表面=領域別深さ(外部ツール≥L2、同期/座標系=L3、知覚=golden+許容差+承認者)。L0 文字列照合は既定不採用(method-v1 §5)。完全性が要る特性(丸め・全フィールド規定)には `test_vectors`(境界値・中間値・反例)を必ず列挙する(CHEAT-005 対策。「受入の被覆が捕捉を決める」)。NFR 特性は L3 execution measurement+許容差で行を立てる(§4.1 三分割)。
- **仕様が固定する数値は「値そのもの」を pin する**(viewprism2 ECO-050 還元): 機能の存在・意味論が
  検査面にあっても、数値既定(既定値・上限・刻み)はどの検査面にも載らないことがある(実例: 既定しきい値
  70 の仕様に対し工場仮置き 80 が混入し、surface 転写でさらに 90 へドリフト — 潜伏 2 週間・発見は実機の
  ユーザー認知)。既定値 pin の unit 1 行(構築直後の値=仕様値)を Control Plan に置く。surface の転写
  (コピー実装)は既存実装でなく**正本(仕様)と突合**する。
- **同等性受入(部品交換・A/B 選定)は「不変量の保存」と「誤差の系統性」の両輪で測る**(viewprism2
  ECO-054 還元): 順位等価(EQ-RANK 型)や集計指標はランダム誤差を吸収するが、**系統誤差は特定の
  入力クラスの体験だけを静かに劣化させる**(実例: adapter 選定 P-08 は速度と順位等価を実測したが、
  異フォーマット複製ペアの絶対スコアは測定外 — 経路非対称で系統的 -18 点が潜伏)。同等性受入には
  **同一入力の経路間距離を入力クラス別(フォーマット・周波数成分等)に測る対照群**を含め、恒久面は
  性質ベースのオラクル行(「同一入力に対する新旧/経路間距離 ≤ ε」または「基準経路との差分 ≤ ε」型)で
  凍結する。
- **共有機能再利用時の非機能契約の再確立**(ViewPrism2 ECO-058/062/064 還元・規則昇格基準〔rule of three〕充足 — 同一製品内 3 連鎖であり統計的独立反復ではない): 共通機能・共通経路・既存実装を別 surface へ再利用するとき、**移植元の結果意味論が維持されるだけでは受入完了としない**。移植先において、計算境界・入力件数境界・materialization 位置・仮想化の責務・キャッシュへの接触/無効化条件を**改めて宣言し、検査する** — 移植元と移植先ではデータ量・呼出頻度・表示寿命・キャッシュ条件が異なりうるため、**非機能契約はコード再利用によって自動継承されない**。機能単位検査および結果意味論だけを確認するオラクルは、**機能結果が正しく性能契約だけが破れる欠陥**を通過させうる(ViewPrism2 で 6,698MiB 消費・26 万件全走査・UI 仮想化と DB 取得制限の混同として反復)。検査型は control-plan「consumer-surface 接続行/非ブロッキング特性の二軸検査」・掃討観点は silence §17。
- **表示パリティ特性(原典ありの UI/帳票 surface)**: pixel-exact ではなく、表示要素集合の全 required elements が存在し意味一致することを Control Plan 行にする。自動で読めるなら L1/L2(アクセシビリティツリー/DOM/スクリーンショット解析)、知覚判断を含むなら G+承認者。
- **Design System coverage 特性(UI-CAD 案件)**: Card / CTA / Chip / Badge / IconButton などの required design parts が実機に存在し、正しい variant/state/application で使われていることを Control Plan 行にする。素の panel/text/button への退化は cosmetic でなく S2 以上の visual gap として扱う。
- **共通契約の受入は「契約×適用面」の全列挙とセットで設計する**(viewprism2 ECO-072〜075 還元・
  規則昇格基準〔rule of three〕充足 — 同一製品内反復): 複数 surface が共有する契約(ダイアログ
  視覚言語・命名規約・状態色・共有挙動)は、**契約の定義と原器だけでは検査可能にならない** —
  是正と検査が「指摘された面」に局所化し、同一契約の他の面が漏れる(同一言語の転写漏れが面を変えて
  4 回連鎖した実績)。共通契約は①**契約×適用面マトリクス**(横断: どの面がどの契約要素を負うか。
  面の新設時に行を追加)と②**面内チェックリスト**(面ごとの検査可能項目: グリフ/配色/タイポ/
  整列/レイアウト/状態切替/書式)の 2 成果物を**対で**持ち、契約に触れる変更はマトリクスの
  **該当列の全面**を検査対象にする(マトリクス=横断被覆・チェックリスト=面内被覆の直交)。
  既存面への遡及は lazy でよい(次にその面へ触れる変更で生成)。CAD 側実装例: ViewPrismUI
  03_dialog_language.md+screen_spec_template の visualContract 必須節。
  潜伏実績の追補(ECO-079): **健全な sibling は「基盤は機能している」証拠に誤読される** — 同型
  19 面中 17 面が契約(i18n 配線)適用済みでも、残 2 面の未適用は初出から潜伏した(サンプリングは
  全数列挙の代理にならない)。マトリクスの「適用面の全列挙」を省略する根拠に sibling の健全を
  使わない。
- **固定オラクル**([templates/41-fixed-oracle.yaml](templates/41-fixed-oracle.yaml)): 仕様化済み契約**だけ**を合否にする固定シナリオ集合。**製造開始前に凍結し(tag/commit)、収束ループ中は不変**(同一ヤードスティック。途中拡張は改善効果測定を無効にする)。凍結の証明は git commit/tag を標準とし、**git 不在/非使用環境では内容ハッシュつきアーカイブ等で代替**する(candidate・transfer-03 実測: Codex 担当者が git 未使用のまま進行し凍結タグ不履行・as-built に prompt_hash not-computed と正直記録 — ゲート実行可能性規則の凍結版)。
- **探索プローブ**([templates/42-exploratory-probes.yaml](templates/42-exploratory-probes.yaml)): §4.2 で `exploratory` 宣言した次元を観測するプローブ。**合否に混ぜない**(v1.1 受入の2層分離)。
- **等価規則**: オラクルは契約セマンティクスを比較し、付随表現(JSON 形状・キー綴り・ID 具体値)を `ignores` に明記する。**検査器自身にも L0 回避を課す**(v1.2: saga で検査器側 C2 が2度再発した)。
- **観測契約**: ワイヤ境界の無い題材(in-process/ライブラリ)は、固定シナリオを流して正規化ログを emit する I/F を設計者側で定義する(v1.2 (a))。
- **治具セルフテスト(フォワードでは必須)**: 採点治具は凍結前にセルフテストを通す(判定ヘルパ・等価規則の実装・デシリアライザの挙動を合成データで検証)。リバースと違いフォワードには原版が無く、**治具は未検証のまま初回採点に臨む**ため、検査器側ずるが初回採点に直接混入する(forward-01 CHEAT-F01-H001: JSON デシリアライザの暗黙日時変換で誤採点)。
- **高リスク検査治具の異系統独立受入検査**(transfer-04 還元): 治具セルフテスト(合成データ)と自己適合ゲートは自己検証であり、検査自身の被覆限界(FINDINGS §11.4 の 4 類型・silence §16)を残す。**高リスクの検査治具は、自己適合ゲートの成立後に、情報遮断した異系統検査官による read-only 受入検査を通す**。初回導入時に加え、**検査意味論の変更・主要リファクタリング・対象形式の変更・または検査沈黙インシデント発生後**に再実施する。運用型: 異系統モデルまたは別実装 / read-only / 差分と是正方針のみ開示 / 既存の検証節と裁定結果は非開示 / ずる語彙による敵対的観点の付与 / 事後に既存裁定と突合。**必須条件は「異系統性」**(モデル系列・統合層・プロンプト系統の独立)であり、別ベンダーであることは必須でない — 同一提供企業でも独立系統なら比較価値がある。
  **治具は前提不成立を検査せよ(plm ECO-002)**: 検査器には「fixture がテストの前提を満たしているか」の precondition 検査を入れ、**治具の前提不成立を製品欠陥と区別して報告**させる(plm ECO-002 H004: suppress fixture の指定ミスを precondition 検査が『前提不成立』と報告し、製品帰属の誤診を防いだ)。
  **セルフテストの盲点(plm-v0)**: セルフテストが検証できるのは**比較器の正しさ**までで、「治具が仕様の**どの断面**を比較すべきか」(全集合か・フィルタ適用後集合か・どの座標系か)は検証できない(CHEAT-PLM-V0-H001: 仕様は「常に全所見」、期待プロファイルはゲート適用後集合——橋渡しフィルタが治具に無く偽 FAIL 5件で S-07 全滅。C2-in-harness の7例目)。処方: ①期待プロファイル側に「どの断面の値か」を宣言し、治具の比較対象と明示的に突合する ②初回採点の FAIL は製品帰属の**前に**治具帰属を必ず検討する(§6.4 `harness_bug` を先に消してから工場能力を疑う)。
- **較正(negative control。既知個体が存在する時点から必須)**: セルフテスト(静的)に加え、凍結前に検査一式を**既知個体**(変更前/旧 rev の As-Maintained 個体)へ動的に実行し、**事前に宣言した期待プロファイル**と突合する — 既存行=PASS(検査器が偽回帰を作らない)/ 新規・昇格行=FAIL(検査器が変更を実際に検出できる)。一致して初めて凍結。初回ループには既知個体が無いためセルフテストまで、rev2 以降と Phase 7 では必須。実証: rev2(S23 の v1 個体 FAIL を確認してから fresh 製造=「修正は BOM に宿る」の証明形式)/ forward-01.5(較正の初回実行が M03 の判定順序干渉 CHEAT-F015-H001 を凍結前に捕捉)。検査器ずる対策はこれで3段: ①治具セルフテスト(静的)②較正(動的)③等価規則の表現非依存(v1.2)。
- **常設検査器の陽性対照(viewprism2 ECO-053 還元)**: 較正は凍結時の 1 回だが、常設検査器
  (validator・リンタ・pre-commit hook)は運用中に**沈黙**し得る — 台帳のキー改名 1 行が検査 3 本を
  無音で no-op 化し、「error/warn 不変」の検証が停止を健全と誤読した(negative-only 検証の罠。
  同期間に重複キー事故が 0-0 で素通し)。陽性対照のない検査器は「沈黙」と「健全」を区別できない。
  処方: ①常設検査器を**台帳スキーマ変更の影響分析・M4 対象**に含める(検査器は台帳の消費者)
  ②**陽性対照**(必ず検出されるべき合成フィクスチャ)を selftest として内蔵し pre-commit 等で毎回実行
  ③「読めなければ空」型の fail-soft を検査器に書かない(キー不在は明示エラー)。較正(negative
  control)の常設版であり、検査器ずる対策の**4段目(運用中の沈黙検出)**。
- **道具の個体参照(candidate・2例: plm ECO-001 / transfer-01 ExpenseDesk ECO-001)**: 検査器・
  サンプル生成・測定の道具が**参照する個体**(ビルド・バイナリ・DB)は製品と同格に構成管理し、
  個体の昇格・交代(deliverable 昇格・As-Maintained 交代)時は**道具の参照先更新を同期対象に含める**。
  個体参照の齟齬は緑・無警告のまま潜伏する(実例: ①依存の実体コピーで再ビルドが import に反映されず
  実行は旧コード=工場側 ②サンプル生成治具の ROOT が旧個体を指したまま、旧版の未知プロパティ無視
  仕様により指定が黙って落ち偽表示=検査側。検出はいずれも期待記述との突合)。処方: 道具は結果に
  **対象個体の識別(タグ/ハッシュ)を刻印**し期待個体と突合する。50-as-built に道具の指し先個体を
  記録する(harness_refs)。検査器ずる対策の**5段目(個体参照の検証)**。
- フォワード固有の役割: 原版が無いため、**期待値はすべて仕様から導く**。「期待値が書けない」ことが判明したオラクル行は、仕様の穴の検出器として Phase 2 へ差し戻す(オラクル設計自体が仕様検査になる)。

### 4.5 M-BOM([templates/32-mbom.yaml](templates/32-mbom.yaml))・Routing([templates/34-routing.yaml](templates/34-routing.yaml))
- M-BOM: 製造単位ごとに `interface_contract / invariants / acceptance_refs(→Control Plan)`。受入3条件=完全性(全数値・全フィールド・丸め・継承・既定値)/ 無矛盾性 / Control Plan 列(method-v1 §3)。
- Routing: 読む→核製造→表面製造→自己受入→As-Built 記録、の工程列。治具・測定器(受入ハーネス)は製品 BOM と同格に管理(Loop2.5,3,5)。

### 4.6 製造パッケージと完了ゲート G3 — ドライラン(転用仮説)
**製造パッケージ** = 製造装置に渡す全部。これ以外を渡さない:
```
仕様書(20-spec.md)               … 契約と不変条件の典拠
E-BOM / K-BOM / M-BOM / Control Plan / Routing(30〜34)
Work Order(40-work-order.md)     … 製造対象・必須受入・ずる報告義務の明示
観測契約(該当題材のみ)           … emit すべき正規化出力の形
```
**渡さないもの**: ブレスト・仕様策定の会話履歴 / 固定オラクルのシナリオと期待値 / 探索プローブ。

**ゲート G3(BOM 自己完結性のドライラン)**: 設計文脈を持たない fresh AI にパッケージだけを渡し、「製造に着手できるか。着手前に必要な質問をすべて挙げよ」と問う。**返ってきた質問=BOM の穴**。質問ゼロ(または残る質問が exploratory 宣言済み)になるまで補正してから製造へ。安価(製造1回より遥かに安い)で、webapi-01 の「seed 2/16」のような初回 blocker を前倒しで潰す狙い。

## 5. Phase 4 — 製造

### 5.1 隔離ファクトリ
- 製造は**クリーンな文脈のサブエージェント**(設計対話・オラクル・前ファクトリの成果と cheat を一切渡さない)に、製造パッケージのみ供与して行わせる。設計者 AI 自身は製造しない(文脈汚染。v2 運用で確立した方式)。
- **ずる報告の義務化**: 「BOM/K-BOM/Control Plan から導けず慣習で埋めた判断」を、実装を止めずに全件 [cheat-log 形式](cheat-taxonomy.md)で報告させる。Work Order に重点報告次元(silence-checklist の `exploratory` 行)を列挙しておく。
- 自己受入: Routing 記載のビルド+受入ハーネスを工場自身に実行させる(緑が納品条件)。**必須範囲には unit 行の test_vectors に加えて L1 API スモーク(プロセス起動+全エンドポイント正常系1本)を必ず含める** — unit 行だけだと表面の実行時欠陥が自己検査に映らない(forward-01: haiku が自己受入 17/17 緑のまま貸出エンドポイントが実行時全滅)。ただし**自己受入緑≠合格**(受入の被覆が捕捉を決める。webapi-01 で seed 6/6・自己 26/26 緑のまま blocker S7 が通過した)。**自己受入赤=stop/report**: 赤のまま「納品」しない(停止し manufacturing nonconformance として申告。40-work-order に明文化。forward-01 factory-06 が赤のまま納品した実測より)。
- **隔離パッケージの組成(実行環境付き題材)**: 依存物(node_modules 等)は**実体コピーでなくパッケージマネージャで再構築させる**(npm ci 等)。実体コピーは環境によってリンクが実体化し「再ビルドが import に反映されない=ビルド緑・テスト緑・実行は旧コード」の検出困難な偽陰性を生む(plm ECO-001 CHEAT-ECO01-F02-001 の実測。工場が正直に申告して発覚)。
- **中断引継(セッション上限・API エラー)**: 工場個体が完遂不能になった場合、**同一の隔離パッケージ内**で後任モデルへ引き継いでよい(隔離条件は変えない——設計対話・オラクル非開示のまま)。規律は3つ: ①引継 work order(完了範囲・残作業・自己受入の現況)をパッケージ内に残す ②後任が発見した前任コードの実欠陥の修正は blocker として cheat-log に記録する(隠すと差分帰属が壊れる) ③As-Built の `ai_model` に**混成個体として正直に記録**する(実測: plm-v0 factory-03 = opus 中断→sonnet 引継で 25/25 到達。blocker 4件は全て引継修正=前任の実欠陥で、正直記録が収束の帰属を保った)。

- **工場成果物の配置(candidate・transfer-02 還元 = T0-10 の昇格)**: 並行工場の成果物・途中経過は
  **main 作業域直下に置かせない** — 工場別の隔離ディレクトリ(`builds/factory-0N/`・worktree 等)で
  製造し、採用個体の昇格(main への統合)は明示手順で行う。統合前に **main 直下の未追跡ファイル検査**
  (git status クリーン確認)を Routing の統合 step に含める。実測: transfer-02 #6 = 並行 worktree
  工場の途中経過(不完全な複製)が main 直下に残存しマージ衝突・環境の削除保護で停止(T0 で 2/3 読者が
  「配置・昇格手順の不在」を静的検出→意図的見送り→実地顕在化の完全な証拠チェーン)。対照: transfer-01 は
  builds/factory-0N 分離で無事故。

### 5.2 工場数の選択(受入経済性)
| ティア | 構成 | 何が得られるか / 何を諦めるか |
|---|---|---|
| 最小 | 1工場+固定オラクル+ずる報告 | 安価。**分散という沈黙検出器を失う**ため、単一ティアの慣習と設計者の暗黙知が一致したままの穴(C2)は通過しうる |
| **推奨** | 初回のみ 2–3 工場(別モデルティア)、収束後は1工場 | 初回分散で BOM の沈黙箇所を一括検出(ばらつきは沈黙箇所に集中する=N=3 再現)。別ティアを混ぜるのが要点——同一ティアは慣習を共有し、穴を揃って隠す(C2) |
| 研究 | 全ループ多工場+metrics 分解 | 決定性/正しさの二軸を完全測定 |

> **工場能力は単一軸でない(plm-v0 実測)**: 同一 BOM・同一パッケージで、parse 系契約は sonnet 優位・family 判定系は opus 優位と、能力差の**向きが領域で逆転**した(52-metrics: f-01 は parse 全滅/f-02 は parse 正解のまま family 判定が壊れる)。多工場の価値はモデルの序列づけ(「一番強いのを選ぶ」)ではなく**分散の観測**にある——「強いモデル1体」運用は、そのモデルが劣位な領域の穴をそのまま通す。ティア選択に迷う場合も、初回は**別ティア混成**を優先する。

## 6. Phase 5 — 受入・収束

### 6.1 測定
1. 固定オラクルを成果物へ実行(原版が無いので 2-way diff でなく**仕様期待値との突合**)。
2. UI-CAD 案件では **Visual Gap Analysis**([templates/43-visual-gap-analysis.md](templates/43-visual-gap-analysis.md))を作り、CAD(HTML mock + UI-IR + UI-BOM + Design System BOM)と実機を突合する。S1(欠落/構造)と S2(設計言語)は cosmetic ではなく blocker として扱う。
3. 探索プローブで未規定次元の値を採取(多工場なら分散、1工場でも値を記録)。
4. **差分帰属**(v1.1 3分類+検査器側+UI-CAD): `unspecified_bom_residue / specified_contract_miss(工場能力)/ exploratory_unspecified_surface / observer_* / design_system_part_missing / design_system_not_applied / display_contract_gap`。
5. metrics は raw 一致率単独で報告しない: `targeted_fix_success / blocker_diffs / new_unspecified_diffs` に分解([templates/52-metrics.yaml](templates/52-metrics.yaml))。

### 6.2 質問リストへの変換(フォワード固有の還流)
ずる報告・探索層分散・`unspecified_bom_residue` を、**ユーザーへの質問リスト**に変換する:
- 各件を「工場は X を慣習で A と埋めた(他工場は B)。これは仕様として固定すべきか?」の形にする。
- ユーザーの裁定は3択: **仕様に昇格**(REQ/仕様/K-BOM へ追記→次ループで `specified`)/ **探索のまま**(どちらでもよいと明文化)/ **対象外**。
- これがフォワードにおける**要求エリシテーションの第2ラウンド**になる——ブレストで思いつけなかった要求は、製造の分散として顔を出す。
- **規律**: 裁定で固定した次元は**次の収束ループの開始時に**オラクルへ昇格させる(study 途中でオラクルに足さない。プローブ→オラクル昇格の禁止)。

### 6.3 収束ループ
補正した BOM を **fresh な工場**(前ループの成果・差分・cheat 非開示)へ渡して再製造する。fresh であることが「修正がコーチングでなく BOM に宿った」ことの証明(webapi 2→3→0 の方式)。
**終了条件**: blocker 差分ゼロ かつ 新規 `unspecified_bom_residue` ゼロ(exploratory 宣言済みの分散は残ってよい)。このとき「**現固定オラクル被覆で未観測差分ゼロ**」と表現する(「完全」とは言わない)。

### 6.4 不具合発覚時の是正経路(直接ソース修正禁止)
受入中、またはユーザー確認で製造物の不具合が指摘された場合、設計者 AI は**ただちにソースを直してはならない**。ソースは最上流の真実ではなく BOM から製造された個体であるため、まず不具合を上流へ帰属する。

1. **不具合を記録**: 観測、再現手順、期待、実際、発見段(固定オラクル/探索/ユーザー確認)を cheat-log または是正ログに残す。
2. **帰属調査**: `仕様 → E-BOM → K-BOM → M-BOM → Control Plan/Oracle → Routing → 製造` を逆引きし、欠陥を分類する。
   - `spec_omission`: 仕様に要求が無い。REQ/仕様を改訂し、E-BOM/M-BOM/Control Plan/Oracle を同期してから再製造する。
   - `bom_sync_gap`: 仕様にはあるが E-BOM/M-BOM/Control Plan に落ちていない。BOM 同期を直して再製造する。
   - `oracle_gap`: 仕様/BOMにはあるが検査が無い、または深さ不足。オラクル・ファーストで検査を追加/較正してから再製造する。
   - `manufacturing_miss`: BOM と検査は十分だが工場が外した。工場能力または製造不適合として扱い、fresh 工場で再製造する。
   - `harness_bug`: 検査器側の誤判定。治具を修正・セルフテスト/較正し、既存測定値を再評価する。
3. **上流改訂を凍結**: 原因が仕様/BOM/検査器なら、該当成果物を改訂し bom_rev と oracle_ref を上げる。既存固定オラクル行はむやみに書き換えず、新規/昇格行として追加する。
4. **工場へ製造指示**: 改訂 BOM と Work Order だけを fresh factory へ渡す。直接ソース修正は、緊急封じ込めとして明示した場合を除き正式是正ではない。封じ込めを行った場合も、後続で必ず上流成果物へ逆流させて再製造し、As-Built に区別して記録する。

2. **影響分析(トレース逆引き+影響なし予測の先行凍結)**: REQ→仕様節→E-BOM 部品(graph_edges 含む)→M-BOM unit→Control Plan 特性→オラクル行、を逆引きし**変更影響集合**を出す。**仕様の閉集合宣言を掃討する**: 新設節・新機能が旧本文の「〜の N 種で全て」「〜のみ」型の宣言(カーディナリティ・閉集合)を破っていないか grep する — plm ECO-002 で「X-* 7種で全て」が新設 X-GIT-001 で破れ、影響分析でなく**製造工場が検出**した(トレース逆引きは ID 参照を辿るが、散文の閉集合宣言は辿れない)。**As-Maintained 実物の呼び出しサイト点検を行う**(forward-02): トレース逆引きは BOM 上の影響品目を特定するが、**品目→unit の写像**(実装がその品目をどのファイル・呼び出しサイトに分布させたか)は BOM だけでは辿れない——部分改修では変更対象シンボルの全参照を As-Maintained 実物で確認してから影響集合を凍結する(forward-02 で「Api 層 diff ゼロ」予測が配線1行で反証・2工場が同一1行に独立収束=「実物が正」の影響分析版)。同時に**「影響なし」を反証可能な予測として製造前に凍結**する(回帰結果が予測の採点になる。外れ=under-inclusion として計上)。予測は「**ファイル単位の diff ゼロ**」級の反証可能な形で書く——外れ方が行単位で測定でき、under/over-inclusion が定量化される(forward-02 の主測定はこの書き方が可能にした)。Service BOM(外部知識劣化)と同じ装置を要求変更に使う——価値は被覆でなく絞り込み。粒度は §4.1 実測フィードバック参照(小規模では E-BOM 粒度+diff 測定で代替)。
3. **BOM 改訂**: 影響部品だけを改訂し `bom_rev` を上げる。変更分の受入(新オラクル行・test_vectors)を**再製造の前に**追加する(オラクル・ファーストは変更でも変わらない。既存オラクル行は不変=回帰のヤードスティック)。**要求自体が改訂され既存行の期待が無効になる場合も、行を書き換えない** — 当該行に superseded(改訂 ECO 参照)を記録して新規行を追加する(64 lineage の oracle 行版・candidate。実測: transfer-03 ECO-001 で in-place 書き換えが発生〔影響分析に明記・隠蔽なし〕→ ECO-002 で新規行+個別命名に自己改善 — 手順の明文があれば初回から踏めた蓋然性)。スキーマ変更を伴う場合は**移行専用オラクル+fixture**を立てる(62。データはコードと違い再鋳造で交換できない)。スキーマ変更が**無くても**、規則変更が既存データ上でどう振る舞うか(**効力=effectivity**: 遡及の有無・確定値と再計算の別)は fixture でしか検査できない——**効力専用オラクル**(62 §5=移行オラクルの一般化。forward-02 E01–E04: 既存データ保持/確定値の非遡及/新規のみ新規則/派生計算が確定値基準)を立て、効力(適用時点)は変更要求の受理時に REQ へ明文化する(例:「値は作成時に確定し遡及しない」)。治具はセルフテスト+**較正**(§4.4: 変更前個体に対し既存行=PASS・新規行=FAIL)を通してから凍結 tag を打つ。
4. **部分再製造**: 変更影響箇所**のみ**を fresh factory(設計対話・オラクル非開示)で改修/再鋳造する。部分**改修**(既存ソースへの変更)の場合は、変更前ソースの複製を設計者が事前コミットして **diff 基準点**にする(63)。非開示リストに**移行オラクル実装・fixture 期待値**を加える。未変更 unit は再利用。**部分再製造できない=粒度設計の失敗**として記録(§4.1 粒度規準へのフィードバック)。
5. **回帰+変更受入**: 既存固定オラクル全行(不変)+新規行+移行オラクルを実行し、**失敗5分類**で帰属する。**diff 監査(63)は機械実行できる**: BomDD-Plm(bomdd-lint)の R-052 を使う場合、register の対象 ECO に `diff_audit: { baseline: <凍結タグ>, allowed_paths: [<影響集合のパスプレフィックス>] }` を宣言し `--eco` で実行 — 実 diff が宣言影響集合の外にあれば R-052(error)が unnecessary modification を機械検出する(plm ECO-002 で自己適用により初稼働・手動 63 と同一結論) — `regression`(既存行の差分)/ `change miss`(新規行の差分)/ `data-preservation miss`(移行の差分)/ `unnecessary modification`(影響分析外への diff。format-noise / test-only / behavior-risk / contract-change に細分)/ `manufacturing nonconformance`(自己受入赤での納品=採点対象外)。
6. **記録**: As-Built 追記(新 unit の来歴・不要改変件数)・Service BOM 更新・cheat-log(変更で露出したずるは C6 系として記録)・metrics に ECO 行(52)。

### 8.2 拡張・撤去 ECO の規律 — 凍結面と到達閉包(viewprism2 ECO-048〜052 還元)

§8.1(unit 再編成)の姉妹。機能拡張と残骸撤去の ECO に固有の棚卸しを工程化する。
- **拡張は「凍結面の棚卸し」から**(ECO-048): 固定オラクルが凍結しているのは期待値だけではない —
  **識別子**(adapter id・this-build golden 値)と**インターフェース実装面**(オラクル内の fake が
  implements する契約)も凍結面である。正規の無効化機構(世代交代等)ですら凍結面と衝突するなら、
  等価効果の**非破壊面**で増築する(実例: default interface method・stale 条件の追加 — 3 例で確立した
  「凍結オラクル無改変と両立する拡張」の系譜)。着手前に「この変更が触る凍結面は何か」を列挙してから
  差し込み方を決める。
- **撤去は「呼び出しグラフの閉包」で**(ECO-051): インターフェースは参照を「生きているように見せる」。
  public API の生死は宣言の存在でなく**呼び出し元の有無**で判定し、撤去対象の閉包を grep 全数で確定する
  (実例: 直接参照の legacy を撤去した ECO が、サービス IF 越しに繋がる旧 UI 一式を漏らした)。
  残骸の実害は実測済み: 診断ノイズ・偽装備(仕様適合値が死者にのみ存在)・台帳乖離・欠陥の隠蔽。
  削除前に**受入観点対応表**(死者テストの観点 → 生存側カバーの突合。不足のみ移行)を作る(移行を
  確認してから削除)。
- **撤去は「マスクの棚卸し」も伴う — 残す経路が生きているかを実測してから撤去する**(ECO-087。
  ECO-077「導線変更は文脈パラメータの棚卸し」の 2 例目=昇格): 冗長な代替導線は、主経路の欠陥を
  **実用上マスク**する装置として機能していることがある。撤去すると潜伏欠陥が「新規欠陥の顔で」顕在化し、
  工程診断を誤らせる。実測: 候補値の並べ替えは ①↑↓ボタン ②行のドラッグ&ドロップの 2 経路を持ち、
  ②はコントロールのクラスハンドラが押下を handled にするため**未選択行では開始不能**という半死状態
  だった(①のマスク+選択視覚が「選択してから再ドラッグ」を成立させ、長期潜伏)— 原器準拠で①を撤去した
  瞬間に「並べ替え不能」として顕在化(GF-087-01)。処方: 導線を減らす/差し替える変更の影響分析に
  「**撤去される導線が運んでいたもの**」(文脈パラメータ〔ECO-077〕・欠陥のマスク〔ECO-087〕)の棚卸しを
  含め、**残る導線が本当に動くか**を probe/golden 基準へ明示的に載せる —「機能は既にある」は動作の証拠
  ではない。前項(呼び出しグラフの閉包)の**振る舞い版**: 閉包は「参照が死んでいないか」を見るが、本則は
  「残す経路が生きているか」を見る。
- **横断的関心の是正は「生成層の全列挙」で**(ECO-079): UI 文言・ログ・a11y 等の横断的関心は、
  宣言面(マークアップ直書き)・算出面(VM プロパティ)・データ面(共有 POCO)・システム定義名の
  複数層に散在する。1 層のみの是正・lint は残層の欠陥をマスクする(実例: XAML 168 サイト是正+
  lint 新設後、VM 層 111 リテラルが golden 流出= GF-079-01)。是正 ECO は、その関心に属する値が
  生成され、保持され、表示へ射影される立地を先に全列挙し、各立地に是正と、その立地に適した検出器
  (静的 lint・移行検査・実行時 pin 等)を持つ(撤去の到達閉包の、横断的関心版)。列挙対象は
  XAML・VM 等のソース表示層に閉じず、シード・永続データ・キャッシュ等の保持立地を含む。特に、
  システム定義の表示名のように UI ロケールから導出される値を表示文字列のまま永続データへ焼き込むと、
  ソース表示層を走査する静的 lint は DB 経由の表示値を構造的に見失う(実例: ECO-095= デフォルト
  作業スペース名がシード定数から DB 名として永続化され、言語切替に不追随)。判定規準は値の権威主体と
  再導出可能性であり、ユーザーが確定した値は永続化し、ロケールから再導出すべき値は意味上の識別子
  またはリソースキーを保持して表示時に解決し、ロケール変更時の再射影を pin する。
  **外延(VM 一時状態・ViewPrism2 ECO-104/106 N=2)**: 権威主体の規準は永続データだけでなく
  **VM 一時状態**にも適用する — Resolve 済み表示文字列の保持はロケール切替に追随できない
  (ErrorCode/キーを保持し表示時に現在ロケールで解決)。脆弱クラスは**「メインタブ面に常駐し、
  次のユーザー操作まで再解決機会がない表示」**(モーダル内一時表示・毎回再計算ラベルは良性)—
  同一アンチパターンが複数面で顕在化したら、悉皆是正でなく**実害条件で層別した脆弱クラスの定義**を
  CP と lint に残す(実測: 全数分布 50 箇所超を層別し lint 検査次元〔解決タイミング〕として恒久化=
  ECO-106/107)。資産の負債は「ある=対応済み」錯覚のマスクとして実害を持つ — 棚卸し(削除)は
  再発防止 lint とセットで一方通行にする(ECO-107= 未使用キー 640 件削除+3 次元 lint 同梱)。
- **真因が構造様式なら「同型全数走査」で閉包する — 様式の特定と掃討は別工程**(ECO-088/089): 欠陥の
  真因が特定の**構造様式**(反パターン。例: 折返しパネルを無限幅測定コンテナ直下に置く)だと確定したら、
  その様式の**同型全数走査**(grep・構造検索)を是正と同じ ECO 内で実施し閉包を確定する。「様式を特定して
  1 面を直す」ことと「様式を全面から掃討する」ことは別工程であり、前者だけで閉じると残存が別 ECO で
  再燃する。実例: ECO-088 は真因様式(折返しパネル×無限幅測定コンテナ)を特定・是正・CP 化までしたのに
  同型構造の grep 全数走査(1 コマンド・14 ヒット・数分)をせず、残存 2 面(タグパレット候補値行・作業
  タブチップ行)が翌日 ECO-089 として再起票された。走査コストは極小で、漏れの実害(再起票・再 fix・
  再 golden の 1 サイクル)より常に安い — **是正 ECO のクローズ条件に「真因様式の同型全数走査+閉包の
  記録」を含める**。「呼び出しグラフの閉包」(参照)・「生成層の全列挙」(層)に続く**真因様式の閉包**で、
  §8.3 read-across(GF-073 の面連鎖・GF-086-01 の dialog language マトリクス・ECO-077 の失効 read-across)
  の構造様式版。
  **追補①(閉包起票の実施様式・OBS-20260715-11 昇格= ECO-089/092/106/108)**: 同型欠陥の 2 面目
  以降は確立済みの是正型(例: キー保持+表示時解決/明示再計算)を列挙し、**全サイトへ「是正 or
  根拠つき良性確定」の悉皆処置を 1 ECO で行う** — 面単位 N 本より効率的で、良性判定も記録に残るため
  閉包の完了が監査可能になる(実測: ECO-108= 18 サイトを是正 13+様式変換 4+良性確定 3 で悉皆)。
  **追補②(水平展開の閉包・GF-079-01→ECO-108 N=2)**: 同型構造を持つ面が複数あるとき、**1 面の
  是正は残る面の「未是正の証明」でもある** — 是正時に同型面を列挙し、各面へ適用または非適用の根拠を
  残す。片面だけの是正は「面間非対称」として次の潜伏を作る(実測: WorkTab のみ是正され、同じ
  焼き込み構造の ImageTab が 4 日後に面間非対称〔作業タブ=英語/画像タブ=日本語〕として実機顕在化。
  昇格済み原則が水平展開の手を離れて再発した形 — §13 運用規則「規約は検査器で運ぶ」も参照)。
- **surface 置換の M4 同期対象は golden チェックリストまで**(ECO-052): 仕様本文・E-BOM・M-BOM に
  加えて **golden チェックリストの操作手順**も as-built へ同期する。旧 surface の手順のまま残った
  チェックリストは「読み替え実施」で項目が暗黙スキップされ、要求脱落をマスキングする(実例: モーダル
  時代の手順のままの G 行が、トレイ置換で脱落した 2 要素を 2 週間隠した)。チェック項目は読み替え
  不能な as-built 記述で維持する。
- **golden 器には機能手順だけでなく「CAD(モック)との並置突合」を観点として明記する**(ECO-056):
  機能観点のみの golden は文言・順序・意匠・導線の製造ロット逸脱を長期にわたり素通しする(実例:
  v1 CAD 定義への逸脱 6 点が歴代 golden を機能観点で通過し続け、ユーザーが実機とモックを並べた瞬間に
  一括顕在化 — 18 日超潜伏)。ECO-050/055「全数がどの検査面にもない」の視覚版。逸脱発見時の是正
  スコープは部分追随でなく**同一部品はモック形へ全面追随**(golden が並置突合で測られる以上、
  部分追随は再所見を生むだけ)。
- **golden 依頼には「見どころ」を仕込む(candidate・transfer-01)**: 人間確認の依頼文に、検出される
  べき陽性例(例: 上限超過で警告が付く実データ)と不変のはずの陰性例(例: 過去データの表示)を
  明示して渡す — 「眺めて OK」型の golden は検出力が判定者依存になる。傍証: 見どころの期待記述
  自体が道具側齟齬(旧個体起動)を即検出した(ExpenseDesk CHEAT-ECO1-H001)— 陽性対照は判定者
  だけでなく**準備側の齟齬**も捕捉する。ECO-052(観点明記)・ECO-056(並置突合)の系譜。
  **拡張(candidate・2例)**: golden は判定の場であると同時に、**準備段階(実機でのサンプル・デモ作成)
  が回帰オラクル被覆外の統合齟齬の最終検出器**として機能する — 2例: 道具の旧個体起動(transfer-01)・
  影響分析 under(transfer-02: CSV 上限列の取りこぼしを golden デモ準備が実機で捕捉。既存回帰行の
  期待値は旧仕様のため回帰では原理的に検出不能だった)。**ECO でも golden 準備を省略しない**。
- **CAD 未確定事項は「実装時裁定の確定手続き」で消化する**(ViewPrism2 ECO-099/100/103・N=3 —
  同一画面系列内であり統計的独立反復ではない): 設計正本(CAD)に未確定として台帳化された事項
  (挙動詳細・様式・確認 UX 等)は、mock 改版を待たず次の定型で確定する — ①依頼文の既定方向を
  **ECO 裁定表(項/裁定/根拠)**として確定・記録 → ②実装 → ③golden で新規面(copy・様式・近似)の
  裁定 → ④**read-across 材料(確定裁定一覧+新 copy 全文+実機キャプチャ)を設計正本側へ納品**して
  未確定台帳をクローズさせる。設計正本は実装リポから変更しない(納品が唯一の経路)。実測: 3 回
  連続で mock 改版ゼロ・介入は golden 裁定のみ(TAG-015/014/016)。クローズ条件に「read-across
  材料の納品」を含める。

### 8.3 欠陥是正 ECO のプローブと案選定の規律(viewprism2 ECO-054/055 還元)

- **差し戻し規律は診断を深める装置**(ECO-054): 是正プローブは着手前に「是正前不合格」を実測してから
  進む(§4.4 較正「新規行= FAIL」の是正版)。是正前プローブが不合格にならない場合、**黙って弱めず**
  コードに触らず診断を精密化する(再現マトリクス: 内容×解像度×アスペクト等の系統走査 → 機構特定 →
  より強い性質プローブ)。実例: 不再現→3 round で発散源(JPEG 側・高周波のみ顕在化)を特定し、
  設計選定まで導いた。
- **新設面の R5 は「API スタブで赤を実測」**(ViewPrism2 ECO-099/100/103・同一系列 3 反復):
  是正前不合格の規律は欠陥是正だけでなく**機能拡張 ECO** でも成立する — 新契約のプローブを先に
  生成し、VM/API に**空実装スタブ**を置いて実行して「契約の不在」を赤で実測してから作る
  (実測: スタブ 10/12・6/7・8/8 赤 → 実装で緑転。スタブでも通る消極ガード pin は区別して記録)。
  §4.4 較正「新規行= FAIL」の feature ECO 適用形。
- **予防ゲートの較正は陽性対照で**(ECO-080): 赤プローブ(是正前不合格)は欠陥是正の較正手段であり、
  現状違反ゼロで新設する**予防ゲート**には赤が原理的に存在しない。合成違反の陽性対照(検出すべき
  違反を注入し検出を固定)+対象列挙の空振りガード(列挙数の下限等)なしの「初回緑」は、意図どおり
  動作と空振りを区別できない。対応表: **是正ゲート → 赤プローブで較正 / 予防ゲート → 陽性対照で較正**
  (control-plan「検査の対照 3 種」の適用分岐。実測: 全ファイル lint 一般化が合成違反 3 件検出+
  列挙≥20 ガードを最初から同梱し、初回緑を動作の証拠にした)。N 蓄積(ECO-081/083・
  2026-07-14): HangDump 恒久宣言・fail-fast 監視とも陽性対照(人工ハング/注入例外)で発火を
  実測してから出荷(2〜3 例目)。さらに較正は**手段選定も裁く** — 人工ハングの陽性対照が候補手段
  (協調キャンセル型 --timeout)を「届いても切れない」と反証し、効かない安全装置の正本化を阻止した
  (control-plan「安全装置の較正」の軸)。
- **沈黙故障(報告されない停止)の診断は fail-fast 化を先に入れる**(ViewPrism2 ECO-083):
  例外が誰にも帰属しない故障(ディスパッチループの静黙死=テスト全緑のまま無限待ち)は、真因調査に
  先立ち fault 監視 → **原因例外全文つき即時失敗**を装備する — 次の発火が診断データになる(調査の
  加速装置)。実測: 監視導入コミット直後の実発火が原因例外+完全スタックを顕在化し、真因確定
  (依存ライブラリ既定値の再初期化レース)の決め手になった。「沈黙する故障は、まず喋らせてから治す」。
- **間欠欠陥の再現ループは発火時に即停止し、観測データを先に退避する**(ViewPrism2 ECO-082/083 —
  不可逆観測データの保存義務〔1 例で本文化可・§13 運用規則〕の適用): 発火時の実行ログ・ダンプ・
  TestResults は後続 run が同一パスへ**不可逆に上書き**する=遡って回復できない。実測: 証拠喪失 2 回
  (発火 run の詳細が後続 run に消され再捕獲を要した)。ループの停止条件は「対象の症状」だけに絞らず、
  非対象の fail も証跡を退避してから続行する。
- **是正案はプロトタイプの実測比較で選ぶ**(ECO-054): 起票時想定案は実測で欠陥が判明しうる(想定案
  pngOnly は逆向き非対称を新設すると実測で判明し不採用)。候補案を同一データ(実ペア+合成対照)で
  比較した表を order に残す(ECO-038「真因構造そのものを消す案を優先」の実測版)。
- **宣言された増分は追跡される場所に置く**(ECO-055): 「後で結線する」型の工程宣言をコード内コメントに
  置かない — **コード内 TODO は台帳ではない**(実例: 「M2 で結線する」が台帳外で 18 日潜伏)。宣言は
  order/register の追跡面へ。納品 diff の受入で工程宣言コメント(TODO/後続結線)を掃討 grep する。
- **既知欠陥様式は自分の新規実装のセルフレビュー面にも適用する**(ECO-055): 既知様式
  (cheat-log/taxonomy)を、その教訓を知る是正実装自身が再発させた実例(転送殻の通知漏れ ECO-038 →
  GF-055-01)。既知様式チェックは他者検査面だけでなく**書き手の自己受入面** — 可能なら様式の機械 fact
  (通知検査等)を新規コードにも走らせる。
- **UI に触れる製造・是正はセルフゴールデン(出荷前の原器並置)を経て golden に出す**(viewprism2
  ECO-072〜075 還元・規則昇格基準〔rule of three〕充足 — 同一製品内 5 連鎖であり統計的独立反復では
  ない): 機械受入(build/test/validator)は原器適合を 1 ピクセルも検査せず、golden は human gate の
  ため、放置すると「実装と原器を初めて並置する人」が承認者になる。視覚原器(mock captures)の同梱は
  「参照可能」を作るが「参照強制」を作らない — **答えがすべて原器に存在する転写漏れ**が指摘駆動で
  4 面連鎖した(GF-072-01→GF-073-01→03→04→07。指摘後の是正は毎回正確=理解能力でなく突合工程の
  欠落)。処方: UI サーフェスに触れる fix は、golden 提示の**前**に是正者自身が各サーフェスを原器と
  並置し(headless レンダリング/実機スクリーンショット)、差分を「裁定済み許容差分(裁定記録を引用)
  /転写漏れ」に**全列挙分類**し、**転写漏れ 0 で出荷**する。視覚 probe は原器のチェックリスト
  (§4.4 共通契約の 2 成果物)から fix 時に**先行生成**する(golden 所見の後追いで pin しない)。
  製品プロファイル実装例: ViewPrism2 R7+/eco-fix 手順 3.5。昇格後の初回実運用(ViewPrism2
  ECO-077・2026-07-13)で機能を実測: fix 内の原器並置が転写漏れ 1 件(選択コントロールの既定枠の
  浮き)を golden 前に検出・是正し、差分 4 件を全列挙分類・転写漏れ 0 で出荷 → golden 一発合格・
  所見 0(有効性 2 例目)。
  **並置の単位は「触った要素」でなくサーフェス全体 — 差分は 3 分類で全列挙する**(ECO-086/087・
  範囲規定の欠落を埋める追補であり新規則ではない): 並置を今回の新設・変更要素に限ると、同じ面に同居する
  **既存乖離**(過去 ECO の転写漏れが凍結したもの)を構造的に素通しする。差分は ①**裁定済み許容差分**
  (裁定記録を引用)②**未裁定の既存乖離**(本 ECO のスコープ外だが「見なかったこと」にはせず **R3 分離
  起票**)③**転写漏れ**(本 ECO で是正)の 3 分類へ全列挙する。実測: 新設要素のみ並置した ECO-086 は
  転写漏れ 0 で出荷したが、golden で面全体を見た maintainer が 6 面中 2 面の全面乖離(過去 ECO の部品単位
  転写に由来)を検出 → 分離起票(ECO-087)で面全体並置 → 転写漏れ 0・golden 視覚所見 0。既存乖離は
  golden の場で発見されるより**起票前に分類**されている方が安い(ついで修正禁止の裏返し — 分離起票が
  正しい処置)。
  **並置はジェスチャを検査しない — 視覚契約に「操作の起点」次元を持たせる**(ECO-087): 静的並置
  (headless レンダリング/スクリーンショット)は静止画の突合であり、**ドラッグ起点・ホバー・キーボード
  経路**は原理的に被覆外。実測: 面全体並置で転写漏れ 0 を確認して出した golden の残所見が、視覚でなく
  対話ジェスチャ(未選択行でドラッグ開始不能)だった。原器がドラッグハンドルを描くのは**操作起点の明示**
  でもあり、視覚だけ転写して操作起点を転写しない忠実化は半分の忠実化になる。視覚契約の検査次元に「どの
  要素がジェスチャを受けるか」を明示行として持たせ、並置で検査できない次元は実ポインタ注入 probe へ

  **実施手段の拡充(ViewPrism2 ECO-099〜103・撮影ハーネスの資産化)**: headless+実描画(Skia)の
  撮影ハーネスを正式資産(M-BOM unit)として track すると、R7 並置(原器 capture との突合)を
  是正者自身が自律実行できる — ホバー・Flyout・スクロール等の状態も撮影可能で、実機 golden は
  「要確認差分の裁定」に集中する分業が 3 便で安定(M-CAPTURE-HARNESS-052)。工具の track は

  あっても同一出力先の再実行が未検証のまま track され 2 回目 NRE。control-plan「正本化前検査」参照)。
- **golden の実施条件に非既定構成を含める**(ECO-079): 既定構成のみの golden は「既定値と一致する
  欠陥」(既定ロケール下の i18n 未配線等)を構造的に見逃す — 歴代 golden 全てが既定単一構成で実施され、

## 11. テーラリング(規模別の省略規約)

**権限分離規則(実証済み N=2 — transfer-01 T0 還元・transfer-01/02 で自発適用)**: 本表と prompts/ 正典の関係は優先順位でなく権限の分離である — **本表は「何を・どの規模で」を決め、prompts は「どう実行するか」を決める**。prompts 内に現れる規模既定(監査体数・工場数・成果物の分割・ゲートの要否)は、本表の該当セルが上書きする。縮退の採否は Phase 0 で裁定し、charter に記録する(記録なき縮退は無言の先送りと同じ扱い)。実測: transfer-01(M規模・縮退なし)/ transfer-02(M規模)/ transfer-03(**S規模** — GPT-5.5/Codex・環境制約〔隔離サブエージェント機構なし〕を縮退経路で吸収)— いずれも charter に裁定記録(**N=3・ベンダー横断**)。

**規模判定基準(candidate・transfer-03 還元)**: 規模区分は判定質問で決める — REQ 数見込み(〜5=S/〜15=M/それ以上=L)・独立責務の数・永続化/移行の有無・利用者/承認者の構成。実測: **同クラス題材で M(transfer-01/02)と S(transfer-03)に割れた** — 明文基準がないと判定は担当者裁量で分散する。**縮退の根拠に環境制約(隔離サブエージェント不可等)が含まれる場合は、規模を名目にせずそれ自体を charter に明記する**。

**リスク等級(candidate・外部レビュー提案 2026-07-10)**: 規模判定 S/M/L とは**直交する**リスク等級を設ける。リスク判定では、重大度、可逆性、観測可能性、外部依存、変更頻度を確認し、その根拠を charter へ記録する。REQ 数または変更量が小さくても、認証、課金、権限、安全性、不可逆なデータ移行などでは高リスクとなり得る。**一つ以上の軸が高リスクと判定された場合、規模が S であっても測定器を規模理由だけで縮退してはならない** — G2 の自己申告化、必要な golden の省略、陽性対照の省略などを行う場合は、明示的な waiver、承認者、期限および代替統制を必要とする。

実用では「重いから使わない」が最大の失敗モード。次の原則で削る:
**削ってよいのは文書の分割粒度。削ってはならないのは測定器**(オラクル・ファースト / 工場隔離 / ずる報告 / 沈黙掃討)——測定器を削った運用はずるが見えなくなり、BomDD ではなくなる。

## 13. 手順の工具化 — 成熟度ラダー(viewprism2 ECO-048〜053 還元)

手順は書いてあるだけでは実行を保証しない。散文の手順・規約は「読み替え」「同期忘れ」「暗黙スキップ」で
静かに死ぬ(実測 3 例: 数値既定がどの検査面にもなく仮置きが潜伏 / golden チェックリストが旧手順のまま
読み替えスキップ / 検査器の語彙ミラー「変更時は同期」コメントが守られず検査 3 本が無音死)。

**成熟度ラダー**: ①暗黙知 → ②散文手順 → ③入口コマンド化(skill/コマンドが順序・停止点を強制)→
④機械検査(validator・hook・pin テストが逸脱を検出)→ ⑤完全自動化(人間ゲート不要のもののみ)。

- 昇格判断= **頻度 × 逸脱実績 × 影響**。バッドプラクティスは「高頻度・逸脱実績ありの手順が②に滞留する」こと。
- ⑤が正解とは限らない: 判断が本体の手順(裁定・還元・golden)は③+④が上限 — コマンドは判断を
  自動化せず、**順序と停止点を強制**する(ViewPrism2 の /eco-file・/eco-fix・/eco-accept、本リポの
  /lesson-promote が実例)。
- **方法論自身の運用手順も対象**(自己適用): 還元手順が「前例コミット様式の暗黙知」に滞留していた —
  /lesson-promote への昇格が本節の初出適用。§12 の prompts/(正典)→ skill 化 adapter 層の方針とも整合。

運用規則(Phase 1.5 編入還元・2026-07-08):
- **非対称の点検(系列比較)**: ラダー段は工程系列(開始/変更/保守)ごとに比較する。片系列だけ③に
  登った状態は「入口の曖昧さ」として現れる — 検出法は**実動線(開発者が実際に踏む手順)と状態機械の
  行の突合**(実測: 変更系= /eco-file が③・開始系はモック作成が bomdd-next 判定表の外 → Phase 1.5 編入)。

  事後復元不可の観測データ(pixel capture 等)は 1 例でも本文化してよい — 判断基準は「遡って回復

  GU1–GU6 PASS+双子出力の存在)。
- **標準手順の改訂には波及監査を付ける**: 点検先は最低 2 面 — (a) 状態機械・判定表の参照(正典を
  差し替えても誘導が deprecated を指し続ける)(b) オンボーディング作業ログ・実例(実測: Phase 1.5

  1 か月残存)により昇格裁定済み** — 実装は方法論リポ側 lint として次の機会に立てる(3 例目を
  待たない。裁定 2026-07-08)。
- **ゲート実行可能性規則(実証済み N=2 — transfer-01 T0 還元・transfer-01/02 で自発適用)**: ゲートで必須とされる検査は、
  配布物(kit)・環境内に**実行手段・代替手順・省略条件のいずれか**が存在しなければならない。
  存在しない場合、そのゲートは「未充足」ではなく**方法論欠陥として停止**する(担当者の推測で
  埋めさせない)。実行体のないゲートは「沈黙」と「健全」を区別できない — 常設検査器の陽性対照
  (§4.4 検査器ずる対策 4 段目)のゲート版。実測: T0-01 = PLM Gate が工具未同梱のまま必須条件として
