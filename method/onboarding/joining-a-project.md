# 進行中プロジェクトへの参加(人間向け・操作チェックリスト)

対象: **すでに BomDD で進行中のプロジェクトに途中参加する人間**(裁定者・技術確認者・golden 承認者のいずれか、または未定の人)。
新規に始める人は [new-project-checklist.md](new-project-checklist.md)、既存リポを BomDD へ移行する人は [existing-project-migration.md](existing-project-migration.md)。
これは読み物ではなく**操作チェックリスト** — 30 分は目安であり、合格条件は §0 の観測可能な 4 状態。
協働の作法(頼み方・報告の読み方)は [working-with-ai.md](working-with-ai.md) — 本ページの後に読む。

> 出典: onboarding-t0-01(fresh 読者 2 体が独立に「参加動線の不在」で停止・2026-07-18)。

## 0. 終了条件(この 4 つにチェックが付けば参加完了)

- [ ] 進行中の ECO または裁定案件を **1 件特定できた**
- [ ] `/bomdd-next` の結果を**人間の言葉で説明できた**
- [ ] 自分が**裁定者/技術確認者/golden 承認者のどれか**を確認できた
- [ ] **判断できない場合の差し戻し先**が分かった

## 1. 一歩目 — 何を開いて、どこに打つか

1. 製品リポ(チームから URL をもらう)を clone し、そのフォルダでターミナルから `claude` を起動する(Claude Code — チャット欄が出る対話ツール)。
2. チャット欄に `/bomdd-next` と打つ。
3. **成功の判定**: 「現在の工程(Phase または運用 regime)・次の一手・人間がやることがあるか」が返ってくること。
   - スキルが見つからない等のエラーが出たら環境未設定 — チームの技術メンバーに「product-profile のスキルは入っていますか」と聞く(これは正規の質問であり、あなたの落ち度ではない)。
4. git 側から自分の目で確認したい場合: `bomdd/60-change-register.yaml`(変更台帳)を開く — **一番下の行が現在地**(進行中・直近の変更)。

## 2. 自分の役割を確認する

役割は 3 つ(1 人で兼ねてよい):

| 役割 | 何を判断する人か |
|---|---|
| **裁定者** | 業務上の判断(どちらの仕様にするか・その挙動で業務が回るか) |
| **技術確認者** | 技術契約の保証(§4 の右列 — CLI 等の非知覚媒体で特に必要) |
| **golden 承認者** | 実機を触って合格基準を確認する人 |

- 誰がどの役割かは **charter(`bomdd/00-charter.md`)** に書いてある。
- 自分が載っていない場合、**載せること自体が裁定** — 既存の裁定者に「私を◯◯として charter に追加してください」と依頼する(AI に頼めば変更の起票からやってくれる)。
- 案件の媒体によって出番の**位置**が変わる: [working-with-ai.md §2.5](working-with-ai.md)。

## 3. 裁定の返し方 — 結果は 4 値(拒否権はあなたの正規の権利)

| 裁定結果 | 意味 |
|---|---|
| **承認** | 提示案のとおり進める |
| **条件付き承認** | 条件を 1 行添えて進める(例:「(b) で。ただし既存データは変換しないこと」) |
| **差し戻し** | どの案にも同意できない — **これは例外ではなく正規の裁定結果** |
| **判断範囲外** | 自分の役割の外 — 正しい裁定者へ回す(誰か分からなければ charter を見る/AI に「これは誰の裁定か」と聞く) |

**差し戻しの様式**(この 3 点で足りる):

1. 判断できない点
2. 不足している証拠
3. 再提出時に示してほしいもの

これを明記しない運用では、裁定者は事実上「提示された選択肢のどれかを承認する人」になってしまう。推奨に従うだけの裁定が続くと感じたら、それ自体を差し戻してよい(「業務観点の差が分かる形で選択肢を出し直して」)。

## 4. CLI など「見た目のない」案件 — 二者承認(1 人で背負わない)

非知覚媒体(CLI・API 等)では人間の出番が製造の**前**(凍結前レビュー)に移る。ここは
**業務挙動は裁定者が、技術契約は技術確認者が承認し、両方が揃うまで凍結しない**。
同席ではなく**分担** — 非技術者が 1 人で最終 OK を出す構造にしないため。

| 裁定者が見るもの(業務挙動 — 全部生活語で確認できる) | 技術確認者が見るもの(技術契約) |
|---|---|
| どの入力を与えるのか | exit code の網羅性 |
| 何が表示・生成されるのか | 権限境界 |
| 正常時と失敗時に、利用者は何を認識できるのか | 不可視の副作用 |
| ファイルやデータは変更されるのか | 中断時の整合性 |
| 再実行すると何が起きるのか | ロールバック・冪等性 |
| **業務上、その挙動でよいか** | 実装と仕様の対応関係 |

AI には「裁定者向けに、左列の形で凍結前レビュー資料を出して」と要求してよい(出せないならそれ自体が差し戻し事由)。

## 5. 報告の見方 — 実物 2 例

あなたが見る箇所は 3 つだけ: **①何が変わったか ②何によって確認されたか ③自分に何を決めてほしいのか**。

**例 1 — 承認できる報告**:

> ECO-042 是正完了。①変更: 作業タブの切替ボタン配線(2 ファイル・+18/−3)。②確認: build 0(エラー 0 件の意味)・テスト 24/24・**是正前にプローブ 2 本が不合格→是正後合格**・オラクル PASS。コミット: a1b2c3d。③お願い: golden 手順 3 項(添付)の実機確認。

→ ①②③が揃っている。golden を実施して OK か所見を返す。

**例 2 — 差し戻す報告**:

> 修正できました。動くと思います。

→ 数字もコミットも「決めてほしいこと」もない。**「build/テスト/オラクルの数字とコミット番号、こちらで確認すべきことを付けて再提出」と返す**(失礼ではなく規律 — [working-with-ai.md §6](working-with-ai.md))。

## 6. ミニ用語集(正典= [docs/terminology.md](../../docs/terminology.md) の人間向け投影)

初出の語は本文中の短い言い換えで足りるように書かれているはずで、この表は詳しく知りたいときに引く。

| 用語 | 生活語での意味 | 人間がすること |
|---|---|---|
| 裁定 | AI が決めきれない選択を選ぶこと | 選択肢の記号で答える。同意できなければ差し戻し(§3) |
| golden | 実機で合格基準を確認する検収 | 手順どおり触って「OK」か見たままを返す |
| オラクル | **比較によって合否を決める基準**(製造 AI には見せない) | その基準が業務上正しいかを確認する |
| kit | 製造者に渡す一式(設計正本+製造指示) | 非知覚媒体では凍結前レビューの対象(§4) |
| 凍結 | 以後変更しない状態にして記録すること | 凍結の承認はあなたの裁定 — 凍結後は直させない規律なので、承認前に §4 の確認をする |
| ECO | 変更 1 件の正式な伝票 | 症状を `/eco-file` で起票してもらう。台帳= `bomdd/60-change-register.yaml` |
| 起票 | 変更・不具合を伝票(ECO)にすること | 「ついでに直して」の代わりに「これも起票して」 |
| PLM | 台帳のつながり(仕様↔部品↔検査)を機械検査する装置 | **直接操作しない** — AI に「PLM で同期して」と依頼し、出てきた指摘の裁定だけする |
| プローブ | 欠陥を実測で捕まえる小さな検査 | 是正報告に「是正**前**は不合格だった」とあるか見る |
| R1〜R5 | AI 側の作業規律の番号(起票なし変更の禁止など。正典= 製品リポの change-management.md) | 番号は覚えなくてよい — AI の逸脱を疑ったら「それは規律違反では?」と聞けば足りる |
| コミットハッシュ | 変更記録の通し番号(英数字 7 桁以上) | 報告に付いているかだけ確認する |
| ずる | 検査を通すための不正(期待値の覗き見など) | 防止は仕組みの仕事 — あなたは工場セッションに情報を貼らないことだけ守る |

## 7. 詰まったら

- 現在地が分からない → `/bomdd-next`。それでも分からない → `bomdd/60-change-register.yaml` の最終行。
- 誰に聞けばよいか分からない → charter の役割表。
- このページで解けない質問 → **設計者 AI に直接聞いてよい**(方法論の説明は AI の仕事であり、あなたが文書を全部読んでから来る必要はない)。
