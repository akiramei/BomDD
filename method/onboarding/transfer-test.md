# Transfer Test — playbook 自己完結性・転移の測定手順(candidate・N=2)

> **ステータス**: **実証済み(N=3・2026-07-09 昇格)** — transfer-01(Claude fable-5)/ transfer-02(Claude sonnet-5)/ transfer-03(GPT-5.5 / Codex CLI = ベンダー横断)の 3 ラウンド・3 題材で機能。次の適用= 人間ラウンド(Paper 1b)。
> **目的**: playbook とテンプレート(kit)だけを渡された fresh 担当者が、**説明者の介入なしに** BomDD の一周+ECO を実施できるかを測定する。第三者再現実験(人間ラウンド)の実験設計書を兼ねる。
> **原型**: BomDD loops/transfer-01/protocol.md・loops/transfer-02/protocol.md(実測データつき)。

## 1. 設計原理 — 方法論装置の自己適用

playbook = 製造パッケージ、fresh 担当者 = 工場、方法論の一周 = 製造、介入 = unspecified residue と見立てる。既存装置をそのまま転用する: G3 ドライラン→ T0 静的検査 / G2 マルチリーダー監査→ 複数 fresh 読者の抽出差分 / 差分帰属→ 介入・詰まりの帰属 / fresh 転移の証明形式→ 補正が文書に宿ったかは次の fresh で測る。

## 2. 段構成

```
T0 静的検査   fresh 読者 2〜3 体に kit のみを渡し、G2 型(手順・停止点・完了条件の独立抽出→差分)
              +G3 型(着手前質問の全列挙+A/B/C 分類)。読者がサブエージェントの場合は
              文脈汚染を遮断できないため「合格は弱く不合格は強い」片側検出器として扱う
  ↓
補正          T0 所見を blocker 級(P1)/軽微(P2)/意図的見送り(P3)に裁定(レビュー停止点)。
              全所見を前倒し検出リストとして凍結(T1 残介入との突合= 前倒し検出率)
  ↓
kit 凍結      commit/tag。manifest から除外: 実験自己記述(FINDINGS の transfer 節等 —
              伏せ項目方式の学習を防ぐ)・採点文書(attribution-scoring)・protocol/brief
  ↓
T1 動的転移   別環境(記憶なし・web 遮断)の fresh 担当者+新規同クラス題材で一周+ECO。
              介入台帳を全交信で記録。実施中は playbook を直さない
  ↓
事後監査      担当者リポの bomdd/ 一式 vs playbook 要求の突合+4 軸採点+還元
```

## 3. 役割規律

| 役割 | してよいこと | してはならないこと |
|---|---|---|
| fresh 担当者 | kit 参照・顧客への質問・全作業 | kit 外の BomDD 情報参照(物理遮断を推奨: web deny) |
| 顧客役 | **業務要件の裁定のみ**(凍結ブリーフの範囲で即答・範囲外は「どちらでもよい。提案されれば裁定」+裁定ログ追記) | 方法論・工程・成果物形式の指南(聞かれたら定型文「あなたの手順書に従ってください」)・設計/実装の示唆(観測者>担当者の能力非対称がある場合は特に) |
| 観測者 | 介入台帳の記録・受領検分(読取のみ) | 担当者への助言。golden 等を促す顧客発話は「顧客起点」として帰属を正直記録 |

## 4. 顧客ブリーフ(凍結・担当者非開示)の様式

1. **題材**: 小型・数 REQ・永続化+移行 1 回を誘発する ECO シード・原版なし。ラウンド間は同クラス・非同一ドメイン(公開済みブリーフの再演は不可)。
2. **ペルソナ**: 業務の専門家・技術非専門(回答の語彙をここに固定する)。
3. **要求シード**: ブレストで小出しにする 5 件前後(数値境界の種を 1 件含める)。
4. **伏せ項目 5 件**(聞かれるまで言わない): 時間/金額の帰属基準・判定時点/分母・単位/刻み/範囲・確定操作の取消可否・0/負数。帰結を「質問で引き出し / 未質問だが明示的仕様化 / 申告なき充填」の 3 値で採点する — エリシテーションと沈黙掃討の測定器。
5. **ECO シード**: 名簿等への属性追加+スキーマ移行+効力(非遡及)を誘発する変更 1 件+準備済み裁定(既定値・遡及・管理方法)。
6. **シードが渡らない場合の扱い(transfer-03 実測)**: 担当者のエリシテーション様式によって要求シードが引き出されないまま実装が進んだ場合、シードは業務上自然なタイミングで**遅延投入**する — 納品直後の変更要求を誘発し、それ自体が「変更即応の規律」の追加測定になる(ただし顧客接点数のラウンド間比較には注意として記録)。

## 5. 介入台帳

全交信を記録し分類: `customer_ruling`(合法・無制限)/ `method_explanation`(**介入** — playbook 未規定の証拠)/ `rescue`(**重大介入**)/ `tooling_gap`(環境・梱包)。各介入に 2 問: 介入なしで進めたか / 成功の帰属を変えるか。フェーズ×分類の分布が「破れの位置」の主データ。台帳ヘッダに担当者モデルに加え**ハーネス(CLI 名・バージョン)**を記録する — ベンダー跨ぎでは軸が複合になるため、破れの帰属切り分け(モデル起因/ハーネス起因)に必須(transfer-03 実測: git 凍結の不履行はハーネスの作業流儀起因の可能性)。

**検査官の記録**(transfer-04 — 独立受入検査を併用する場合): 検査官(独立受入検査を担う異系統エージェント)も、**要求値と実到達値を分けて**記録する — provider / requested model・alias / **resolved model(確認できない実行は `unknown`。推定で埋めない)** / クライアント種別(CLI・plugin 等)/ クライアント・統合層の版 / harness・skill・prompt bundle の版または commit / read-only の強制方法 / 検査官へ開示した資料 / 非開示にした資料 / 実行日時とセッション分離の有無。要求モデルと実到達モデルは乖離しうる交絡がある(transfer-04 実測: `gpt-5.6-sol` を要求したが codex プラグインの app-server クライアント識別ゲート〔plugin.json 版を名乗る〕により `gpt-5.5` へ到達 — 到達モデルは CLI 版でなく統合層の識別で決まった)。

## 6. 採点(4 軸+突合)

**A. 自己完結性**(方法論質問 0 が直接証拠)/ **B. 実行忠実性**(全ゲート記録・凍結タグ・隔離・lineage・クローズ様式の事後監査)/ **C. 介入依存度**(説明+救済の件数と寄与)/ **D. 帰属可能性**(ずる・欠陥・harness の帰属記録が機能したか)。加えて: 前倒し検出率(T0 所見 vs T1 残介入)・伏せ項目の帰結・影響なし予測の的中(ECO)・工数。

**人間ラウンドの追加指標**: 入口到達時間・読み直し回数・質問の種類・テンプレート誤記・ゲート飛ばし・顧客語翻訳の失敗(AI 転移で潰した kit を「人間がどこで違う形で誤読するか」に焦点)。

**時間測定の規律**(t2 実測: 自己申告 3 分 45 秒に対して観測壁時計 1 時間 05 分、約 17 倍の乖離): トランスクリプトのタイムスタンプから直接得られる値は、労働工数ではなく、**セッションの経過時間および観測可能な活動区間**である。壁時計時間、人間応答待ち区間、機械活動区間(モデル推論・検査実行・環境処理)、未分類区間に分けて記録する(治具: `method/tools/time-decomposition.py`)。
タイムスタンプは observed、区間への意味付けは derived/classified として来歴を区別する。並列に実行された活動については、経過時間上の union duration と、各活動の span sum を混同しない。自己申告値または UI の Worked 表示を保存する場合は self-reported と明記する。
ラウンド間比較は、同じ測定成分かつ同じ工程構造の間で行う。壁時計時間をそのまま比較せず、顧客往復数、工場数、並列度、git 操作の有無など、時間を生じさせた工程構造を併記する。

## 7. 実測済みの帰結(この手順が測れたもの)

- transfer-01(fable-5): 介入ゼロ・N=1 成立。翻訳層の自然発生(FINDINGS §11)。
- transfer-02(sonnet-5): 介入ゼロ継続・N=2。**補償機構** — ティアで劣化した自発性を工程のゲートが捕捉(FINDINGS §11.1)。P1〜P3 の事前予測方式・「変える軸は 1 つ」の統制もこの 2 ラウンドで確立。
- transfer-03(GPT-5.5 / Codex): 介入完全ゼロ・**N=3 = 初のベンダー横断**。文化は横断・様式は横断せず/環境制約はテーラリング縮退で吸収/決定の外部化=第 3 のエリシテーション型(FINDINGS §11.2)。

## 8. CAD 適格性判定への転用(candidate・CLI 媒体 N=1 実施待ち= cli-cad-01)

CAD(設計正本)の適格性は二段で判定する(2026-07-18 裁定・修正採択):

1. **構成的適格条件(定義)** — 次を欠く成果物は CAD 候補でない:
   - **設計権威**: その領域の設計上の正本であること。
   - **上流性**: 実装コードより上流にあること。
   - **権威境界**: 何を規定し、何を規定しないかを宣言していること。
   - **未確定会計**: 未確定事項を未確定のまま台帳保持できること。
   - **裁定来歴**: 変更履歴と裁定主体を持つこと。
   - **BOM・Control Plan トレース**: E-BOM/M-BOM/Control Plan へ接続できること。
2. **認定試験(転移実測)** — 構成的条件を満たす CAD 候補に対し、次の 2 条件を実測で判定する:
   - ①それのみを渡された fresh 製造者が、説明者の介入なしに **CAD で規定した意味論**の製品を
     製造できる(**原品との同一性ではない** — CAD が未規定と宣言した次元の差は不合格事由でなく
     探索観測として採取する)。
   - ②その正本のみから、製造と独立した受入検査(oracle)を導出できる(transfer-04 型)。

チェックリスト適合(構成的条件)だけで CAD 認定しない — 認定は試験合格。新媒体の CAD 導入は
宣言でなく実験で earn する(FINDINGS §10「職人芸は翻訳層へ移動する」に基づき、媒体固有の翻訳層の
位置予測を事前登録してから測る — 実験設計= loops/cli-cad-01/protocol.md)。媒体固有の意味契約
(冪等・TZ・終了コード等)は silence-checklist の観点行として実測起点で吸収し、媒体タクソノミー
(Software CAD 階層)の正本化は媒体実証 3 例まで凍結する(OBS-20260718-03)。
