# モック忠実性ポリシー(権威宣言)

生成: {{DATE}}(bomdd-init)。本書は {{PRODUCT}} の UI/UX 適合を測る**権威の所在**を宣言する。

## P1 — モック+screens 文書が設計原器(CAD)である

製品 UI の適合は、原典・実装・記憶ではなく、本リポの**モック(`資料/`)と
`docs/screens/*.md` を基準にした視覚**で測る(golden-in-the-loop)。

## P2 — 乖離時は常に CAD が正

実装と CAD が食い違ったら、原則として**実装側を CAD へ寄せる**。実装都合で CAD を
変えたくなったら、それは P3 の手続きを踏む。

## P3 — 意図的 mock 是正は裁定記録つきでのみ許す

CAD 側に欠陥がある(操作性・安全・整合の問題)と判断した場合のみ、**人間の裁定**を経て
CAD を是正できる。是正は `docs/review_points.md`(と必要なら `docs/decisions/` の裁定資料)に
「何を・なぜ・いつ・誰が」を記録してから行う。無記録の CAD 改変は禁止。

## 運用

- 新画面: モックを `資料/` へ → 挙動仕様を `docs/screens/<screen>.md` に散文で固定 →
  製品リポの抽出(UI-IR/UI-BOM)の入力にする。
- 未確定事項: `docs/review_points.md` に ID(画面接頭辞+連番)で登録し、決定するまで
  「未確定」を明示する(推測で確定させない)。
- golden 所見が出たら、まず工程診断(mock/UI-IR/BOM/実装のどこの欠陥か)。CAD 欠陥なら
  本書 P3 で CAD から直す(製品コードが先ではない)。
