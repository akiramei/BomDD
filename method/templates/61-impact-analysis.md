# Impact Analysis — <ECO/CAPA-ID> 影響分析(Phase 7)

> [60-change-order.md](60-change-order.md) §2 の詳細版(小規模 ECO なら 60 内に直書きでよい)。
> 規律: 影響「あり」だけでなく**影響なし予測を反証可能な形で製造前に凍結**する。回帰の結果が予測の採点になる(Phase 7 の測定値。forward-01.5 で全的中=under/over-inclusion 0、forward-02 で under-inclusion 1 を行単位で定量化=§1.2 の点検を工程化した契機)。
> 粒度: 標準は M-BOM unit。小規模 ECO で影響集合が全 unit に達する場合は **E-BOM 部品粒度+不要改変 diff 測定([63-diff-audit.md](63-diff-audit.md))で代替**(playbook §4.1 実測フィードバック)。
> 欠陥是正では、影響分析は「どこを直すか」だけでなく「欠陥がどの上流成果物に宿ったか」を確定する。原因が仕様/BOM/検査器なら、その上流成果物の改訂を影響ありに含める。ソースだけを影響ありにして始めない。

### 1.1 閉集合宣言の掃討(plm ECO-002 の教訓)
新設節・新規則・新出力が、旧本文の「〜の N 種で全て」「〜のみ」型の宣言を破っていないか grep する
(トレース逆引きは ID を辿るが散文の閉集合宣言は辿れない — plm ECO-002 では工場が検出した)。
- [ ] カーディナリティ宣言(「N 種」「N 分類」)
- [ ] 閉集合宣言(「〜で全て」「〜のみ」「〜に限る」)
- [ ] 用語表・掃討表の転記値(件数・列挙)

### 1.2 As-Maintained 実物の呼び出しサイト点検(forward-02 の教訓)
トレース逆引きは BOM 上の影響**品目**を特定するが、**品目→unit の写像**(実装がその品目を
どのファイル・呼び出しサイトに分布させたか)は BOM だけでは辿れない。部分改修では、
影響集合を凍結する前に As-Maintained 実物を点検する(「実物が正」の影響分析版):
- [ ] 変更対象シンボル(関数・定数・型)の**全参照**を実物で grep し、参照サイトの unit/ファイルを影響表と突合する(**tests/ を含む** — scale-02 第1段でテスト直結合 84 箇所の見落としが blocker 化した実測)
- [ ] シグネチャ変更を伴う場合、呼び出し側の追従が「影響なし」宣言したファイルに及ばないか確認する
- [ ] 及ぶ場合は影響表へ移すか、予測を「配線 N 行のみ」の形で明示する
(forward-02: 「Api 層 diff ゼロ」予測が配線1行で反証。2 fresh 工場が事前申告つきで同一の1行に
独立収束=取りこぼしは工場でなく影響分析に宿る。E-BOM 粒度の予測は的中しており、外れるのは写像の次元)

### 1.3 ビルド派生物の波及先(plm ECO-004/005 の教訓・2例実測)
リポがビルド派生物(dist・ビルドキャッシュ)をコミットする方針の場合、派生物の更新先は
変更したパッケージに閉じない — インクリメンタルビルド(tsc -b 等)は**依存元パッケージ側の
キャッシュ(tsbuildinfo)も更新する**(plm で core 変更→ cli 側 tsbuildinfo 更新を2度実測)。
- [ ] 変更パッケージに依存する全パッケージの派生物(dist / *.tsbuildinfo)を影響表に含めたか
- [ ] diff 監査(63/R-052)の allowed_paths に派生物の波及先を宣言したか

### 1.4 ハブ台帳 — under 実績 unit の既定点検(scale-01 の教訓)
影響分析は「品目の隣接」を辿るが「**実装の重心**」を辿らない。実装がハブ unit(god-VM・共有配線
ファイル等)に集中しているリポでは、同じ unit を系統的に取りこぼす(scale-01 実測: 部品数十の実題材で
実 under を持つ ECO 14/16・**同一の単一ファイル unit を 10/16 ECO で取りこぼし**=規模ではランダム誤差
でなく系統誤差)。
- [ ] このリポの**ハブ台帳**(過去 ECO で under 実績のある unit のリスト。52-metrics に記録)を確認し、
      載っている unit を影響表に入れるか「影響なし」の根拠を書いたか(**複合宣言 unit**(多:1)は
      宣言品目のいずれの変更でも予測に入れる — scale-02 第5段)
- [ ] ハブ台帳が無い場合: ECO 台帳が数件以上溜まっているなら**遡及採点**(宣言影響集合 vs 実 diff の
      全 ECO 一括突合)で機械的に発見できる — 汎用治具: [tools/impact-retrospective.py](../tools/impact-retrospective.py)
      (impacted_bom / affected_refs 両対応・test-only 分離・ハブ集中度を出力。初出= scale-01)

## 2. 影響なし予測(反証可能 — 製造前に凍結)
> 製造後の回帰で当たるかを検証する。外れ=**under-inclusion**(取りこぼし。最危険)。
> 既存オラクル行を1行ずつなめ、「なぜこの行は変わらないか」の根拠を書く(根拠が書けない行=影響分析の穴)。
> **書き方**: 「ファイル単位の diff ゼロ」級の反証可能な宣言にする — 外れ方が行単位で測定でき、
> under/over-inclusion が定量化される(forward-02 の主測定はこの書き方が可能にした)。
> 依存グラフを辿った上で「依存があるのに影響しない」と切る行(保存値経由の自動追従等)は、
> その根拠と、追従を検証する新規オラクル行(変更受入側)をセットで書く。
