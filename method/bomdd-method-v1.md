# BomDD Method v1 — ソフトウェア工業化の方法(薄い正規版)

MoviePad 再実装の**7ループ**([../FINDINGS.md](../FINDINGS.md))で実証した規則だけを束ねた、最小の正規方法論。
各規則に**根拠(ループ番号)**を付す。詳細な物語は FINDINGS、公開版は [../WHITEPAPER.md](../WHITEPAPER.md)、運用表は method/ 各ファイル。
> **v1.1 addendum (v2 Web/API)** — 本書 §0–§7 の規則は **MoviePad 7ループ(v1)**で実証したもの。§5「受入の2層分離」と §6「差分帰属の3分類」は、**v2 Web/API(公開リポジトリ [BomDD-WebApi-Sample](https://github.com/akiramei/BomDD-WebApi-Sample)、webapi-01〜02)で追加された v1.1 規律**であり、本文中で「(v1.1, webapi-02)」と明示する。読者は v1(MoviePad)と v1.1(Web/API 追補)を区別されたい。根拠ループは `webapi-*`。

## 0. 第一原理 — 核と表面
> 部品を「**ロジックの核**(要求から導出可・unitで検査・鋳造できる)」と
> 「**外部表面**(外部ツール文法・画面の見た目=外部/設計知識の転記が必須・受入は execution/golden+人間判断)」に**分けて管理する**(本研究の観測に基づく作業仮説。一般法則としては未検証)。
> 観測した範囲では**ずるは表面で出た**。工業化は「核を鋳造し、表面を仕上げ・検査工程で保証する」こと。(Loop1-7)

**用語「鋳造できる(castable)」**: 原版非開示の製造装置が BOM/工程のみから成果物を生成し、定義済み受入(unit/L2/L3/golden)を**手修正なしで通過**すること。比喩でなくこの操作的条件で測る。

**昇格命題(Loop7)**: 仕様の正しさは振る舞いだけでなく**根拠の精度**に依存する。根拠の精度が**検査深さと許容差**を決める。よって入口品質は「振る舞いを写したか」でなく「受入の深さ・許容差を決める根拠を正しい精度で写したか」で測る。

## 1. 工程フロー
`要求 → E-BOM → M-BOM(+Control Plan/FMEA) → 工程(製造装置) → 受入 → S-BOM` を、PLM のトレースで貫く。
既存資産からの場合は入口に**リバース工程**(下記 1.5)を置く。

## 1.5 リバース工程(入口品質 — CHEAT-001 対策)
原版からBOMを起こす工程は「完全読解」でなく「**双方向トレース + カバレッジ監査**」。(Loop7)
- **トレース行列**: `原版証拠 | 振る舞い | REQ | 核/表面 | E-BOM部品 | K-BOM依存 | M-BOM AC | 検査深さ | FMEA | 治具` を前方+逆引き両方向に組む。
- **カバレッジ監査**: 未トレース要素を `out-of-scope / unknown / ignored-with-reason` に分類。unknown は S-BOM へ登録し版更新で再検査。危険点(座標系等)が FMEA と深い受入へ到達することを確認。
- **マルチファクトリ・リバース**: 同じ原版を複数リーダーに読ませ抽出REQの差分を測る。差分箇所=「仕様として自然に読めない場所」=弱点。
  実証(Loop7): 振る舞い・座標系は全員一致(コードから読める)が、**根拠の精度**(例: 「AV同期」と粗く読み「AACサンプル精度」を取りこぼす)が弱点。**根拠は受入の許容差を決める**ため、過小読みは粗い受入を招く。→ 受入深さを決める根拠行は K-BOM(外部知識)で補強する。

## 2. E-BOM(設計部品表)
- **核/表面で型分け**する。核=`requirement` 由来。表面=`external-knowledge`/`design-token` 由来で、出所(ツール仕様/デザインシステム)へトレース必須。表面の変更は defect でなく**知識更新**。(Loop2,3)
- **存在理由の型**: `requirement / defect-class / constraint / external-knowledge / design-token`。(CHEAT-003,007,010)
- **木でなくグラフ**: 横断(woven)部品に owner/consumers を持たせ影響を辿る。(CHEAT-002)
- **K-BOM(知識部品表)**: 製造装置が参照する外部/設計知識(例: ffmpeg文法パック=[k-bom-ffmpeg.md])を要求BOMと別に管理。K-BOM に無い判断は CHEAT として記録。(Loop4)

## 3. M-BOM(製造部品表)
- 各製造単位に公開**インターフェース契約**と**受入基準**。受入基準は以下3条件を満たすこと:
  - **完全性**: 全数値/全フィールドを規定(丸め方式・継承・既定値)。未規定次元は「テスト緑でも乖離」を生む。(CHEAT-005)
  - **無矛盾性**: AC 同士が矛盾しないか検査(識別子ポリシー等)。(CHEAT-004)
  - **Control Plan列**: 各 AC に `検査深さ(L0/L1/L2/L3/golden)+許容差+承認者`。([control-plan.md], Loop2.5,3,5)
- **不変条件を明示**: 識別子の払い出し(永続単調)・**座標系**(例: シーク時の音声は絶対時刻)。表面バグの主因。(CHEAT-008)
- **FMEA 先行**: M-BOM 確定前に部品ごとの故障モードを列挙し、受入を狙い撃つ。(Loop4)

## 4. 工程(製造)
- **製造装置の隔離**: 製造は M-BOM+工程のみを渡した装置(原版/正解非開示)に行わせる。「気づかれないずる」を可視化する防壁。(Loop1)
- **ずる報告の義務化**: 装置は「BOMに無く慣習で埋めた箇所」を網羅報告する。最も効いた測定器。(Loop1.5,2)
- **マルチファクトリ**: 同じBOMを複数装置(別モデル/別プロンプト)で製造し、出力の**ばらつき**を測る。低能力工場も K-BOM で均等化。(Loop4)
- **治具・測定器を成果物管理**: execution harness / golden generator / L3 解析器は製品BOMと同格に repo 管理。(Loop2.5,3,5)

## 5. 受入(検査の梯子)
深さ = `L0文字列 < L1存在 < L2メタデータ < L3 content/signal < golden+承認者`。深いほど忠実・高コスト。
- **核 = unit(exact, 丸め規定込み)**。
- **表面 = 領域に応じた深さ**: 外部ツール=既定 L2 以上、同期/座標系=L3、知覚=golden+許容差+承認者。
- **L0 文字列照合・pixel-exact は過剰結合のため既定不採用**(妥当な様式差を誤検出し、意味乖離を見逃す)。(Loop2,3)
  - **例外**: canonical serialization が仕様化されている文字列、または deterministic raster そのものが製品特性である場合は L0/pixel-exact を採用可(例: JSON正規化形式・QRコード・ピクセルアート生成器・暗号ハッシュ)。この場合「様式」ではなく「仕様化された正規形」を検査している。
- 「テスト緑 ≠ 仕様適合」は完全性+適切な深さで塞ぐ。(CHEAT-005,009,011)
- **受入の2層分離(v1.1, webapi-02)**: 固定オラクルには**仕様化済み契約だけ**を入れる(status code・error code・状態・ID の pattern・冪等 replay の同一ID性 等)。**ID の具体値・日時の小数秒・応答スキーマの未規定フィールドなど、BOM が値を固定していない次元は合否に混ぜず「探索プローブ層」で variance として観測**する。混ぜると「途中でオラクルを拡張した」ことになり、改善効果測定(鋳造性が上がったか)が濁る。固定オラクルはループ間で**不変**に保つ(同一ヤードスティック)。

## 6. 品質指標 — 二軸
BOM 品質は2軸で独立に測る:
1. **決定性 = 工場間ばらつき**(low ほどBOMが出力を一意化)。(Loop4)
2. **正しさ = 受入(L2/L3/golden)**。
**一致 ≠ 正しさ**: 複数工場が揃って同じ誤りを出す「共有暗黙知の罠」がある。決定性が高くても正しさは別途検査する。(CHEAT-012, Loop5でL3が座標系バグを捕捉)
- **ばらつきの帰属 3分類(v1.1, webapi-02)**: 観測した工場間差分を ①**BOM未規定残渣**(仕様に無く工場が自由に埋めた。別ティアで露見しやすい=C2)②**仕様化済み契約の取りこぼし**(BOMは明記したが低能力工場が外した=**工場能力**。K-BOM の価値の裏返し)③**探索的未規定面**(合否外で観測した分散)に分類する。「AI ごとに精度が違う」で止めず、差分を **BOM欠落 / 工場能力 / 未規定面**に切り分けるのが BomDD の成果。実証: 締めた BOM を opus/sonnet/haiku に同一供与し、opus 0(転移)/ sonnet 1(②not_found 未規定)/ haiku 3(③仕様取りこぼし)。

## 7. S-BOM / PLM(保守層 — Loop6 で検証)
表面部品は時間で劣化する(ffmpeg仕様・OS・UIフレームワーク・デザイントークン・AIモデルの癖)。S-BOM は依存ライブラリ一覧に加え、
**(a) 表面部品が依存する外部知識とそのバージョン (b) 変更時に再実行する検査 (c) AI製造装置のモデル変更による再製造差分** を追う。
劣化イベントを逆引きし、影響部品・再検査深さ・交換判断(交換不要/K-BOM更新のみ/再製造必要)を導く。PLMの価値は被覆でなく**絞り込み**。
**位置づけ**: Loop6 は実 ffmpeg 更新でなく K-BOM 知識更新イベント注入による**概念実証**。「再検査を絞れる」は実証済、「現実運用での劣化自動検出」は次段。(method/s-bom-template.md)
