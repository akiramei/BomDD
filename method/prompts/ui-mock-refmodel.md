# UI モック受入検査 層2.5: 参照概念モデル差分(candidate)— 外→内導出

位置づけ: [ui-mock-coverage.md](ui-mock-coverage.md)(内→外導出)の**補完**。製品カテゴリの一般概念モデル
(ユビキタス言語)を立て、モックとの差分から「カテゴリ丸ごとの欠落」を検出する。製品につき**1回**で足りる
(画面ごとに繰り返さない)。初出: mock-inspect-02(MoviePad・2026-07-05・盲検追試)。

分解能の性質(**望遠鏡効果** — 実測): カテゴリ丸ごとの欠落(空状態・エラー系・進捗・D&D・定型設備)は
確実に見つけるが、「カバー済み」と写像されたカテゴリ**内部**のインスタンス欠落(操作の対称性・設定項目の
個別漏れ)には構造的に盲目。カテゴリ内は内→外(coverage 層3)の受け持ちであり、**両導出は代替ではなく補完**。

運用: 検査エージェントは隔離して実行(入力はモックパッケージのみ)。出力は全て候補と質問であり、
最終事実にならない(§12 三行原則)。ヒアリング回答は①〜④の裁定として 37 台帳へ記録する
(①意図的な非採用=negative scope 裁定は「二度と聞かない」資産)。

---

あなたは UI モックの受入検査官である。今回の任務は「参照概念モデル差分」— 製品カテゴリの一般概念モデルを立て、モックとの差分(gap)を抽出することである。

## 厳格な入力制限(違反は検査を無効にする)

読んでよいのは指示者が列挙するモックパッケージのみ(典型: 設計者注記+状態ごとの DOM snapshot+ui-ir.raw.json)。
それ以外のファイル・ディレクトリ(実装、台帳類、git 履歴、その他一切)を読むこと・検索することを禁止する。
このアプリの実装が存在するか・どう実装されているかを推測して使ってはならない。

## あなたの立場

- あなたの仕事は判定ではなく候補と質問の提示である。「あるべき」を確定してはならない
- 参照モデルとの gap は欠陥リストではない。「意図的な非採用」は敬意をもって扱われる一級の回答であり、gap は全てヒアリング(裁定)のための質問である

## 検査タスク(4段)

### 段A: 製品クラス較正

まずモックだけから製品クラスを推定する。

1. 設計者に聞くべき較正質問を最大3問書く(製品クラスの粒度を決めるための質問。例: マルチトラック合成を将来含むのか、単一クリップ加工に徹するのか)
2. 続けて、モックの証拠から各較正質問への「暫定回答」を推論し、仮定として明示する(以降の作業はこの仮定の上に立つと宣言する)

> 較正が先である理由: 上位クラス(例: プロ用 NLE)を参照にすると gap が数百件出て道具が死ぬ。
> プロスペクティブ運用では較正質問を実際に設計者へ聞いてから段B以降に進んでよい。

### 段B: 参照概念モデルの生成

段Aで較正した製品クラスについて、一般概念モデルを**機能カテゴリ粒度**で構築する。

- 粒度規律: 「音声制御」のようなカテゴリで書く。「音量スライダー・ミュートボタン・レベルメーター」のような機能インスタンスの羅列にしない
- 範囲規律: 較正したクラスの標準に限る。上位クラスの願望的機能を持ち込まない。範囲規律で除外したものは末尾に明示し、「較正回答が変われば改訂」と宣言する
- 各カテゴリに1行の説明と「このクラスで標準とされる理由」を付ける
- 構成例(これに限らない): コア機能 / 再生・操作 / 入出力 / 永続化 / エラー・例外提示 / 空状態・初期状態 / セッション保護 / アプリ定型設備 / ウィンドウ・環境 / 外部依存・セットアップ / アクセシビリティ・国際化

### 段C: カバレッジ写像(簡潔版)

モックがカバーしている機能カテゴリを列挙する(段Bのモデルと同じカテゴリ語彙で。判定なし・記述のみ)。

### 段D: gap 一覧(最大15件・手戻りコスト順)

参照モデルにあってモックに無い(または痕跡が薄い)カテゴリを列挙する。各 gap に必ず付けるもの:

- カテゴリ名と、モックでの状態(完全に無い / 痕跡のみ)
- なぜこのクラスの標準と考えるか(1行)
- ヒアリング質問: 設計者が次の4分類で答えられる形にする
  - **① 意図的な非採用**(理由も)→ negative scope 裁定として台帳へ(資産=二度と聞かない)
  - **② 検討漏れ** → モック改訂 or 要求台帳へ還流
  - **③ 対象だがスコープ外** → scope 裁定として台帳へ
  - **④ フェーズ計画**(後のバージョンで)→ deferred として記録(将来 ECO の予告)
- 放置した場合の手戻りコスト見積(high=データモデル・状態設計・操作の意味論に波及 / medium=画面構造 / low=見た目)と理由1行

gap は手戻りコスト降順に並べる。15件の予算を使い切る必要はない。較正したクラスの標準と言えないものは出さない。

## 出力

報告書全文(markdown・段A〜Dの見出し付き・段Dは順位番号付き)。末尾に範囲規律による除外リスト。
