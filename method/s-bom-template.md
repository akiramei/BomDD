# S-BOM テンプレート(保守・劣化・交換判断)

依存ライブラリ一覧では足りない。表面部品は**外部知識**に依存し、それが変わると核を触らずとも保証が劣化する。
S-BOM は「何が影響し / 何を再検査し / 交換・再製造が要るか」を導けるトレースを持つ。

## S-BOM 行(稼働中の各部品)
`部品 | 種別(核/表面) | 依存する外部知識(K-BOM ID + version) | 受入深さ | 現build | 影響時の再検査 | 交換判断`

- **依存する外部知識**: 要求でなく外部/設計知識(K-BOM)への依存を明示。劣化イベントはここに紐づく。
- **受入深さ**: その部品の保証を再確認するのに必要な検査(unit/L2/L3/golden)。
- **交換判断**の語彙: `交換不要 / K-BOM更新のみ(再認証) / 部品交換+全再認証 / 再製造必要`(forward-03 で拡張)。
  - **部品交換+全再認証**: 調達部品(パッケージ等)の版だけを替え、コードは製造しない処置。fresh 工場は不要 —
    **設計者が交換を適用し、既存オラクル資産(固定/移行/効力)の全走行+処置固有の検査(例: 脆弱性 0 件)で再認証**する。
    既存オラクルが「交換部品の互換性検査器」として再利用される(forward-03: 交換 csproj 2行・新規に書いた検査は V01 のみ)。

## 劣化イベント → 影響分析 手順
1. K-BOM 更新イベント(または外部ツール版・OS・デザイントークン・AIモデルの変化)を受領。
   **検出チャネルにずる報告を含める**(forward-03): 製造工場の正直な環境ノイズ申告(例: NuGet の
   脆弱性 warning NU1903)は劣化イベントの受信機を兼ねる — cheat-report に現れた外部依存の異常は
   設計者が確定検査(例: `dotnet list package --vulnerable --include-transitive`)にかけ、DEG として起票する。
2. S-BOM を逆引きし、**その外部知識に依存する部品だけ**を影響範囲とする(他は再検査不要)。
3. 各影響部品の「影響時の再検査」(深さ)を実行する。L2 で足りるか L3/golden が要るかは部品ごとに記録済み。
   **調達部品の劣化では BOM トレースだけで処置を決めない**(forward-03): 推移的依存の影響範囲・修正版は
   BOM に写っていない — アドバイザリ本文(影響バージョン範囲・修正系列)と**依存解決の実測**
   (scratch 複製での交換検証: ビルド+受入+脆弱性検査)を一次資料にして処置を確定する。
   当初処方が実測で反証されたら再裁定する(「実装検証が裁定を確定する」の調達版 —
   forward-03 では 10.x 更新単独では未解消と実測で判明し、修正系列の直接参照を併用した)。
4. 再検査結果で判断:
   - 合格 → 旧buildは新知識でも保証成立 → **K-BOM更新のみ(再認証)**。
   - 調達版の交換で解消し、コード不変+全再認証合格 → **部品交換+全再認証**。
   - 不合格 → **再製造必要**(M-BOM/工程で作り直し)。
   - そもそも依存無し → **交換不要**。

   **「再製造必要」への昇格パスの実像(forward-04 で初適用)**: 昇格は実運用では
   **適用後の回帰赤ではなく、手順3の scratch 検証の赤で決まる** — 決定的な非互換
   (コンパイル不成立・決定的挙動差)は scratch が先取りするため、適用後に回帰が赤くなるのは
   scratch で再現できない非互換(環境依存・非決定的・実データ規模依存)に限られる。
   昇格後の再製造 ECO の正常形(forward-04 実測: S28+E4+M4+V01 全緑・不要改変 0・cheat 0):
   - 改訂は **K-BOM(新部品の慣行知識)+M-BOM(procurement)に閉じる**。REQ・仕様・E-BOM・
     オラクルは無改訂が正常形(挙動契約が部品に依存していたら、それ自体が設計欠陥のシグナル)。
   - K-BOM v+1 に**適応の裁量を吸収する知識**(型対応・パラメータ/機能の可用性)を明記する —
     適応が機械的作業に落ち、工場の裁量申告が消える(forward-04 で cheat 0 件= forward 系列初)。
   - 影響 unit は 61 §1.2 の実物点検(旧部品 API の全参照 grep)で絞り、再製造は影響 unit のみの
     部分再製造+**全**再認証(交換は挙動全域に触れ得る)。既存 DB 等の実データがあるなら
     移行オラクルが「旧部品が書いたデータを新部品で読む」**データ互換性検査**を兼ねる。
5. 重要: 目的は**再検査を最小に絞る**こと(影響しない核を巻き込まない)。PLM の価値は被覆でなく**絞り込み**。
6. **退役条件付き調達**(forward-03): 推移的依存の上書き(修正版の直接参照)のような**恒久でない処置**は、
   退役条件(例: 上流パッケージが修正系列を自身の依存として参照した時点で直接参照を外す)を
   procurement に明記し、S-BOM の watch 項目にする。処置が新しい監視対象を生むのは正常(記録しないのが異常)。

## 交換性の事前測定(vendor leak 指標 — 2 例で較正済み・sbom-vendor-leak-01/02)

部品交換のコストは交換前に測定できる: 調達部品の**固有識別子**(32 procurement の identifier_set)の
層別出現(製品契約 10/20/30/オラクル 41・治具/K/M 31/32+実装)を grep し、交換クラスを判定する。
判定結果は 53 の measured_leak に記録する(**方針= substitutable とは別軸** — 宣言 false でも実測最軽量の
部品が実在する。宣言は「変えない」であってコストではない)。

| 交換クラス | プロファイル | 交換手順 |
|---|---|---|
| **K/M-contained**(軽) | 契約 0・オラクル 0(治具実装のみは可) | K/M 改訂+部分再製造で閉じる(LibraryLending forward-04 で予測一致= match) |
| **oracle-coupled**(再認証つき移行) | **期待値が部品挙動の関数**(例: pHash と decode 経路) | leak は除去不能 — 正応答は「**宣言された結合**」への変換= 版 exact ピン+世代識別子(例: hash_adapter)+仕様固定。交換手順に再校正を含める(ViewPrism2 ECO-054 で実コスト実証) |
| **structural**(構造材) | UI 定義層等へ大量拡散 | 交換= 再製造級。事実上の基盤選定(ADR 事項) |

測定の前提 3 つ:
- (a) 部品同定は**調達部品の固有識別子**で行う — 部品ファミリ名は誤予測源(エンジンとプロバイダは別品目。
  naive な grep は交換されない部品の参照を「重い交換」と誤読する)。
- (b) 識別子希薄部品(拡張メソッド構文等)は grep **偽陰性**型 — identifier_set に not-greppable と
  宣言し、境界 interface(抽象の実装数)で代替測定する。
- (c) 契約層出現は **provenance(external_source 等の管理された参照= 正常)と汚染を区別**する。

注: watch 検査は As-Maintained 個体へのポインタを持つため、再製造は watch の追随更新を生む(微小・記録対象)。
非挙動劣化(ライセンス・商用条件・メンテ鮮度)は挙動面に現れない= 全オラクル緑のまま使用不能化する。
検査面が原理的に無く、**53 の license_watch / maintenance(鮮度閾値)が唯一の検査器**になる —
メンテ放棄は「シグナル不在が劣化」であり、閾値超過を DEG に変換する(陽性対照=検査の生死判定、の依存版)。
