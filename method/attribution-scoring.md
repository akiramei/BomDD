# 帰属ベース採点(attribution-based scoring)— 設計思想の定式化

> **ステータス**: draft(candidate・**未レビュー**・本文織り込み未実施)。起票 2026-07-08。
> **出典**: 外部レビュー提案(2026-07-08「BomDD Benchmark」節)の中核着想を、本リポの実測語彙(playbook §6.1/§6.4/§8、FINDINGS)で定式化したもの。ベンチマーク実装は時期尚早(rule of three 未充足)だが、採点思想は先に固定する価値があるため独立文書化する。
> **検証状況**: 語彙は N=3+forward/scale 系列の実測に基づく。採点器としての実装・題材横断適用は未実施。

## 1. 主張

AI 製造の評価は「合格コードを出せたか」(合否正解率)ではなく、**成果物と期待の差分を、宣言された工程成果物へ帰属できたか**(帰属可能性)を採点対象にすべきである。

- 合否採点が測るのは**装置の当たり外れ**であり、工程の管理可能性を測らない。同じ 90% でも「残り 10% がどこで生まれたか説明できる工程」と「説明できない工程」は工業的に別物である。
- BomDD の実測が一貫して示してきたのは、価値が合否でなく帰属にあること: metrics を raw 一致率単独で報告しない規律(playbook §6.1 手順 5)、失敗 5 分類(§8 手順 5)、是正の帰属調査(§6.4)、影響なし予測の under/over 採点(§8 手順 2)は、すべて帰属を一次データにしている。

## 2. 採点の定義

タスクの提出物を「成果物」単体ではなく三つ組で定義する:

```
提出物 = (成果物, 宣言集合, 帰属台帳)
  宣言集合 … 製造前に凍結された宣言: BOM(E/K/M)・Control Plan・固定オラクル・
             探索宣言(silence 3択)・影響なし予測(ECO の場合)
  帰属台帳 … 観測された全差分への帰属分類
```

採点軸は三つ:

| 軸 | 定義 | 機械判定可能性 |
|---|---|---|
| **帰属被覆率** | 観測差分のうち、帰属分類が付いた割合 | 高(未分類差分の計数) |
| **帰属正当性** | 帰属が正しいか。検証形式は分類ごとに定義(§3) | 中(再現実験を要する分類あり) |
| **予測の反証可能性** | 宣言(影響なし予測・探索宣言)が反証可能な形式で書かれ、事後採点されたか | 高(ファイル単位 diff ゼロ級の形式検査) |

## 3. 帰属分類と検証形式(実測済み語彙の集約)

| 分類 | 意味 | 帰属が正しいことの検証 |
|---|---|---|
| `unspecified_bom_residue` | BOM が沈黙していた | 宣言集合を grep して不在を示す(機械) |
| `specified_contract_miss` | 宣言済みだが工場が外した | 宣言行の存在+fresh 再製造で消えるか(§6.3 の証明形式) |
| `exploratory_unspecified_surface` | 探索宣言済みの分散 | 事前の 3 択宣言の存在(機械) |
| `harness_bug` / `observer_*` | 検査器側 | 治具セルフテスト・較正・陽性対照(§4.4 の 4 段) |
| under/over-inclusion | 影響予測の外れ | 凍結済み予測 vs 実 diff(機械。R-052 型) |

要点: **帰属の正しさは多くが機械検査に落ちる**。「BOM に無かった」は grep で、「予測が外れた」は diff で示せる。これが帰属採点をベンチマーク化可能にする根拠であり、逆に言えば宣言集合の事前凍結(tag/commit)がない提出物は帰属被覆率が定義できない。

## 4. なぜ BomDD 固有の採点になるか

帰属は**帰属先**が存在して初めて定義できる。合否採点は宣言なしの単発生成でも可能だが、帰属採点は「製造前に凍結された宣言集合」を要求する — つまりこの採点方式を採る評価系は、被評価工程に BomDD 型の成果物(BOM・Control Plan・凍結オラクル・影響予測)を構造的に要求する。既存ベンチマーク(単発正解率・issue 解決率)に対する差別化はここにあり、「どの LLM が強いか」ではなく「どの工程が管理可能か」を測る。

多工場の価値の再解釈もここに接続する(§5.2): 分散の観測はモデルの序列づけでなく、**差分を工場能力と BOM 沈黙へ切り分ける装置**である。帰属採点はこの切り分けを点数にする。

## 5. 適用面(想定・未実施)

1. **transfer 系(実施予定)**: fresh 担当者転移テストの介入・詰まりを playbook へ帰属する(loops/transfer-01/protocol.md §4 — 本思想の初適用)。
2. **遡及採点(実施済みの一般化)**: scale-01 の宣言影響集合 vs 実 diff 全数採点は、帰属採点の ECO 特化版として既に一度実行されている(method/tools/impact-retrospective.py)。
3. **ベンチマーク(時期尚早・据置)**: タスク群(Forward 生成/Reverse 再実装/ECO/依存劣化/UI surface/オラクル劣化/多工場分散)への展開は、第三者再現の実例と rule of three を待つ。本文書は採点思想の予約であり、タスク設計はしない。

## 6. 未解決

- 帰属正当性のうち「fresh 再製造で消えるか」検証は高価(再製造 1 回)。抜き取り検査で足りるかは未測定。
- `operator_capability`(担当者能力)と `playbook_ambiguity` の境界判定は人間裁定が残る(成熟度ラダー ③+④ が上限の型)。
- 帰属台帳自体の不正(過少申告)への対抗は、ずる報告義務+診断的読解(FINDINGS の該当節)の転用で足りるか未検証。
