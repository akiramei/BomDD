# Control Plan(製造条件表 + 検査計画)テンプレート

製造業の Control Plan に相当。各 AC(特性)に「何を・どの方法で・どの深さで・どの許容差で・誰が承認」を持たせる。
受入を「テスト緑」から「特性ごとの管理」へ引き上げ、CHEAT-005/009/011(テスト緑≠仕様適合)を構造的に塞ぐ。

## 列定義
`特性 | 種別(核/表面) | 検査深さ | 許容差 | 治具/オラクル | 承認者`

検査深さ(梯子):
- `L0 文字列照合` — 過剰結合のため**既定不採用**(Loop2 で実証)
- `L1 存在・終了コード` — 成果物が生成/起動するか
- `L2 メタデータ` — ffprobe 等で寸法/コーデック/尺(Loop2.5 で実証)
- `L3 content/signal` — 復号して**内容・信号を比較**(尺/音声相関/PSNR/SSIM)。L2 が同一でも座標系・同期バグを捕捉(**Loop5 で実証**)
- `G golden+承認者` — 知覚(見た目)。golden画像+許容差(ΔE/SSIM)+人間承認(Loop3 で実証)

深いほど忠実・高コスト。FMEA で要注意と判定した特性にのみ深い段を割り当てる。

## 数値既定の pin と常設検査器の陽性対照(viewprism2 還元・2026-07)
- **仕様が固定する数値(既定値・上限・刻み)は「値そのもの」を unit 1 行で pin する**: 機能の存在・
  意味論の特性行だけでは数値はどの検査面にも載らず、仮置き・転写ドリフトが潜伏する(ECO-050:
  既定 70 に対し 80/90 が 2 週間)。転写は正本(仕様)と突合。
- **常設検査器(validator/リンタ/hook)は陽性対照(selftest)行を持つ**: 「必ず検出されるべき合成
  フィクスチャ」を定期実行し、検査器の沈黙(no-op 化)を検出する。0-0 は健全の証明にならない
  (ECO-053: キー改名 1 行で検査 3 本が無音死)。承認者=自動・pre-commit 常設。
- **検出器の新設・変更は実在標本で較正してから収載する**(harness ECO-006 還元・2026-07-10):
  実在の既知陽性標本を捕捉し、既知陰性標本を誤検出しないことを確認してから control plan へ収載する。
  **合成 fixture だけで較正を完了してはならない** — 初版検出器は既知の実在事例にさえ適合しない場合が
  ある。初出: self-conformance C8 が U+F03A 化石を見逃し、実名調査によって捕捉。

## 視覚・挙動の golden 所見の headless プローブ化(viewprism2 還元・2026-07)
golden(人間)所見を受けたら、「視覚だから golden 任せ」とする前に**決定論的に実測可能な物理量**を
探す — レイアウト寸法(Bounds)・実効スタイル値(前景色の実効値)・通知/イベント回数は headless で
実測でき、「是正前不合格→合格転化」の較正手順がそのまま成立する(実例 3 種: パネル高さ 274 vs 198・
checked ラベル実効 Foreground=White・タブ切替 CollectionChanged 4 回 — いずれも是正前不合格を実測し
fact 化で恒久封止)。golden は発見器・headless fact は封止器、の分業。

**視覚 probe は「導入コンテナの測定文脈」の下で測る**(ViewPrism2 ECO-088/089 還元・2026-07-15):
物理量を測る際、部品を**単体で**測ると通り、**実際の親子ネスト下**で測ると落ちる欠陥がある —
UI レイアウト部品(折返しパネル・等分配分・内容幅ボタン等)は「正しい部品を選ぶ」だけでは機能せず、
**親コンテナが子へ渡す測定制約(有限幅か無限幅か)**が部品の前提と一致していなければ黙って無効化される。
実測 3 例で系譜が確立: GF-084-01(アイコン用固定幅部品をテキストへ流用)・GF-086-01(等分 Stretch 前提
部品を内容幅へ流用)は「部品の取り違え」、ECO-088/089(折返しパネルが**無限幅を子へ渡すコンテナ**
〔水平スタック・水平可視性未指定のスクロールビュー等〕の直下で折返し契機を失う)は「正しい部品を殺す
文脈」で、適用条件の検査対象が部品選定から**導入コンテナ**へ拡張された。処方: 視覚 probe は部品単体でなく
**実際の親子ネスト下で実描画矩形を測る**。特に「無限幅を子へ渡すコンテナの直下に折返し/幅依存部品」は
既知の反パターン — 是正の read-across 対象(下記 playbook §8.2 真因様式の閉包)。「headless で物理量を測る」
(上)の**測定文脈**の明示化。

## 検査の対照3種と空集合の扱い(transfer-04 還元・2026-07)
検査器は次の3種の対照を分けて設計する。「対象を不在にして FAIL を見る」試験は、検査器にとっては
**既知違反を与える陽性対照**であって「陰性対照」ではない — 名称を混同しない。
1. **違反陽性対照**: 検出対象となる違反を注入し、FAIL することを確認する。
2. **対象欠落チャレンジ**: 必須の対象集合を空・不在・実行不能にし、FAIL または明示的 ERROR に
   なることを確認する(治具の消失・空 manifest を健全と誤認しないため)。
3. **正常陰性対照**: 正常な対象を与え、PASS することを確認する。

一般規則: **各検査は対象集合が空の場合の意味を宣言する**。対象の存在が必須なら fail-closed、任意なら
明示的な SKIP または適用外理由を記録し、**空集合上の vacuous pass を黙って許さない**。
実証例: self-conformance C1/C2 の `bool(files)` 非空ガードはこの一般規則の一適用形(対象が必須の
検査)。ただし「すべての検査を非空必須にする」一般化はしない — 任意対象の検査は SKIP/適用外を
明示記録する側で満たす。初出: transfer-04 で異系統独立検査が C5(対象欠落チャレンジの欠如)・
C9(空 manifest の vacuous pass)を提起(FINDINGS §11.4・silence §16)。

**検査の暗黙前提は明示条件へ昇格する**(3 様態で確立・ViewPrism2 GF-073-08/ECO-078/ECO-079 還元・
2026-07-13): 検査は自身が暗黙に固定している前提(線形履歴・既定設定・単一構成・特定経路)を持ち、
その前提と一致する欠陥を構造的にマスクする。前提を破る条件を検査自身の明示条件へ昇格する —
(i) **手順**: golden 手順は「その経路で本当に検証可能か」を fix 時に実測で確認する(GF-073-08:
移動検出が同一コレクション再取込では構造的に発火せず、手順が検証不能だった)。
(ii) **検査器**: 暗黙前提の反例を selftest の陽性対照として持つ(ECO-078: E19 の線形履歴前提×
マージで誤検知 — 検査の谷間と selftest の欠落は同時に空く)。
(iii) **人間 golden**: 既定構成での実施は既定値と一致する欠陥をマスクする — 実施条件へ
**非既定構成**(設定切替の往復)を明記する(ECO-079: 既定ロケール+歴代 golden 全て単一言語で
i18n 未配線が全 golden を素通り)。
対照 3 種(上)が「対象」の軸なのに対し、これは「検査自身の前提」の軸。

**安全装置(防護手段)の較正は「真の故障モード」での発動実測で行う**(ViewPrism2 ECO-081 還元・
2026-07-14): タイムアウト・キャンセル・ガード等の安全装置には**発動を要求する機構(協調)と発動を
強制する機構**があり、協調機構は協調に応じない故障(無限待ちスレッド・ブロックした I/O 等)に無力。
「オプションが存在し引数が届いている」ことは「効く」ことの証拠にならない — 実測: テスト基盤の
組み込み `--timeout` は引数の到達を起動コマンドラインで確認した上で真のハング(無限待ちスレッド)を
切れず(協調キャンセル)、活動監視型の強制終了(HangDump)は 45 秒で発火+診断採取。人工故障の
陽性対照を**手段候補すべて**に当ててから正本化する。対照 3 種が「対象」、暗黙前提が「検査自身の
前提」の軸なのに対し、これは「検査・防護の手段」の軸 — 較正はゲートの動作確認だけでなく手段選定を裁く。

## 検査ハーネスの時間上限は被呼び出し側の契約(ViewPrism2 ECO-081 gate② 裁定・2026-07)

呼び出し側(AI・人間・CI)がそれぞれ私的タイムアウト+出力監視を持つ構図は、上限を誰の契約にも
しない(ハーネスは無限に待ち得る)。上限は**被呼び出し側=ハーネス自身の既定宣言**にする:
1. 上限はテストプロジェクト/ハーネス自身の既定引数・設定として宣言し、呼び出し側ごとの独自
   タイムアウトは使用しない。
2. **上限の意味論を契約文書に必ず明記する** — 総実行時間上限(対象が増えると誤爆)か、無活動上限
   (進捗がある限り発火しない)か。意味論を欠いた「5 分」はどちらにも読め、運用を壊す。
3. 発火時は診断を自動採取して非ゼロ終了する(実行中テスト名+全スレッドダンプ等)。一次調査の
   順序(名前→スタック)まで契約に含める — 発火のたびに真因特定の証拠が自動的に残る。
4. 一次装置の不作動に備え、外側に粗い共通プロセスツリー上限を **1 層だけ**残す(層構造:
   真因の構造除去 → fail-fast 監視〔即時診断〕→ 無活動上限〔最終安全弁〕→ 実行基盤上限)。

実測(ViewPrism2 ECO-081/083): 私的タイムアウト運用の廃止+HangDump 既定宣言後、間欠ハングの
実発火が自動診断つきで捕獲され(ハング中テスト名+ダンプ)、真因除去まで到達した。

## consumer-surface 接続行と二軸境界(ViewPrism2 ECO-062〜064 還元・2026-07)

同一製品内 3 連鎖(接続= ECO-041→062→063 / 再確立= 058→062→064 / 二軸= 059→062→064)による
**規則昇格基準(rule of three)充足** — 統計的独立反復ではない。出典: ViewPrism2
60-change-order-eco-062/063/064.md。掃討観点は silence §17・工程規律は playbook §4.4。

**consumer-surface 接続行**: 設定→保存→消費、生成→転送→表示など、**複数 unit を跨いで初めて
成立する特性**には、端点ごとの unit 行とは別に、consumer から観測可能な**接続不変条件**の行を置く。
producer 側の生成・保存・単体検査の合格は、consumer がその出力を参照していることを証明しない。
**入力または保存値を変化させ、consumer 側の観測結果も対応して変化すること**を恒久 probe で確認する。
可能な箇所では次を対にする(既定値・キャッシュ値を偶然表示しているだけの偽陽性を排除するため):
- 正常値 A を保存すると consumer が A を使用する
- A を B へ変更すると consumer も B へ変化する
- 接続を切断または参照を固定すると検査が FAIL する

対象は全 unit 境界ではなく、**consumer での効果が契約成立条件となる境界**に限る。
surface を置換・統合・廃止するときは、旧 surface が担っていた受入観点について、**移管先との
機械検査可能な被覆対応、または明示的な廃止理由**をクローズ条件にする(既存 golden の機械的複製は
不要 — 表現が変わる場合は観測意味の同等性または後継検査への対応表でよい)。
潜伏実績: home 設定の保存・解決 Core は CP 健全のまま消費側(LoadViewAsync)未参照が約 3 週間潜伏
(ECO-063)・CAD 定義済み機能の UI 殻のみ実装(ECO-041)。

**非ブロッキング特性の二軸検査**: 非ブロッキング性は、処理をどの実行文脈へ移すかという
**scheduling boundary** と、境界内へ持ち込む処理件数をどこで制限するかという **cardinality
boundary** を別契約・別検査行として扱う。`Task` 返却・`Task.Run`・`ConfigureAwait`・UI 仮想化
などの**機構の存在は、いずれの軸についても単独では証明にならない**(片軸だけの最適化は別の
待ち時間を温存する)。
- scheduling 側の検査対象(最低限): 重い処理が background 境界より**前**に実行されていない /
  UI・主実行文脈で全件走査・同期 I/O を行っていない / background 化された処理の完了待ちが
  主経路を再度ブロックしていない。
- cardinality 側は「全件 API 呼出 0」を一般テンプレートに固定せず、**特性ごとに宣言**させる:
  境界へ渡してよい最大件数 / limit・filter・page が適用される**位置** / materialize される件数 /
  外部呼出回数または走査量の上限。
  (製品側の具体行では「無制限の全件 API 呼出 0」「取得件数 exact」のような固定も正当 —
  ViewPrism2 CP-STARTUP-028 の型。)
潜伏実績: 仮想化(描画実体数)は DB materialize 量を制限しない(ECO-064)・262,046 件 collection で
表示 folder 約 701 件に対し全体直列 pHash 生成(ECO-062)・1 万件で Private 6,698MiB(ECO-058)。

**停止意味論の 2 不変条件(token/generation)**(ViewPrism2 ECO-064→066→075 還元・2026-07-13
規則昇格 — 同一製品内 3 連鎖であり統計的独立反復ではない): キャンセル可能・追い越し可能な
非同期処理は、**token(資源を止める= CTS 等による実行停止)**と **generation(鮮度=古い結果を
現在の文脈へ公開しない世代ガード)**を**別々の不変条件・別々の検査行**として持つ。両者は独立に
破れる — token だけでは「止まったのに古い結果が上書き」、generation だけでは「止まらない浪費」が
残る。`Task` 返却や表示用フラグの存在は、処理の所有者・停止契約・結果公開世代が同じ session 境界に
無ければ停止を証明しない。開始/終了だけでなく target・scope・mode・window 切替の**全ライフサイクル
境界**で検査する。**同期完了に依存した暗黙の直列性は、非同期化(Task.Run 等)した瞬間に並行再入へ
変わる** — 非同期化の是正には generation ガードの導入を対で検査する(3 例目= ECO-075 GF-075-01:
候補探索の Task.Run 化が fire-and-forget 経路との並行再入を生み、世代ガードで是正)。
実例: 起動 loading(ECO-064= generation+CTS+closing cancel)・検索セッション(ECO-066=
画面 reset が実行中処理を止めず遅延完了が新文脈へ侵入)・修復候補探索(ECO-075)。
