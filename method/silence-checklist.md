# Silence Checklist — BOM が沈黙しがちな次元カタログ

> **用途**: BOM・仕様設計時の掃討表。N=3(MoviePad / Web-API / 分散Saga)の全ずる・全探索層分散から「**BOM が沈黙し、製造装置が慣習で埋め、工場間で分岐した次元**」を抽出した。
> フォワード・モード([bomdd-playbook-v1.md](bomdd-playbook-v1.md))では原版が無いため「何が未規定か」を差分で教えてくれる相手がいない——本表がその代替の第一段(第二段はマルチファクトリ分散)。
>
> **使い方**: 仕様化(第1回掃討)と M-BOM 設計(第2回掃討)で全行をなめ、各次元を必ず**宣言**する。終端語彙は3つ:
> - `specified` — M-BOM/K-BOM/仕様に明記(どこに書いたか参照を残す)
> - `exploratory` — 今回は固定しない。探索プローブ層で観測(**合否に入れない**)
> - `out-of-scope` — 理由付きで対象外
>
> **第1回掃討(仕様)に限り**第4の語彙を許す:
> - `deferred-to-phase3` — 仕様レベルでは決めず、M-BOM 設計(第2回掃討)で宣言する。**明示的な延期**であり未宣言ではない。第2回掃討の終了時点で `deferred-to-phase3` が残っていたらゲート不通過(必ず上の3択に解消する)。
>
> **未宣言=ずるの温床**。N=3 で、工場間分散は常に BOM が沈黙する場所に正確に集中した。宣言済みの延期(deferred)と未宣言を区別するのが本表の役割。
>
> **本表は閉じていない**: 新しいドメイン・新しいループで観測された沈黙次元を追記して育てる(完全性は漸近)。各行に証拠を付すこと。

## 1. 数値・量
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 丸め方式 | AwayFromZero / ToEven / 切り捨て。中間値の扱い。テストベクタに中間値を含める | CHEAT-005(テスト緑でもサイレント乖離) |
| 精度・有効桁 | 内部表現と境界での桁。許容差の単位 | Loop7(根拠精度が許容差を決める) |
| 単位・スケール | ms/s/tick、px/dp、%/比率 | Loop5(L3 治具の精度), Loop6(治具劣化) |
| 範囲・境界値 | min/max、開区間か閉区間か(半開 `[start, end)` 等)、境界一致時の判定 | webapi E-BOOKING(隣接予約は重複でない) |

## 2. 既定値・継承
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 未指定フィールドの既定値 | null / 空 / ドメイン既定。「指定なし」と「明示的に空」の区別 | Loop1.5(丸めを潰すと Excluded 継承が顔を出した) |
| 設定の継承 | 親→子へ何が継承され、何が上書きか | 同上 |

## 3. 識別子
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 採番アルゴリズム | 連番(基点・単調性)/ ハッシュ(入力フィールド列挙)/ ランダム(形式) | CHEAT-004(識別子ポリシー矛盾), webapi 探索層(ID hash 仕様) |
| 形式 | prefix・長さ・文字種(例: `bk_`+16hex。工場間で 12/16/24hex に分岐した) | webapi-02 探索層 |
| 一意性の根拠 | 内容依存 / キー依存 / 時刻依存(同内容別キーで衝突するか、が変わる) | webapi-02 探索層(ID 一意性が3方式に分岐) |
| 永続性・単調性 | 再起動・再生成で同じ ID か。欠番を許すか | method-v1 §3(永続単調) |

## 4. 日時
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| タイムゾーン表現 | literal-Z 必須か offset(+00:00/+09:00)許容か。非 Z 入力の扱い(拒否/換算) | CHEAT-WEBAPI(非Zオフセット: 原版400/工場201 の機能差) |
| 小数秒 | 桁数・精度(ms/µs/tick)。応答にどこまで出すか | webapi/saga 探索層(精度が工場間で分岐) |
| 基準時刻 | 現在時刻の取得点・固定可能性(テスト決定性)。タイムスタンプの単調性 | saga 探索層(UtcNow vs 固定 BaseTime) |

## 5. 座標系・基準系(同期系の最重要行)
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 相対か絶対か | 例: シーク後の音声時刻は**絶対時刻**基準(相対と取り違えると実バグ=音ズレ) | **CHEAT-008**(C2 が機能バグ化した唯一例。L3 でのみ捕捉) |
| 原点・基点 | sequence は 0 起点か 1 起点か。座標原点・Y軸向き | saga 探索層(sequence 基点分岐) |
| 検査深さ | 座標系・同期の特性は **L3(content/signal)** を割り当てる(L2 では区別不能) | Loop5(L2 同一・L3 のみがバグ捕捉) |

## 6. エラー・例外
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| エラーコード語彙 | code 名の全列挙(`not_found` か `booking_not_found` か。無指定だと別ティアで露見) | CHEAT-WEBAPI-02-001(sonnet 1/16 の原因) |
| status code 対応表 | 状況→status の全マッピング(409 vs 400 等。明記しても低能力工場は外す=工場能力の検査になる) | webapi-02 haiku(specified contract miss) |
| エラー応答スキーマ | 形式・必須フィールド・detail の粒度 | webapi K-BOM(ErrorResponse) |
| 検証順序 | 複数エラー時にどれを返すか(認証→検証→競合 の評価順) | webapi(401/400/409 の優先) |

## 7. 冪等性・リトライ
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| fingerprint 構成 | 正規化対象フィールドの**全列挙**(「正規化する」だけでは不足。customerId 落ちで機能差=blocker) | **CHEAT-WEBAPI-01**(S7。v1 CHEAT-008 の Web 版) |
| リプレイ時の応答 | 同 key 同内容の再送は 200 か 201 か。同一 ID を返すこと | webapi S5 |
| keyless 再送 | 冪等キー無しの同一内容再送の扱い | webapi-01.6(未プローブ次元として工場が申告) |
| 重複検出キー | inbox key の構成と区切り(handler+eventId、`:` vs `#` で分岐した) | saga 探索層 |
| リトライ方策 | 最大回数・バックオフ・dead-letter 行き条件 | saga M-BOM(maxAttempts=3) |

## 8. 出力スキーマ・表現
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 応答フィールド集合 | 未規定フィールドを返すか(createdAt/cancelledAt の有無が工場間で分岐) | webapi-02 探索層(opus は最小形) |
| JSON 形状 | map か array か。ネスト構造 | CHEAT-SAGA-01-002(**検査器側**が map 形に過剰結合した) |
| キー綴り | snake_case / camelCase | CHEAT-SAGA-02-002(観測出力の camelCase 化。製品差分でなく表現差) |
| 正規形の有無 | canonical serialization を仕様化するなら L0 採用可。しないなら L0 検査禁止 | method-v1 §5 例外 |
| 一覧の順序 | sort キー・方向・安定性 | webapi-01.6(list 順序を工場が申告) |

## 9. 非同期・分散(イベント駆動題材)
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 配信保証 | at-least-once / exactly-once。重複到達時の挙動 | saga M-BOM |
| 順序保証 | どの単位で ordering を保証するか(グローバル/集約毎) | saga M-BOM |
| イベント封筒 | eventId 形式・timestamp・sequence・causationId/correlationId の方針(null/orderId/parent で分岐した) | saga 探索層(eventId 4通り) |
| 補償の順序 | 失敗時にどの補償をどの順で発火するか(補償規則は**核**=要求由来) | saga E-BOM(payment 失敗→在庫解放 等) |
| 観測契約 | ワイヤ境界が無い場合、正規化イベントログを emit する I/F を設計者が定義 | v1.2 (a) |

## 10. 外部ツール・プロトコル表面

## 付録: 宣言の記録様式
M-BOM(または仕様)に次の表を持たせる:
```yaml
silence_sweep:
  - dimension: 丸め方式           # 本表の行
    decision: specified           # specified | exploratory | out-of-scope
    where: 32-mbom.yaml M-CORE-001 invariants   # specified の場合の参照
    note: MidpointRounding.AwayFromZero
  - dimension: 応答フィールド集合
    decision: exploratory
    note: createdAt の有無は固定しない。probe P-03 で観測
```
sweep の実施記録自体が成果物(どの行を見て・どう宣言したか)。証拠として cheat-log と同格に扱う。
