# Silence Checklist — BOM が沈黙しがちな次元カタログ

> **用途**: BOM・仕様設計時の掃討表。N=3(MoviePad / Web-API / 分散Saga)の全ずる・全探索層分散から「**BOM が沈黙し、製造装置が慣習で埋め、工場間で分岐した次元**」を抽出した。
> フォワード・モード([bomdd-playbook-v1.md](bomdd-playbook-v1.md))では原版が無いため「何が未規定か」を差分で教えてくれる相手がいない——本表がその代替の第一段(第二段はマルチファクトリ分散)。
>
> **使い方**: 仕様化(第1回掃討)と M-BOM 設計(第2回掃討)で全行をなめ、各次元を必ず**宣言**する。終端語彙は3つ:
> - `specified` — M-BOM/K-BOM/仕様に明記(どこに書いたか参照を残す)
> - `exploratory` — 今回は固定しない。探索プローブ層で観測(**合否に入れない**)
> - `out-of-scope` — 理由付きで対象外
>
> **第1回掃討(仕様)に限り**第4の語彙を許す:
> - `deferred-to-phase3` — 仕様レベルでは決めず、M-BOM 設計(第2回掃討)で宣言する。**明示的な延期**であり未宣言ではない。第2回掃討の終了時点で `deferred-to-phase3` が残っていたらゲート不通過(必ず上の3択に解消する)。
>
> **未宣言=ずるの温床**。N=3 で、工場間分散は常に BOM が沈黙する場所に正確に集中した。宣言済みの延期(deferred)と未宣言を区別するのが本表の役割。
>
> **本表は閉じていない**: 新しいドメイン・新しいループで観測された沈黙次元を追記して育てる(完全性は漸近)。各行に証拠を付すこと。
>
> **原典ありの移植・再現案件**: 画面や帳票などの「見えている表面」は、振る舞いとは別に**表示要素集合**を掃討する。原典に存在する表示フィールド・ラベル・画像・状態表示・空/エラー表示は、仕様/E-BOM/Control Plan のいずれかへトレースするか、理由付きで out-of-scope にする。凍結後の工場は原典を見ないため、原典表示パリティは G2/G2' の凍結前ゲートでしか検出できない。

## 12. 状態機械
| 次元 | M-BOM に書くべき決定 | 証拠 |
|---|---|---|
| 不正遷移時の挙動 | 拒否(エラー種別)か無視か。終端状態からの操作 | webapi(cancel 済みの再 cancel), saga 状態遷移 |
| 遷移の原子性 | 部分失敗時に状態がどこで止まるか | saga 補償規則 |

> **台帳自身も状態機械である**(方法論ハーネス面・実測 1 例): status 遷移は、対応する履歴証拠の存在・
> 参照先の実在・祖先関係・遷移順序を**状態不変条件**として検査面に列挙する。検査されない状態遷移は、
> 台帳上の 0 エラーと実際の工程完了が乖離する無音面になる。初出: ViewPrism2 ECO-060 運用違反
> (2026-07-11)。

> **状態×操作マトリクスの谷間掃射**(ViewPrism2 ECO-046→102→104・N=3 — 同一製品内系譜):
> 状態(dirty・placing・一時表示等)×操作(定義変更・削除・遷移・ロケール切替)の**交差セル**は、
> 単線の golden 手順・probe から構造的に漏れる — 状態を持つ面の ECO は交差マトリクスで検査面を
> 掃射する。副則①: 保護ガードは「何を守るか」を分解して宣言する(保護対象と巻き添えを分ける —
> 閉じなさすぎ〔dirty 中の削除素通り= ECO-046〕と閉じすぎ〔dirty ガードが表示鮮度まで犠牲=
> ECO-102・潜伏 1 か月超〕は同じ欠落の両面)。副則②: 一時表示の解除経路はタイマだけにしない —
> 表示の前提(クリーン・当該ビュー)を壊す状態遷移を全列挙して解除に接続する(タイマは「前提が
> 保たれたままの寿命」でしかない= ECO-104)。

- 配布物・テンプレートについて、**利用者に渡される実ファイルそのものを lint・parser・治具へ直接入力する
  機械読取り経路を陽性対照に含めたか**。AI による全文再生成、整形または中間表現への変換後だけを検査すると、
  元の構文欠陥が救済され、fail-late として潜伏する。初出: テンプレート YAML 不正の 3 ラウンド潜伏、
  harness ECO-001。

## 16. 検査治具自身の沈黙(自己検証の被覆限界)— 実測(transfer-04)

- 検査を書いたら、次を確認する: **(a)** 対象集合が空・不在・実行不能な場合に無音 PASS しないか
  (例: 対象スクリプトが削除されても Python の script-not-found が同じ exit コードを返して偽 PASS・
  空の manifest でループが 1 度も回らず vacuous PASS)。**(b)** 検査の説明が約束する全分岐を実際に
  通るテストがあるか(docstring が「X 不在でも止めず記録」と約束する分岐が一度も踏まれていない)。
  **(c)** 単なる存在確認を完全性検査の代用にしていないか(`exists()` が中身の完全性を保証しない)。
  **(d)** 主経路以外のオプション分岐でも実行結果と記録が一致するか(部分集合オプション時に台帳が
  全集合を記録して食い違う)。初出: transfer-04 で異系統独立検査が harness 治具に提起した欠陥
  (self-conformance C5/C9・bomdd-init)。自己検証は happy-path と設計陽性対照は張るが、この 4 観点を
  構造的に張り損なう(FINDINGS §11.4)。
- 追記(実測例): 4 観点は検査治具だけでなく**是正と影響予測**でも同型で再演された(ECO-008 変異
  B2= (a) を是正が・ECO-009 の CAD 経路落ち= (d) を影響予測が、各 1 例)。是正 diff と影響なし予測
  にも同じ 4 観点を当てる価値がある — 規則昇格は rule of three 待ち・現状は実測記録。

## 17. 境界接続・性能・データ経路 — 実測 3 連鎖(ViewPrism2 ECO-062〜064 還元)

規則昇格基準(rule of three)充足 — 同一製品内 3 連鎖(ECO-041→062→063 / 058→062→064 /
059→062→064)であり統計的独立反復ではない。検査型は control-plan「consumer-surface 接続行/
非ブロッキング特性の二軸検査」・工程規律は playbook §4.4。

1. **二軸境界**: 非ブロッキング性を主張する経路について、scheduling boundary(どの実行文脈へ
   移すか)と cardinality boundary(境界内へ持ち込む処理件数をどこで制限するか)を**別々に**
   宣言・検査したか。`Task` 返却・`Task.Run`・`ConfigureAwait`・仮想化など、**機構の存在を
   結果の証明に置き換えていないか**。件数制限が全件取得・全件 materialize・全件 filter の
   **後**に置かれていないか。
2. **非機能契約の再確立**: 共有機能や共通経路を別 surface で再利用した箇所について、結果意味論
   だけでなく、計算量が決まる位置・取得件数・materialization・仮想化・キャッシュ接触条件を
   **移植先で再宣言**したか。
3. **producer–consumer 接続**: 設定→保存→消費、生成→転送→表示などの経路について、consumer
   から観測できる end-to-end 受入経路があるか。端点ごとの検査だけで、参照・配線・変換層が
   無宣言になっていないか。**producer の値を変更したときに consumer の結果も変化すること**を
   確認しているか。
4. **surface 置換時の検査継承**: surface の置換・統合・廃止時に、旧 surface が担っていた
   golden/受入観点の移管先・後継検査・または廃止理由が**機械的に追跡できる**か。
5. **停止意味論の 2 不変条件**(ECO-064→066→075 の 3 例目で昇格・2026-07-13): キャンセル可能な
   長時間処理・追い越し可能な非同期処理について、**資源停止(token)**と**遅延結果拒否
   (generation)**を別々に宣言・検査したか。処理を非同期化(background 化)した箇所について、
   それまで同期完了が保証していた**暗黙の直列性**(fire-and-forget 経路との非重複)を明示の
   世代ガードへ置き換えたか。検査型は control-plan「停止意味論の 2 不変条件」。
6. **供給側の入力規模**(OBS-20260713-04 の 3 例目で昇格・2026-07-15): 新機能の影響分析では、
   変更される部品とその消費側だけでなく、**新機能が既存部品へ供給する入力を独立した影響軸**
   として確認したか。入力の**件数・サイズ・頻度・種類・組合せ・寿命**が既存想定を変える場合、
   供給先の計算量・保存量・上限値・テスト範囲への影響を BOM(影響分析・受入 fixture)に
   記録したか。観点 2(非機能契約の再確立)の**供給側対偶** — あちらは部品を動かした側・
   こちらは部品を動かさず入力だけが変わる側。
   昇格記録: decision= promoted / target= silence §17 / basis= 独立 3 例(ECO-062= 閲覧
   コンテキスト非限定→既存 pHash 走査へ 26 万件供給・ECO-073→075= missing 26 万行登録→
   修復画面 O(M×N) 2,000 件 71 秒・ECO-086→088= 定義値展開 47 件→チップ行折返しが影響分析・
   fixture に不在)/ automation= validator へは昇格しない(影響次元はドメイン依存で意味論
   判断を要する)。
