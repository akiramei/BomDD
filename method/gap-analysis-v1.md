# BomDD ギャップ分析 v1 — 工業化の課題と改善(2026-06-10)

> **ステータス**: working analysis。N=3 の実証記録(FINDINGS・ずる台帳・探索層分散・schema candidates)と未解決論点を突合し、**仕様→BOM / BOM→製造 / 検証パターン / 運用**の4軸でギャップを洗い出した。各ギャップに証拠・提案・優先度を付す。
> 「適用済み」= 本分析と同時に playbook/templates へ反映したもの(いずれも prescriptive candidate であり未実証。forward ループで検証する)。

優先度の基準: **P1**=典型シナリオ(ブレスト→仕様→BOM→AI製造)を一周することを直接妨げる / **P2**=一周はできるが品質・保守で破綻リスク / **P3**=スケール・成熟段階の課題。

## A. 仕様→BOM 変換のギャップ(BOM の情報・プロパティ不足)

### A1. NFR(非機能要求)の置き場所が未確定 — P1
- **現状**: E/M-BOM・Control Plan は機能契約(status code・状態遷移・補償規則)中心。性能・セキュリティ・可用性・リソース上限が**どの BOM 層に住み、どう受入されるか**が未定義(open question #3。product-schema-candidates §5 にも未確定と明記)。
- **証拠**: N=3 全題材が機能契約のみで NFR 特性を 1 行も持たない。Control Plan に NFR 行型が無い。仕様に「速い」と書かれた瞬間に CHEAT-006(C4 受入不能)型のずるが再発する構造。
- **提案(適用済み・candidate)**: 三分割 — **目標値=E-BOM**(`nfr_targets`: 測定可能な数値+条件)/ **実現手段=M-BOM**(予算配分: どの部品が何 ms 使ってよいか)/ **検証=Control Plan**(NFR 特性行。深さは原則 L3 execution measurement+許容差)。G2' 測定可能性ゲートは NFR にも適用する(「速い」のままでは unmeasurable=仕様未完成)。
- **残課題**: セキュリティ NFR は測定器が別物(SAST/依存監査/ペネトレ)。検査の梯子に乗るか要検証。

### A2. 横断的関心が部品にならない — P2
- **現状**: woven(graph_edges: owner/consumers)は「複数部品から共有される部品」を表せるが、ロギング・認可・監査・エラー処理方針のような「**全部品に塗られるポリシー**」は品目にできない(open questions の未討議項目)。
- **証拠**: CHEAT-002 の解決(グラフ化)は共有部品まで。N=3 題材は横断ポリシーが薄く(認証は API key 1 点)、未踏。
- **提案(未適用・要実験)**: E-BOM に `policy` 型部品の候補を立てる — `applies_to: 部品集合(またはセレクタ)/ rule: 規則 / verified_by: 横断検査`。受入は「全エンドポイントが認可チェックを通る」のような**全数横断検査**になり、Control Plan に per-part でなく per-policy 行が要る。早すぎる形式化を避けるため、横断関心が一級の題材(認可付き Web/API)で1ループ回してから型を決める。

### A3. データモデル・永続化の BOM 表現が無い — P1
- **現状**: **N=3 全題材が in-memory**。DB スキーマ・マイグレーション・データ整合性制約・トランザクション境界が E/M-BOM のどこに置かれるか前例ゼロ。
- **証拠**: webapi=in-memory store、saga=synthetic bus、MoviePad=ファイル IO のみ。スキーマは「要求から導出できる核」か「DB 方言という表面」かの判定例が無い。データは**コードと違い再鋳造で交換できない**(永続状態が残る)ため、「コード=鋳造品」の前提が崩れる最初の場所。
- **提案(部分適用)**: silence-checklist に「永続化」カテゴリを追加(適用済み)。本丸は **forward-01 の題材に永続化を含める**こと(§C 参照)。仮説: スキーマ定義=核(ドメイン由来)/ DB 方言・インデックス・マイグレーション手順=表面(K-BOM)/ **マイグレーション=データの変更オーダー**(コードの再製造と違い一方向・不可逆なので、Routing に独立工程が要る)。

### A4. テストベクタの規定が Control Plan に無い — P2
- **現状**: CHEAT-005 の教訓「丸め規定+**中間値テストベクタ**必須」は improvements.md に記録されたが、Control Plan テンプレに置き場(プロパティ)が無く、規律が口伝のまま。
- **証拠**: CHEAT-005(テスト緑でもサイレント乖離。テスト側も中間値ベクタ未整備)。webapi-01 のメタ発見「受入の被覆が捕捉を決める」(seed 受入 6/6 緑のまま blocker S7 通過)。
- **提案(適用済み)**: Control Plan 特性に `test_vectors`(境界値・中間値・反例の列挙。完全性が要る特性は必須)を追加。FMEA の故障モードがベクタ選定の根拠になる。

### A5. 変更管理(ECO)工程が無い — P1
- **現状**: playbook は Phase 6(引き渡し)で終わり、**仕様変更・機能追加の再入口が無い**。Service BOM は「外部知識の劣化」(ffmpeg 更新等)を扱うが、**要求側の変更**は別物。bom_rev(seed→rev1→rev2)は収束ループ内の運用で、納品後の変更オーダーは未定義。
- **証拠**: 実務ソフトウェアの大半のコストは初回製造でなく変更。製造業には ECO/ECN(設計変更指示)が標準装備だが BomDD に対応物が無い。「バグ=劣化→部品交換=再製造」(core-concept)を実行する工程が具体化されていない。
- **提案(適用済み・candidate)**: **Phase 7 変更オーダー**を playbook に追加。`変更要求 → 影響分析(トレース逆引き: REQ→部品→AC→オラクル行) → BOM 改訂(bom_rev++) → 部分再製造(変更された M-BOM unit のみ・fresh factory) → 回帰(既存固定オラクル不変+変更分の新オラクル行) → As-Built/Service BOM 更新`。テンプレ [templates/60-change-order.md](templates/60-change-order.md)。**部分再製造の単位=M-BOM unit**(これが「交換可能な鋳造品」の操作的定義になる)。

## B. BOM→コーディング(製造)で欠けている情報

### B1. 調達部品(外部ライブラリ)が BOM に無い — P1
- **現状**: M-BOM に**依存パッケージの選定・バージョン・固定方針**のプロパティが無く、工場任せ。製造業 M-BOM の「購入部品表」に相当する欄の欠落。
- **証拠**: webapi/saga の M-BOM に調達欄なし(題材が標準ライブラリのみで偶然顕在化しなかった=未観測だが構造的な穴)。依存選定は挙動オラクルに映りにくい一方、再製造・セキュリティ(本物の SBOM=Software Bill of Materials への輸出)・ライセンスに直結する。
- **提案(適用済み)**: M-BOM に `procurement`(package / version / 選定理由 / 代替可否 / license)を追加。silence-checklist に「調達部品」カテゴリ追加。工場には「procurement に無いパッケージを使う判断はずる報告対象」と work order で明示。

### B2. コード規約・プロジェクト構造が未規定 — P3
- **現状**: artifact.path/type はあるが、命名・ディレクトリ規約・lint・コードスタイルは無規定。挙動には出ないため固定オラクルは見ないが、**別工場による部分再製造**(Phase 7)の際に「読めない・混ざらない」コストになる。
- **提案(未適用)**: K-BOM の `design-system` 種別にコード規約パックを置けば現行構造で表現可能。優先度低(挙動契約が先)。多工場で構造分散の実測を取ってから定式化する。

### B3. 組立検査(integration acceptance)が無い — P2(規模の壁)
- **現状**: 受入が**最終成果物への固定オラクル一点**に集中。N=3 は全て 1〜数 unit の小規模で成立したが、部品数十の規模では「unit 単体は AC 通過、組み合わせで乖離」が捕捉できない。製造業の部品検査→**サブアセンブリ検査**→完成検査の中間段が無い。
- **証拠**: saga が初の複数 consumer 統合だったが、観測契約が end-to-end のみ。M-BOM unit 間の interface_contract 適合を直接検査する治具が無い。
- **提案(未適用・要実験)**: M-BOM unit 間の契約に **組立検査行**(consumer-driven contract test に相当)を Control Plan へ追加する案。大規模題材(§C)で必要性を実測してから導入。

### B4. 製造中の blocker エスカレーション手順が無い — P3
- **現状**: ずる報告は「止めずに進んで記録」。BOM の自己矛盾など**進めない級**の発見時プロトコルが未定義(G3 ドライランで前倒しするが網羅は漸近)。
- **提案(適用済み・軽量)**: work order に「矛盾・実装不能を発見した場合: 当該 unit を `blocked` とマークし他 unit を続行、cheat-log に C6(手戻り)で記録して納品時に報告」を明記。製造の中断・往復は隔離を壊すのでしない(fresh re-run が原則)。

## C. 検証パターンの不足(外部妥当性のギャップマップ)

### C1. 題材×特性のカバレッジ
| 特性 \ 題材 | MoviePad(GUI) | WebAPI(HTTP) | Saga(分散) | **未検証の穴** |
|---|---|---|---|---|
| 核/表面・鋳造 | ✓ | ✓ | ✓ | — |
| 多工場決定性 | ✓(Loop4) | ✓(0/1/3) | ✓(0/0/0) | — |
| **永続化・スキーマ進化** | ✗ | ✗(in-memory) | ✗(in-memory) | **最大の穴(A3)** |
| 並行・競合 | ✗ | △(概念のみ) | △(synthetic bus) | 実 DB ロック・レース未踏 |
| 横断ポリシー(認可等) | ✗ | △(API key 1点) | ✗ | A2 と同根 |
| 規模(部品数十〜) | ✗(1スライス) | ✗(4 unit) | ✗(数 unit) | B3 と同根 |
| 既存システム統合 | △(ffmpeg=CLI) | ✗ | ✗ | 外部 API モック・契約テスト未踏 |
| フロントエンド状態管理 | △(描画のみ) | ✗ | ✗ | SPA・リアクティブ未踏 |
| 組込・リアルタイム | ✗ | ✗ | ✗ | FINDINGS §5 future work |
| **変更オーダー(納品後)** | ✗ | ✗ | ✗ | A5。全題材が初回製造のみ |

### C2. 優先順位の提案
1. **永続化を含む小規模 Web/API**(forward-01 の題材に推奨): A3(データモデル表現)・C1 永続化行・さらに playbook 自体の検証を 1 ループで兼ねられる。SQLite 等の軽量 DB で、マイグレーション 1 回を意図的にスコープへ含める。
2. **同題材への変更オーダー 1 件**(forward-01.5): Phase 7(A5)の検証。納品済み成果物に REQ 追加→影響分析→部分再製造→回帰、を実測(部分再製造が全再製造よりずる・コストが少ないか)。
3. 横断ポリシー一級の題材(認可付き API)で A2 の policy 型を試す。
4. 規模実験(部品数十)で B3 組立検査の必要性を実測。

## D. 運用の壁(実用化を妨げるコスト)

### D1. テーラリング不在 — P1
- **現状**: playbook はフル装備(Phase 0–7・ゲート G0–G3・テンプレ16本)のみ。小さなツールに全部は過剰で、**実用では「重いから使わない」が最大の失敗モード**。
- **提案(適用済み)**: playbook に規模別テーラリング表(S/M/L)を追加。**削ってよいもの=文書の分割粒度**(成果物の統合・簡略化)、**削ってはならないもの=測定器**(オラクル・ファースト/工場隔離/ずる報告/沈黙掃討)。測定器を削った運用は BomDD ではなくなる(ずるが見えなくなる)。

### D2. オラクル・治具のコストと再利用 — P2
- **現状**: 固定オラクル・治具は題材ごとにゼロから作る。K-BOM(製造知識)に対応する**検査知識のパック**(observation 側の再利用単位)が無い。
- **証拠**: webapi の blackbox-oracle.ps1 / saga の score.ps1 は同型の構造(固定シナリオ→正規化→等価規則 diff)を2回手作りした。等価規則の `ignores`(JSON 形状・キー綴り・ID 具体値)はドメインを跨いで再出現している。
- **提案(未適用・schema candidates と整合させて検討)**: ドメイン別「**検査パック**」(HTTP 用: status/error/冪等の標準オラクル骨格+等価規則、イベント駆動用: 観測契約+正規化器)を method/ 配下に蓄積する。早すぎる一般化を避け、forward-01 で3個目の同型治具を書いた時点で共通骨格を抽出する(rule of three)。

### D3. E-BOM 品目境界の規準が薄い — P2
- **現状**: 「どこで部品を切るか」の規準が暗黙(open questions: 設計工程の自動化=最大レバレッジ点)。プロンプトは型分けを指示するが粒度を指示しない。
- **提案(適用済み・1行規準)**: M-BOM unit の粒度に操作的定義を与える — 「**独立に再製造・交換でき、単独で受入できる最小単位**」(Phase 7 の部分再製造単位と同一)。E-BOM 品目はその設計版(存在理由 1 つ・受入観点が閉じる単位)。粒度に迷ったら「この部品だけ fresh factory に再製造させられるか?」で判定。
