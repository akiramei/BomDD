## 1. 総括

3 体の自己評価は独立に同じ判定へ収束した: **「条件付きで実行できる(質問への回答があれば着手できる)」**。3 体とも「方法論の骨格(フェーズ・ゲート・テンプレ・隔離・オラクル・ファースト・ずる報告)は kit 内で自己完結」と評価し、障害は方法論の論理でなく **(a) 工具の梱包**(PLM 工具・オラクル実行系の実例が kit に無い)と **(b) 規範の優先順位**(テーラリング S 列と prompts 正典の衝突)に集中した。

重複統合後の相異なる所見は **30 件**。うち 3 体一致 **6 件**・2 体一致 **8 件**。読者間の検出差分も観測された(A は「G4」未定義を見逃し B が検出、C のみ 60-change-register テンプレ欠落を検出)— マルチリーダー方式の検出力の追加実証。

## 3. メタ観測(方法論への含意)

1. **穴の主成分は「論理」でなく「梱包と優先順位」**: playbook_unspecified_residue(手順自体の欠落)は Phase 6 ゲート等少数で、大半は kit_packaging_gap と playbook_ambiguity。転移可能性の見通しとしては良い兆候(ただし片側検出器の注意の下で)。
2. **§13(手順の工具化)の予言どおりの故障様式**: 検出された穴の多く — silence §14 の未追随(T0-06)・prompts/README の未更新(T0-16)・deprecated 参照(T0-15)— は §13 運用規則「標準手順の改訂には波及監査を付ける」が名指しする型そのもの。方法論は自分の故障様式を正しく予言していた。
3. **G2/G3 の転用は機能した**: 読者間差分が実際に曖昧箇所を指し(T0-02 は 3 体全員が「割れる」と申告)、単独所見の相補性(G4/60-register/deprecated がそれぞれ 1 体のみ検出)は多リーダーの価値の追加実証。
4. **T0-15 は昇格裁定の材料**: deprecated 参照の掃討 lint は §13 で「候補(2 例目で昇格裁定)」と記帳済み。本件が 2 例目 → 昇格裁定をレビューに付す。

## 4. 裁定案(レビュー停止点 — 採否はユーザー裁定)

補正の目的は T1 前に**確認済みの穴**を塞ぐこと(G3 → BOM 補正 → 製造の相似形)。T1 は補正後 kit の残渣を測る。

### P1 — T1 前に補正すべき(blocker 級)4 件
1. **T0-01**: テーラリングに PLM Gate 行を追加 —「PLM 工具が無い環境では plm-ready-contract の手動照合で代替し、その旨を charter に記録する(逸脱ではなく正規の縮退)」。または kit に bomdd-lint を同梱。
2. **T0-02**: 優先規則を 1 行明文化 —「規模テーラリング(§11)は prompts 正典に優先する。縮退の採否は Phase 0 で裁定し charter に記録する」+ plm-ready-contract §13 に S 規模の統合ファイル読み替えを追記。
3. **T0-05**: Phase 6 の完了ゲートを定義(「G4 = チャーター『完了の定義』の充足+納品物一式の確認」を正式化)し、example log の G4 と接続。
4. **T0-03**: オラクル実行系の最小規約を 1 節追加(自作でよいこと・exit code 規約・セルフテストの合格基準・最小実例 1 本の同梱または参照)。
