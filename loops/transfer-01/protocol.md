# transfer-01 — fresh 担当者転移テスト(playbook 自己完結性の検査)プロトコル

> **ステータス**: protocol draft(未レビュー・実施前)。起票 2026-07-08。
> **研究問い**: playbook とテンプレートだけを渡された、設計文脈を持たない fresh 担当者は、説明者の介入なしに BomDD の一周を実施できるか。詰まる箇所はどこか。
> **位置づけ**: RQ-A(第三者再現性)の予備実験。人間第三者より弱い証拠だが、単独・低コストで実施でき、Paper 1 級実験の設計を de-risk する。

## 0. 設計原理 — 方法論の自己適用

本テストは新しい装置を持ち込まない。playbook が既に持つ装置を playbook 自身へ向ける:

| 既存装置(典拠) | 本テストでの適用 |
|---|---|
| G3 ドライラン(playbook §4.6): fresh AI にパッケージだけを渡し「着手前に必要な質問を全て挙げよ」→ 質問=BOM の穴 | **kit を playbook 一式、fresh AI を転移先担当者にして同じ問いを立てる** → 質問= playbook の穴 |
| G2 マルチリーダー監査(§3.2): 独立読者の抽出差分=一意に読めない箇所 | 複数 fresh 読者に手順・停止点・完了条件を独立抽出させ、差分= playbook の曖昧箇所 |
| 差分帰属(§6.1)・fresh 転移(§6.3): fresh 工場での成功=「修正が BOM に宿った」証明 | 介入・詰まりの帰属分類(§4)/ fresh 担当者での完遂=「方法論が文書に宿った」証明 |
| 前倒し検出率(§10 ①): G3 質問数 vs 製造後残渣数 | T0 検出数 vs T1 残介入数 |

見立て: **playbook = 製造パッケージ、fresh 担当者 = 工場、方法論の一周 = 製造、介入 = unspecified residue**。

UI-CAD 系列の中心的発見「職人芸は翻訳層へ移動する」が本テストの動機である: playbook を渡しても発明者の暗黙知が「説明」という翻訳層に滲み出れば、第三者再現(H3 可搬性)は偽装される。説明の総量と内容を測定対象にする。

## 1. 段構成

```
T0  静的検査(同一セッション・サブエージェント)
    ├ G2型: fresh 読者 2 体が手順・停止点・完了条件を独立抽出 → 差分=曖昧箇所
    └ G3型: fresh 読者 1 体が着手前質問を全件列挙+A/B/C 分類
      ↓
補正  T0 検出穴の裁定(採用/棄却はレビュー停止点。§3.2 の裁定規律を準用 —
      監査者の誤読は理由付きで棄却し、全件採用による最小公倍数化を避ける)
      ↓
凍結  kit を commit/tag で凍結(T1 期間中は不変 — 同一ヤードスティック)
      ↓
T1  動的転移テスト(別ディレクトリ・新規 Claude Code セッション・新規小型題材)
    fresh 担当者が Phase 0 → 納品まで一周。観測者は介入台帳を記録
```

T0 → 補正 → T1 の関係は G3 ドライラン → BOM 補正 → 製造の相似形。T0 は前倒し検出器、T1 は残渣測定器。

## 2. kit の組成(manifest)

転移キット=公開リポジトリの文書一式。**新規題材を採用したため遮断規則は最小限**(既存証拠リポへのリンクは題材の答えを含まない)。

- 含む: `README.md` / `WHITEPAPER.md` / `FINDINGS.md` / `method/**`(playbook・method-v1・templates・prompts・checklists・onboarding・tools)/ `docs/**`
- 含まない: `paper/` / `loops/transfer-01/`(本プロトコル・顧客ブリーフ・台帳 — **担当者へ非開示**)/ **`method/attribution-scoring.md`**(採点思想は評価用であって運用義務でない — 担当者に採点の観点を事前開示しない)/ このセッションの記憶・設計対話
- T1 の担当者への追加供与: 顧客役への連絡手段、作業用ワークスペース、(必要なら)BomDD-Plm の bomdd-lint(§6 受入で使用する場合は kit に含めて凍結。含めない場合は PLM-lite — plm-ready-contract §12.5 — の初回実測になる)

### 2.1 T1 開始条件(凍結チェックリスト)
1. T0 の P1 所見が解消済み、または明示的に waive 済み
2. playbook が参照する必須テンプレートが kit 内に全て実在する
3. 正典・実例に未定義のゲート名が残っていない
4. テーラリングの優先規則(権限分離)が宣言済み
5. オラクル実行系の最小契約が kit に含まれる
6. kit が commit/tag で凍結済み
(**全 P2 解消は要求しない** — 残 P2 は T1 の測定対象)

## 3. T1 の役割分担(§9 役割規律の転用)

| 役割 | 担い手 | してよいこと | してはならないこと |
|---|---|---|---|
| fresh 担当者 | 別ディレクトリの新規セッション(プロジェクト記憶なし) | kit の参照・顧客への質問・一周の全作業 | kit 外の BomDD 情報の参照(証拠リポ実装の丸写し含む) |
| 顧客役 | ユーザー(または設計 AI が代行) | **業務要件の裁定**(凍結済み顧客ブリーフの範囲で回答) | 方法論・工程・成果物形式の指南。聞かれたら定型文「あなたの手順書に従ってください」で返す |
| 観測者 | 本セッション側 | 介入台帳の記録・ゲート観測 | 担当者への助言 |

顧客対話(要求の裁定)は**合法・無制限**。方法論の説明・救済だけが介入としてカウントされる — これが RQ-A の「説明者の介入量を記録する」の実装形。

## 4. 測定

### 4.1 介入台帳(一次データ)
全交信を記録し分類する([intervention-ledger.md](intervention-ledger.md)):

| 分類 | 意味 | 扱い |
|---|---|---|
| `customer_ruling` | 業務要件の裁定(顧客ブリーフ内) | 合法。件数は参考記録 |
| `method_explanation` | playbook の読み方・手順の説明 | **介入**。playbook 未規定の証拠 |
| `rescue` | 停止からの救済(説明なしでは先へ進めなかった) | **重大介入**。playbook の穴 |
| `tooling_gap` | 環境・ツール・スキル未配布による停止 | kit 梱包欠陥(方法論の穴と区別) |

各介入には追加で 2 問を記録する(成功の帰属を後で分離するため):
- `would_have_progressed_without`: この介入なしで担当者は進めたか(yes / no / unclear)
- `changes_attribution_of_success`: この介入は成功の帰属を変えるか(yes / no / unclear)

### 4.2 詰まり・介入の帰属(終了時)
各介入を §6.1 の語彙の転用で帰属する:
`playbook_unspecified_residue`(playbook に書いていない)/ `playbook_ambiguity`(書いてあるが一意に読めない)/ `kit_packaging_gap`(文書は正しいが配布物が欠けた)/ `operator_capability`(書いてあり一意に読めるのに外した=担当者能力)/ `subject_specific`(題材固有で一般化しない)。

### 4.3 集計指標
1. 介入件数と分類分布(主指標。目標は method_explanation + rescue = 0)
2. **前倒し検出率**: T0 検出穴のうち T1 で実際に介入・詰まりとして再現した数 / T1 介入総数(§10 ① の相似形)
3. ゲート通過状況(G1/G2/G2'/G3・機械ゲートの exit code)
4. 成果物完全性(onboarding pack §7 必須フィールドの充足。bomdd-lint が使えるなら機械判定)
5. 製造品質(オラクル合否・残渣・ずる件数)— 参照点: forward-01(残渣 0・収束ループ 0・G3 質問 7+矛盾 2)
6. 工数(担当者セッションの実時間・介入までの経過)

集計は最終的に 4 軸へ要約する: **A. 自己完結性**(kit だけで計画・成果物・ゲート・記録様式を理解できたか)/ **B. 実行忠実性**(playbook の要求を過不足なく実施したか。省略時に正しい根拠を記録したか)/ **C. 介入依存度**(介入なしにどこまで進んだか。介入の成功寄与)/ **D. 帰属可能性**(失敗・逸脱を文書欠陥・題材曖昧性・担当者判断・観測者介入へ帰属できたか — attribution-scoring.md を事後採点器として使用。運用義務にはしない)。

### 4.4 成功・失敗の判定
- **成功**: method_explanation+rescue が 0(または全件が軽微な導線問題)で一周完了 → 「playbook は fresh AI 担当者に対して自己完結」の初証拠
- **失敗も成果**: 各介入=playbook の改訂候補。lesson-promote の入口へ送り、補正後の transfer-02 で再測定(fresh 転移の証明形式と同じ: 補正が文書に宿ったかは次の fresh で測る)

## 5. 妥当性への脅威(事前申告)

1. **同一ベンダー慣習の共有(C2 型)**: fresh 担当者も Claude であり、発明者側 AI と慣習を共有する。人間第三者や他モデルなら詰まる箇所を素通りしうる。→ 本テストの合格は「AI 担当者への転移」までの主張に限定。多様化(他モデル担当者・人間担当者)は transfer-02 以降。
2. **T0 の文脈汚染**: T0 読者は本セッションのサブエージェントで、プロジェクト文脈の注入を完全には遮断できない。汚染は理解を底上げする向きに働くため、**T0 は合格が弱く不合格が強い片側検出器**として扱う。T1 は別ディレクトリ新規セッションでこれを遮断する。
3. **顧客役の漏洩**: 顧客役が方法論知識を持つため、回答に工程知識が滲むリスク。→ 顧客ブリーフを事前凍結し、ブリーフ外の質問には「業務上どちらでもよい/あなたが提案して」型の回答に限定。逸脱は台帳に自己申告。
4. **題材の易しさ**: 小型題材での成功は規模一般化を主張しない(scale-01 の教訓)。

新規小型題材(LibraryLending と同クラス・未使用ドメイン)。詳細は [customer-brief.md](customer-brief.md)(**担当者へ非開示**。担当者はブレストで引き出す)。永続化(SQLite+移行 1 回を誘発する ECO)をスコープに含める(§10 の推奨に従う)。
