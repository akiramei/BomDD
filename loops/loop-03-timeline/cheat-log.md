# ずる台帳 — Loop 3 (タイムライン描画スライス)

分類は [../../method/cheat-taxonomy.md](../../method/cheat-taxonomy.md)。

### CHEAT-010  [C2 暗黙知/設計知識] 幾何の定数群は要求から導けず、BOM が設計知識を運んで成立
- 発生段: ①リバース/③M-BOM/⑤製造
- 手法が与えなかったもの(を明記して回避): レイアウト定数 `TrackTop=30`/`TrackH=92`、`NiceStep` の目標 `70/px` と steps 配列 `{0.5,1,2,5,10,15,30,60,120,300}`。
- 観測: これらを BOM が運んだので Unit G は単体テスト全green(industrializable)。だが値自体は要求から導けない**設計知識**。CHEAT-007(ffmpeg白名簿)と同型 — 「設計」=知識転記。
- 重大度: friction。想定修正: E-BOM に「設計トークン/レイアウト規約」を明示部品化し、デザインシステム(Tokens)へトレース。

### CHEAT-011  [C4 受入不能] 知覚的受入は要求由来オラクルを持たず、golden は妥当な設計差を拒否する(CHEAT-006 の特性化)
- 発生段: ⑤製造/⑥合否
- 手法が与えなかったもの: 「速度→ティント色」の対応。原版は OKLCH 設計トークンだが要求からは導けない純粋な**設計判断**。BOM 未規定にした。
- 観測(800x122=97600画素): 指定トークン+幾何は**画素完全一致**(67.5%)。製造装置は速度色を独自発明(遅い=青/速い=オレンジ赤)→ **32.5%(31,672画素)が乖離、すべて速度ティント矩形に局在**(領域外差分=0)。
  - **画素一致 golden は妥当な設計差を「失敗」と判定**(過剰結合)。製造物の配色は valid な設計だが golden と違うだけで不一致。
  - 「正しい見た目」は要求から導けない設計知識であり、**golden は人間が authoring/承認**せねば存在しない。→ golden-image は問題を「誰が golden を作る/承認するか」へ移すだけ。
- 重大度: blocker(知覚部品の工業化の核)。
- 想定される手法的修正:
  1. 設計トークンを E-BOM/M-BOM の一級部品にし(色・寸法・タイポ)、デザインシステムから供給=知覚部品の「外部知識」を要求側へ固定。
  2. 受入は golden(L3)だが、**許容差(色差/SSIM 閾値)**と**承認者(デザイン責任者)**を AC に明記。pixel-exact は過剰結合、緩すぎる許容差は微小レイアウト崩れを見逃す(二重性, H9)。
  3. 「要求で決まる部分(幾何・配置)」と「設計で決まる部分(配色・装飾)」を E-BOM で分離し、後者は defect でなく**設計変更**として扱う。
