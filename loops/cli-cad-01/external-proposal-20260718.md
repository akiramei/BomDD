## 原文(転記)

NGではありません。むしろ、BomDDのCADは本来UI/UX専用ではなく、「実装前に製品の意味・構造・振る舞いを固定する設計原器」と捉えるのが自然です。

現在のBomDDでUI-CADが目立つのは、画面の見た目という外部表面が文章だけでは欠落しやすく、HTMLモックやキャプチャという明確なCAD表現を必要としたためです。サービス、API、CUIにも、それぞれに適したCAD表現があります。

### BomDDにおけるCADの位置づけ

広義には、次の全体がソフトウェアCADに相当します。

```text
要求・設計意図
    ↓
ドメインモデル/状態機械/外部契約
    ↓
E-BOM(論理部品)
    ↓
M-BOM・Control Plan(製造・検査条件)
```

UI-CADはこのうち、画面という外部表面を精密に表現する専用CADです。

| 対象 | CAD表現の例 | 主な受入 |
|---|---|---|
| GUI | HTMLモック、画面状態、遷移、visual contract | UI自動テスト、golden、人間承認 |
| API | OpenAPI、スキーマ、状態遷移、エラー契約、シナリオ | ブラックボックスHTTPオラクル |
| イベント/Saga | イベント契約、状態機械、シーケンス、補償規則 | 正規化イベントログ、dispatch trace |
| バッチ | 入出力スキーマ、処理規則、再実行契約 | fixture比較、件数・内容検査 |
| CUI | コマンド文法、help、出力、終了コード、対話遷移 | subprocessによるブラックボックス検査 |
| DB/データ | 概念モデル、制約、効力、移行規則 | migration oracle、データ同値検査 |
| 内部サービス | ポート契約、状態機械、不変条件、障害モデル | unit/component/観測契約 |

### API-CADは十分に成立する

例えば予約サービスなら、API-CADには次を含められます: リソースと識別子/エンドポイント/request・response schema/status codeとerror code/認証・認可/冪等性/日時とタイムゾーン/同時更新時の規則/状態遷移/ページング、並び順/retry、timeout/後方互換性/代表シナリオ/未規定次元。

OpenAPIだけでは少し弱いです。OpenAPIは「外形」をよく表しますが、次のような意味契約は落ちやすいからです。

- 同じidempotency keyなら同じIDを返す
- キャンセル後の再キャンセルをどう扱うか
- 競合時にどちらを勝たせるか
- 日付境界をUTCと利用者ローカルのどちらで判定するか
- リトライによってイベントを重複発行しないか

したがって、API-CADは概ね次の組み合わせになります。

```text
OpenAPI/JSON Schema
+ ドメイン状態機械
+ 振る舞いシナリオ
+ エラー・冪等・時刻・同時実行契約
+ 非機能条件
+ 未確定台帳
```

OpenAPIからサーバーコードを生成できることより、「別のAIが原版を見ずに同じAPI意味論を製造できること」がBomDD上のCAD品質になります。

### APIがない内部サービスもCAD化できる

外部HTTP APIがないサービスでも問題ありません。例えば画像スキャンサービスなら、入力ポート/出力イベント/状態機械/バッチ公開条件/キャンセル意味論/再試行/順序保証/重複排除/障害時の部分結果/永続化の確定点をCAD化できます。

この場合、実装内部のクラス名をCADにするのではなく、外部から観測可能な「観測契約」を設けます。

```text
固定シナリオを投入 → 正規化イベントログを出力 → 期待する状態・順序・回数と比較
```

### CUIのCAD化はかなり相性がよい

CUIはCAD化に向いています。GUIより外部表面が決定的で、headless検査しやすいためです。CLI-CADとしては、次を定義します。

**コマンド文法**: サブコマンド/オプション/必須・任意/排他条件/既定値/省略時の挙動/引数の解釈/オプション順序の扱い。

**出力契約**: stdoutとstderrの使い分け/text出力の意味/JSON出力スキーマ/文字コード/改行/locale/並び順/色やANSIエスケープ/TTYとリダイレクト時の差/secretのマスキング。人間向けtext出力は通常は全文完全一致にせず、見出し・必須フィールド・意味・重要な順序を検査する。機械可読出力(`--format json` 等)はスキーマや正規形を厳密に固定できる。

**終了契約**(例):

| Exit code | 意味 |
|---:|---|
| 0 | 正常、問題なし |
| 1 | コマンド実行成功、検査上の問題あり |
| 2 | 引数・設定不正 |
| 3 | 外部依存または一時的障害 |
| 4 | 内部エラー |

「失敗した」という文章だけでなく、シェルやCIが判定する終了コードが製品契約です。

**対話状態**: 対話型CUIなら状態機械がCADになる(起動→入力待ち→確認→実行中→成功/失敗→再試行/終了)。Ctrl+C/EOF/timeout/キャンセル後の部分成果物/確認の既定値/非TTY環境での動作/`--yes`指定/CI上で対話が発生しないことも明示する。

**設定優先順位**: `command-line > environment variable > project config > user config > built-in default`。この優先順位、設定ファイル探索位置、存在しない場合、壊れている場合までCADに含める。

### CADとオラクルは分ける

- CAD: 何を作るべきか
- Work Order: 今回どの範囲を製造するか
- Oracle: 製造品がCADの契約を満たすか
- Exploratory probe: CADが未規定の差を観測する

例えばOpenAPIからサーバーとテストを同時生成し、それだけで合格させると、同じ解釈ミスを共有する危険があります。API-CADから独立したブラックボックスシナリオを作り、HTTP越しに検査する方が安全です。CUIも同様に、CLI内部の関数を直接呼ぶだけではなく、実プロセスを起動して、引数・stdout・stderr・終了コード・生成ファイルを観測します。

### 「CAD」と呼ぶための最低条件

- その領域の設計上の正本である
- 実装コードより上流にある
- 設計意図と存在理由を持つ
- インターフェースと不変条件を持つ
- 状態・例外・境界条件を表す
- 未確定事項を未確定のまま保持できる
- 変更履歴と裁定主体を持つ
- E-BOM/M-BOM/Control Planへトレースできる
- 別の製造者が原版なしで実装できる
- 独立した検査方法を導出できる

単なるクラス図、OpenAPIファイル、画面画像、コマンド一覧だけでは「CAD部品」ではあっても、単独では完全なソフトウェアCADとは言いにくいです。

### 用語としての提案

```text
Software CAD
├─ UI-CAD
├─ API-CAD
├─ CLI-CAD
├─ Workflow-CAD
├─ Event-CAD
└─ Data-CAD
```

ただし、これらが別々の島になってはいけません。例えばCUIの`scan`コマンドなら、CLI-CAD → Scan要求 → E-BOM Scan部品 → Scanner service契約 → DB更新契約 → CLI exit/output Control Plan まで接続します。

結論として、UIのないサービスやAPIをCAD化することはまったくNGではなく、BomDDの自然な拡張です。CUIも非常に適しています。UI-CADが「見える画面の設計原器」なら、API-CADやCLI-CADは「観測可能な契約の設計原器」です。媒体は異なりますが、実装から独立した正本を作り、BOMと検査へ接続するという役割は同じです。
