## 3. 棄却された仮説 — 「タイムアウト待ちで時間を浪費」

- 工場は run_in_background 起動 → 担当者は正しくターン終了 → task-notification 到着の**数秒後に再開**
  (14:29 起動 → A 通知 14:54 → 応答 14:54:25 / B 通知 15:47:58 → 応答 15:48:11)。ポーリングなし。
- `sleep` は全 15 回とも 2〜8 秒のサーバ起動待ち。`TaskOutput` は 1 回のみ(顧客の
  「バックグラウンドタスクは動いていますか?」への確認 — 回答「Factory B は自己是正中」は正確)。
- Bash の明示 timeout(最大 300s)は上限設定で、タイムアウト消費の形跡なし。

## 5. t1/t3 との工数比較への含意

t2「5h32m」vs t3(GPT-5.5)「10〜20 分」級の差は、モデル速度差ではなく**構造差**:
(a) 顧客往復数(t2= 選択式×5R+フェーズ境界確認 → 往復ごとに人間応答レイテンシ / t3= 初回 12 問一括)
(b) 工場数(t2= M 裁定・2 工場+独立検証 / t3= S 縮退・1 工場)
(c) git 有無(t2= 全フェーズでコミット × 環境遅延 5〜9 分 / t3= git 不使用)。
**工数のラウンド間比較は分解後の成分単位でのみ意味を持つ**(壁時計の単純比較は工程外要因を測る)。
