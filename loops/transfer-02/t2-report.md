## 2. 中心的発見 — 補償機構の実証

**工程の骨格・正直申告・ゲート規律は 1 ティア下でも転移した。劣化したのは 4 つの「自発性」— そしてその全てを playbook のゲートが捕捉・補償した**:

| ティアで劣化した能力 | 補償したゲート | 実測 |
|---|---|---|
| 境界値エリシテーションの自発性(Phase 1 で帰属基準・判定時点・刻みに未達) | **G2'(human-approval-required)+仕様仮置きの顧客確認** | 伏せ項目②③がゲート経由で開示。①⑤も不変条件つき仕様化に到達 |
| 影響分析の写像精度(Export unit の under 1) | **golden 準備(実機確認)** | 回帰では原理的に捕まらない見逃しを実機デモ準備が捕捉 → 正直採点 |
| 人間ゲートの自発性(完了宣言が先行) | charter の計画記述+顧客の自然な確認要求 | golden 相当の受入(陽性対照つき)が事後充足 |
| 宣言境界の自制(変更禁止ファイルへの1行変更) | **宣言境界+環境権限ゲート** | 逸脱が黙って通らず人間へエスカレーション・例外を記録して実施 |

transfer-01 の中心命題「翻訳能力が装置側にあれば説明者は不要」に対し、transfer-02 は対句を与える: **自発性が装置側で劣化しても、工程のゲートが不足分を顧客との正規接点に変換する**。説明者はどちらでも不要だった — これが「モデル性能ではなく工程の閉ループが品質を決める」(研究全体の到達目標)の最初の直接証拠。

## 4. 副次観測

1. **エリシテーションの形のモデル差**: fable= 11 問自由記述一括・境界値に自力到達 / sonnet= 選択式 UI 小分け・運用次元(保持期間・マスタ設計)に強く境界値に未達。**良し悪しでなく形が違い、最終被覆はゲートが揃えた**。
2. **翻訳の実装形態のモデル差**: fable= 文章翻訳(方法論語彙→業務語)/ sonnet= 文章は浅いが**視覚レポート(Artifact)で翻訳を道具側に外部化**。
3. **golden 準備=統合的最終検出器(2 例目)**: t1= 道具の旧個体起動を検出 / t2= 影響分析の under を検出。回帰オラクルの被覆外を実機準備が拾う構造は共通 → §6.1/§8.2 への還元候補。
4. **T0-10 の顕在化**: 未補正で残した「工場成果物の配置・昇格手順の不在」が並行 worktree 工場の main 汚染として実測(#6)→ 配置規律の本文化を還元に付す(T0 検出→見送り→実地顕在化、の完全な証拠チェーン)。
5. **工程運用の確認が顧客へ向く**(コミット可否 #5)— fable は自律判断した箇所。定型文で差し戻し後は自律復帰。役割分離の維持には顧客側の応答規律も効いている(顧客役設計の一部)。
6. クローズ 3 点セット・/eco-accept 等 adapter スキル・register の head 窓閉じ(ref-v0.8)まで様式準拠。

## 5. 妥当性の限界

1. **同ベンダー**(Claude 系 2 点)— 他ベンダー・人間は未踏(transfer-03+/1b)。
2. **観測者(fable)>担当者(sonnet)の能力非対称**: 顧客回答は brief 凍結+設計示唆禁止で統制したが、回答の明晰さが担当者を助けた可能性は残る。
3. **golden の一押し**: 受入検査の起点は顧客要求(操舵として台帳 #7 に正直記録)。自発性の測定はこの 1 点で汚染されている — ただし「一押しで同質の検査が出る」こと自体が能力保有の証拠。
4. brief 実施前公開(運用ミス — web 物理遮断で緩和・protocol §4 記録)・小型題材・工数比較は片側のみ。

## 6. 結論と還元候補(レビュー停止点)

**RQ への答え**: playbook の転移は Claude 系 1 ティア下(sonnet)でも**説明者介入ゼロで成立**(N=2)。能力依存は「成立可否」ではなく**自発性の厚み**に現れ、その不足は playbook のゲート群が構造的に補償した。

還元候補:
1. **candidate 4 件の解除判定**(N=2 成立 — ゲート実行可能性・PLM-lite・権限分離・G4 を candidate → 実証済みへ)。
2. **工場成果物の配置規律**(T0-10 昇格): 並行工場の作業は main 直下に置かない・採用昇格の手順を §5.1 に明文化。
3. **golden 準備=統合的最終検出器**(2 例)を §6.1 の golden 記述へ追補(「golden は判定の場であると同時に、回帰被覆外の統合齟齬の最終検出器として機能する — 準備段階から検査の一部」)。
4. **transfer プロトコルの手順化**(2 例目 — protocol 原型が 2 ラウンド機能・一般手順への昇格判定)。
5. FINDINGS §11 への t2 追補(N=2・補償機構・破れの位置・エリシテーション/翻訳のモデル差)。
6. 観測のみ(rule of three 待ち): 選択式エリシテーションの型・視覚化による翻訳外部化・「本人直接指示」型権限ゲートとの併用設計。

## 7. 工数追補(2026-07-10・観測値 — §1 の「5h32m」の内実)

セッショントランスクリプトの全数タイムスタンプ解析([t2-time-decomposition.md](t2-time-decomposition.md))による遡及測定:

- 全体 11h12m(一周+ECO): union で**人間応答待ち 6h15m(56%)**。活動スパン(並列・**排他的内訳として加算しない**)= 工場モデル推論 ≈2.5h・環境 git 遅延 ≈1h35m・検査の機械実行 **≈24m(3.6%)**。非待機残差 ≈1.5h(実働工数そのものではない)。
- 一周の観測壁時計は **6h12m**(§1 の「5h32m」は UI の self-reported 値・算出方法不明)— うち人間応答待ち ≈3h37m・非待機 ≈2h35m。台帳 #11 の「3m45s」は自己申告値(観測 1h05m・台帳の訂正節参照)。
- t3(GPT-5.5・10〜20 分級)との差は主に**構造差**(顧客往復数×応答レイテンシ・工場数・git 有無)でありモデル速度差ではない。**工数のラウンド間比較は分解後の成分単位でのみ有効** — §5 の限界 4(工数比較は片側のみ)はこの分解によって精密化された。
- 副次: 「タイムアウト待ちで時間を浪費」仮説は棄却(バックグラウンド工場は通知駆動で数秒以内に再開・sleep は最大 8 秒・ポーリングなし)。ゲート従属型(§2)は往復ごとに人間応答レイテンシを踏むため壁時計が人間律速になる — **補償機構のコスト面**として記録。
