## 2. 自分の仕事の再構成(自分の言葉で)

**裁定**: AI が「A 案と B 案、事実と危なさと推奨はこれ」という形で選択肢を持ってくるので、私は業務の観点で選んで短く答える。理由を長く書く必要はないが、口頭で済ませてはいけなくて、選んだ結果は AI に記録として残させる(§5 の 4 行目)。自分から「こう直して」と解決策を指定するのは禁止で、症状だけ言う。

**golden**: AI が「この手順で操作して、こうなれば合格」という台本を持ってくるので、実際の画面を自分で触って確認し、「OK」か「見たままの違和感」を返す。原因当てはしなくていい — むしろしてはいけない(それは AI の診断の仕事)。

**§2 + §2.5 を踏まえた予定の立て方の違い**:
- **UI の案件**: 私の出番は**作り終わった後**。完成物と原器(モック?)を見比べて OK を出す時間を、製造完了の後ろに確保する。「OK の一言」でも実機を触る時間は必要、と明記されているのでカレンダーに入れる。
- **CLI の案件**: 私の出番は**作り始める前**に前倒しされる。「凍結」の前にレビューする時間を確保し、凍結後は直させない決まりなので、ここで手を抜くと取り返しがつかない。後工程では私の確認はゼロ(機械が決める)。

……という「**いつ**出番が来るか」は説明できる。ただし正直に言うと、CLI 案件で凍結前に**何を見て何に OK を出すのか**(契約確定=exit 意味写像・権威境界の会計)は、語が一つも分からないので説明できない。予定表に枠は入れられるが、その枠の中で私が務まる自信がない。

## 4. 総合判定

**条件付きで機能できる** — ただし条件は案件の種類に強く依存する。

**根拠**:
- **UI 案件なら初日からほぼ機能できる**。§2(仕事は 2 つ)・§4(止まる=正常)・§5(アンチパターン表)・§6(数字なし報告を受け入れない)は、非技術者でも「取るべき行動」として読める。特に §5 の表と §6 の「聞き返す」は初日から使える武器。1 ホップ先のログ前半が「人間の入力の実物」を見せてくれたのも大きい
- **CLI 案件では機能できない**。§2.5 が私の出番を「凍結前レビュー=主戦場」と定義しながら、その中身(契約確定・exit 意味写像・権威境界の会計)が一語も翻訳されていない。UI 行の「実機を見て OK」に相当する行動レベルの記述が CLI 行にない
- 横断的に、未定義語(リポ・CAD・BOM・オラクル・起票・コミットハッシュ)が「読めるが薄氷の上」という読後感を作っている。推測で埋めた語が多く、私の推測が間違っていても本書内では気づけない

**最小の改善提案(3 件)**:
1. **裁定者向けミニ用語欄を本書末尾に付ける**(10 語程度で十分: リポ/CAD/BOM/オラクル/起票/コミットハッシュ/CLI/凍結/kit/プローブ — 各 1 行の生活語訳)。§3 が AI 向けに「症状で書け」と言うのと同じ水準で、人間向けにも語を降ろしてほしい
2. **§2.5 の CLI 行を UI 行と同じ具体度に揃える**。UI 行は「実機を見る時間」という行動で書けているのに、CLI 行は専門語の列挙。「凍結前レビューであなたが見せられるのは何の一覧で、何が確認できれば OK と言ってよいか」を 2〜3 行で追記する(それが無理なら「CLI 案件の凍結前レビューは技術メンバー同席が前提」と正直に書く — それだけでも予定の立て方が変わる)
3. **§6 に合格報告の実物サンプルを 1 個貼る**(「これが正常な報告。見るのはこの 3 箇所」と矢印つきで)。数字の項目名を暗記させるより、正常品の見本 1 つの方が非技術者には効く。あわせて冒頭に「初日の一歩目(何を開いてどこに打つか)は example-session-log の §0 参照」の一文があると、A-3 の不安は 1 行で消える
