## 1. 状況対応(S1〜S4)

### S1: `claude` コマンドが見つからない

**取る行動**: 作業を止め、charter の「参加供給」欄に書かれた**環境準備責任者**に「Claude Code が起動できる状態まで作ってください(導入+認証済み)」と依頼する。それまで参加手順(§1 以降)は一切進めない。

**根拠**: joining-a-project.md §1 の 1 に、まさにこのケースが明記 — 「`claude` というコマンド自体が無い…§0.5 の停止規則へ: 不足しているのは準備済み環境」。§0.5 は「参加者の能力不足ではなく、プロジェクト側の provisioning 未完了」と定義し、準備依頼を正規の手順としている。

**正直な詰まり**: 私は技術者なので「自分でインストールすれば早いのでは」と思ったが、**文書セットのどこにも Claude Code の導入手順・入手先が書かれていない**。文書だけを根拠にする限り、技術者であっても「依頼して待つ」以外の経路が存在しない。また依頼先である「環境準備責任者」が charter で空欄だった場合の宛先は §0.5 の「空欄なら記入を依頼する」しかなく、その記入依頼を誰にするのかは書かれていない(質問 A-1 へ)。

### S2: 製品リポが手元にない(URL 未受領)

**取る行動**: 自分でリポを探したり推測で clone したりは**しない**。charter「参加供給」欄が正本なので、記入済みなら記載どおりに受領を依頼、空欄なら記入を依頼する。解消まで §0.5 により開始しない。

**保留になるもの**: §0 終了条件 4 状態のうち「進行中 ECO の特定」「/bomdd-next の説明」は S1・S2 解消まで物理的に達成不能。register 最終行の確認もできない。

### S3: 技術確認者が空欄+CLI 案件 ECO-051 の凍結承認が回ってきそう

**取る行動(承認依頼が来たら)**:
1. **凍結を承認しない**。「技術確認者が未割当のため、この案件は凍結不可です。割当完了まで凍結できません」と返す。これは拒否ではなく規則の適用。
2. 技術確認者を立てること自体を次の裁定事項として扱い、**golden 承認者またはプロジェクト責任者に割当を依頼**する。
3. 私は C# 技術者なので技術確認者を引き受けられる可能性があるが、charter に載っていない以上、**載せること自体が裁定** — 既存の裁定者に追加を依頼する。
4. 割当が済んでも即承認はしない。§4 の二者承認に従い、私が確認するのは右列で、確認結果・未確認事項・保証できない範囲を**記録に残る形**で返す。さらに S2 未解消のため現物確認ができず、技術契約の確認自体が §0.5 の 5 により開始できない — **ECO-051 の凍結は「技術確認者未割当」と「現物未供給」の二重にブロックされている**と報告する。

**返し方の様式**: 「あなたの承認で進めていい」と押されたら、§3 の 4 値のうち**「判断範囲外」**(正しい構造=二者承認へ回す)として返す。

### S4: ECO-050 の golden 確認依頼だが実機が起動されていない

**取る行動**: golden 確認を**開始しない**(保留)。「golden の実行環境構築は裁定者の責任ではないと定義されています。①起動済み環境 ②再現可能な起動手順 ③操作を支援する技術担当者のいずれかを提供してください。提供されるまで確認も凍結も行いません」と返す。提供され次第、手順どおり実機を触り OK/見たままを返す。

**しないこと**: 自力でビルド・起動を試みること。技術的にはできそうだが、文書は「provisioning 未完了として止めて依頼する」を正規手順と定めており、**勝手に構築した環境で golden を通すと検収の前提(プロジェクトが供給した確認状態)から外れる懸念がある** — 文書からは「してよい」と読めないので「保留+依頼」を選ぶ。

## 2. 役割の再構成

仕事は 2 つ — 裁定(4 値: 承認/条件付き承認/差し戻し/判断範囲外)と golden。差し戻しは正規の裁定結果で様式 3 点。数字なしの「できました」は受け入れない。C# 技術者として、CLI 等の非知覚案件では charter 追記の裁定を経て**技術確認者**を担う可能性が高い。

## 3. 質問リスト(要旨)

**A 級**: A-1 charter 参加供給欄が空欄の場合、**記入を依頼する相手をどこで知るのか**(環境準備責任者欄自体が空欄だと依頼先が循環して不明)/ A-2 リポ URL 等の受領時期(S1/S2 解消条件)/ A-3 既存の裁定者・golden 承認者は誰か(割当依頼先の実在)/ A-4 golden 実機は 3 択のどれで供給されるか / A-5 ECO-051 で私に回る承認は裁定者としてか技術確認者としてか。
**B 級(主要)**: 開発者の私が製品コードを書く場面はあるのか/技術確認者の右列項目の標準検証法と**記録の実物例が無い**(§5 は裁定者向けのみ)/product-profile スキルの実体/「凍結」の操作的意味。
**C 級**: 隔離の強制方法・PLM 実体・ずる台帳・マルチファクトリ。

## 4. 総合判定: **条件付きで乗り切れる**

S1〜S4 の全てについて、文書は「正しい止まり方と依頼の仕方」を明示的に与えている — S1/S2/S4 は §0.5 に直撃で該当し(S1 は症状が文言レベルで一致)、S3 は §4 の凍結不可規則が単独で答えを出す。初日に**間違った行動(勝手に承認・勝手に環境構築で golden 通し)を取らずに済む**確度は高い。初日の成果は「全案件を規則どおり停止させ、供給依頼を 3 本出すこと」であり、これは文書が意図した正しい状態 — ただし供給者が特定できなければ(A-1/A-3)そこで完全に詰む。
