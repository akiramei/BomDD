# Loop 2.5 総括 — execution 受入(CHEAT-009 手法修正の効果測定)

Loop 2 の CHEAT-009(文字列照合は過剰結合かつ被覆不足)への手法修正=**受入を文字列照合から execution(ffmpeg 実行+ffprobe で要求性質を照合)へ格上げ**し、効果を実測した。

## 方法
`build/ExportAcceptance`(console)= 製造ライブラリ `MoviePad.ExportSlice` が生成したコマンドで**実際に ffmpeg を実行**し、出力を ffprobe して要求由来の性質(寸法/vcodec/acodec/尺)を照合。
ソースは lavfi で生成(1920x1080@30 H.264/AAC 6s)。実行環境フラグ `-nostdin -y` のみ前置(変換ロジックは製造物のまま)。

## 結果
| シナリオ | execution 受入 | Loop2 文字列オラクルでの扱い |
|---|---|---|
| S1 恒等コピー | PASS (1920x1080/h264/aac/6.00s) | B13-15 PASS だが入力様式 B10-12 は誤検出FAIL |
| S2 スケール1280x720 | PASS (1280x720/h264/aac) | B02 PASS / scaleフラグ差は沈黙 |
| S3 速度2x | PASS (尺3.00s=半分) | B03 は `fps=` 有無のみ。尺の正しさは検査不能 |
| S4 トリム+シーク[3,6] | PASS (尺3.00s) | B08で音声時刻バグ検出→修正済 |
| S5 Lossless(要求FFV1/PCM) | **FAIL** (h264/flac ≠ ffv1/pcm) | **沈黙通過**(コーデックを未assert) |
| 合計 | PASS 4 / FAIL 1 | — |

## CHEAT-009 修正の効果(実証)
- **過剰結合の誤検出が消滅**: 出力メディアで検査するため、Loop2 で誤検出だった様式差(連結ラベル `cv`/`v0`、引用 `-i in.mp4`)は execution では問題にならず S1/S2/S4 が PASS。
- **被覆不足が解消**: S5 で Lossless の別コーデック(製造物 `libx264`/`flac` vs 要求 `ffv1`/`pcm_s16le`)を execution が捕捉。文字列照合が見逃した沈黙乖離を検出。
- **新規の機能検査**: S3 の尺=3.00s は「速度2xが実際に出力尺を半減させる」確認。文字列照合では不能。`setpts` の綴り差(製造物≠原版)が**機能的に等価**だと execution が裏付けた。

## 新たな発見 — execution 受入には「深さの段階」がある
| 深さ | 捕える乖離 | コスト | 本ループで該当 |
|---|---|---|---|
| L0 文字列照合 | 様式の一致のみ(過剰結合) | 極小 | Loop2 |
| L1 存在/終了コード | コマンドが ffmpeg で通るか | 小 | — |
| L2 メタデータ(ffprobe) | コーデック/寸法/尺 | 中 | **Loop2.5**(S5 を捕捉) |
| L3 内容/信号(PSNR/音声相関) | フレーム/音の位置・品質 | 大 | **B08級はここが要る**(S4の尺では出ない) |

→ S4 が尺で PASS したのは、B08(音声位置)が L3 でしか捕まらないため。**「execution 受入」は一枚岩でなく梯子**。
M-BOM の受入モダリティは、AC ごとに**必要な深さ**(L0〜L3)を指定すべき。深いほど忠実だが高コスト。

## 構想へのフィードバック(受入モダリティの型)
- 三層 BOM の M-BOM 受入基準に **モダリティ列**を `string < exists < metadata < content` の階層で持たせる。
- 外部ツール連携部品は既定 L2 以上、知覚/同期が要る部品は L3。純粋ロジックは L0(unit)で足りる。
- これにより「テスト緑 ≠ 仕様適合」(Loop1 CHEAT-005, Loop2 CHEAT-009)を、領域ごとに必要十分な深さで塞げる。
