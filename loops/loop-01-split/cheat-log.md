# ずる台帳 — Loop 1 (区間分割スライス)

BOM由来でない産物・手法ギャップの記録。判定基準と分類は [../../method/cheat-taxonomy.md](../../method/cheat-taxonomy.md)。

---

### CHEAT-001  [C3 工程欠落] リバース工程が手法に定義されていない
- 発生段: ①リバース
- 手法が与えなかったもの: 「既存ソースから要求/機能仕様をどう復元し、**復元の完全性をどう判定するか**」の工程と受入基準。
  BomDD は E/M/S-BOM を定義するが、その入口である**リバース工程が空白**。
- 代替した従来技術: 私の判断で全ドメイン公開操作・全テスト・全キーバインドを横断読みし、REQ へ手で写像した。
  完全性は「主要な公開APIとテストを網羅した」という主観的確信のみで、**抜けを証明できない**。
- 重大度: friction(今回は小スライスで網羅可能だったが、全アプリ規模では blocker 化する)
- 想定される手法的修正: リバース工程を定義し受入基準を「カバレッジ証明」にする
  — 全public ドメイン操作・全テスト・全入力イベントが ≥1 REQ に写像済み、逆に全 REQ が ≥1 出典を持つ(双方向トレース)。

### CHEAT-002  [C5 粒度崩壊] 1スライスのE-BOMが独立サブツリーにならない(woven)
- 発生段: ②E-BOM
- 手法が与えなかったもの: 横断的に共有される部品(EB-EDIT-DOC/TIMEQ/HISTORY/SELECT/KEYMAP/TIMELINE)を
  「スライスのBOM」にどう含めるかの表現。木構造の親子では表せない。
- 代替した従来技術: E-BOM表に「共有/横断」列を即席で足し、構成関係を木でなく有向グラフ(uses/trigger/observe)で描いた。
- 重大度: blocker(構想の「BOMツリー」前提そのものに関わる。これは初回未解決論点 C5 の実地確認)
- 想定される手法的修正: E-BOM を「製品全体の部品グラフ」とし、スライスは**そのサブグラフ抽出+共有部品への参照**として定義。
  共有部品には所有機能(owner)と利用機能(consumers)を持たせ、変更影響を S-BOM 的に追えるようにする。

### CHEAT-003  [C1 表現ギャップ] 存在理由が「前方仕様」でない部品を記録できない
- 発生段: ②E-BOM
- 手法が与えなかったもの: EB-SELECT(`ResolveSelection`)はバグ修正で生まれた部品。存在理由が要求でなく
  「観測された欠陥クラス(分割解除/Undo で選択が死ぬ)」。E-BOM の「存在理由=どの仕様による」欄に収まらない。
- 代替した従来技術: 存在理由欄に REQ-SPLIT-3 を当てたが、実体は欠陥起因。原版テストのコメント(回帰防止)から意図を補完した。
- 重大度: friction(が、工業化の根幹に触れる: 実ソフトの部品の相当数は「学習された制約」由来)
- 想定される手法的修正: 存在理由の型を拡張 — `requirement` だけでなく `defect-class`(回帰防止)/`constraint`(技術制約)を許す。
  defect由来部品には原因(再現条件)を必須化し、保守BOM(S-BOM)の劣化イベントと同じ語彙で結ぶ。

### CHEAT-004  [C6 手戻り / C1] M-BOM の受入基準(AC-12)が自己矛盾し、製造物が原版と発散
- 発生段: ⑥合否(根は③M-BOM)
- 手法が与えなかったもの: AC-12 が「既存最大ID+1**から**」と「単調増加」を併記し、**両立しない領域(Undo後の再Split)で一意でない**。
  製造装置は前者(再計算)を選び、Undo→再Split で同IDを再利用。隠しオラクル `Controller_NewIds_MonotonicAcrossUndo` が捕捉(13/14)。
- 代替した従来技術: 原版を参照し `SplitController` の nextId を増分カウンタ化(Undo/Redoで巻き戻さない)。3行修正で全green。
- 重大度: friction(が、BOM自身の論理矛盾という質の悪いギャップ。製造装置も Gap#14 で自己申告していた)
- 想定される手法的修正: M-BOM に「受入基準の無矛盾性検査」工程を追加。識別子(identity)払い出しポリシーを
  E-BOM/M-BOM の標準項目化(永続単調カウンタ / 再計算 / UUID のいずれかを明示)。矛盾するACはCIで弾く。

### CHEAT-005  [C4 受入不能] 数値丸め方式が未指定で、テスト全greenでも実挙動がサイレント乖離
- 発生段: ⑤製造 / ⑥合否
- 手法が与えなかったもの: `SnapFrame` の中間値丸め方式。製造物=AwayFromZero、原版=ToEven(C#既定)。
  移植テストのベクタが中間値を踏まず、**オラクルは乖離を検出できない**。
- 代替した従来技術: 製造報告 Gap#1 で人手検知(コード修正は今回見送り=既知の残課題)。テスト側も中間値ベクタ未整備。
- 重大度: blocker(工業化の中核。「テスト緑 ≠ 仕様適合」を地で示す。AI非決定性論点と同根)
- 想定される手法的修正: 数値ACに丸め方式・誤差許容・**中間値/境界のテストベクタ**を必須化。
  受入オラクルの「完全性」を、未指定次元ゼロ(全数値判断に丸め規定あり)で定義する。

### CHEAT-006  [C4 受入不能] UI/知覚部品(EB-KEYMAP/EB-TIMELINE)は客観検査不能で未製造・未検査
- 発生段: ④工程 / ⑥合否
- 手法が与えなかったもの: 「S キーで境界線が現れる」「後半が選択される(見た目)」を判定する非単体の受入手段。
  併せて、原版の MVVM バインド(コマンド活性・通知)を素の POCO 統合に置換(U2/U3)した差も検査外。
- 代替した従来技術: スライスから UI を切り出し、ロジックのみ POCO 化して単体検査。UI は記述のみで先送り。
- 重大度: friction(ただし対象アプリの相当部分が UI/知覚であり、累積すると blocker)
- 想定される手法的修正: 知覚的受入のモダリティを定義(ゴールデン画像 / 操作再生 / プロパティ通知トレース)。
  M-BOM の受入基準に「検査モダリティ」列を設け、`unit`/`golden`/`interaction`/`manual` を明示。manual は不足として可視化。
