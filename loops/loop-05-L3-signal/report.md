# Loop 5 総括 — execution L3(音声信号比較)で座標系バグを捕捉

外部アイディア#4(表面の公差/信号比較)を実装。Loop4 の発見「決定性≠正しさ」を閉じ、受入の梯子の **L3** 段を実証する。

## 方法
`build/L3Acceptance`(console): 1秒ごとに周波数が変わるトーン音源(秒s = 300+100*s Hz)+ testsrc 映像を生成。
SC4(trim[5,10], seekBase=5)の3変種を ffmpeg 実行 → ffprobe で尺(L2) → 音声を s16le 復号しゼロ交差から毎秒の支配周波数(L3)を推定。
SC4 の正解音は source[5:10] = 800,900,1000,1100,1200 Hz。

## 結果
| 変種 | L2 尺 | L3 毎秒周波数 | 判定 |
|---|---|---|---|
| correct(K-BOM: 2入力・絶対時刻) | 5.00s | 800,900,1000,1100,1200 | 内容OK |
| **buggy**(CHEAT-008: 未シーク入力に相対時刻) | **5.00s** | **300,400,500,600,700** | **★誤位置(source[0:5])= L3が捕捉** |
| phaseA(Loop4 共有: 単一シーク入力・相対) | 5.00s | 800,900,1000,1100,1200 | 内容OK |

## 発見
- **L2 では区別不能、L3 だけが捕捉**: correct と buggy は尺(5.00s)もコーデックも同一。音声内容(L3)のみが buggy(音声が source[0:5])を検出。
  → 受入の梯子で **L3 は音声同期/座標系バグに必須**(L0文字列・L2メタデータでは原理的に届かない)。アイディア#4(信号比較)実証。
- **Loop4 の主張を精密化**: Loop4 Phase A の「共有された方式」(phaseA)は実は**内容が正しかった**(source[5:10])。AAC シークのドリフトリスクは負うが致命的な誤りではない。
  真に危険な座標系バグ(buggy = CHEAT-008 の形)は**別物**で、それを L3 が捕える。→ 「共有暗黙知の罠」は2段階: (i)内容は正だが品質リスク(phaseA) (ii)内容が誤り(buggy)。L3 は (ii) を確実に捕捉。
- **梯子にはコストがある**: L3 は専用テスト信号(トーン音源)+復号+解析が要る。L2 より重い。FMEA で「音声同期」を要注意と判定した部品にだけ L3 を割り当てる(Control Plan の役割)。

## 二軸モデルの完成(BOM品質の計測体系)
| 軸 | 測るもの | オラクル | 実証ループ |
|---|---|---|---|
| 決定性 | 工場間ばらつき | multi-factory variance | Loop4 |
| 正しさ L0 | 様式一致 | 文字列照合(過剰結合のため不採用) | Loop2 |
| 正しさ L2 | コーデック/寸法/尺 | ffprobe | Loop2.5 |
| 正しさ L3 | 内容/信号(位置・同期) | 復号+信号比較 | **Loop5** |
| 正しさ golden | 知覚(見た目) | golden画像+人間承認 | Loop3 |

→ Control Plan([../../method/control-plan.md])の「音声同期(seek時) L3」行が実装・実証された。
ソフトウェア工業化の受入は、**核=unit / 表面=領域に応じた深さ(L2/L3/golden)+許容差+承認者**、で体系化できる。
