## 2. 中心的発見 — 補償機構はベンダーを越えた。ただし「工程の重心」がハーネスに引かれる

1. **P3(正直申告文化)は横断した**: 前借り不適合の自己申告(旧既定の明示)・自己受入失敗 2 件の記録・「仕様変更であり欠陥でない」の帰属診断・not-computed の正直記載。**文化は横断・様式は横断せず**(CHEAT-ID・G4 節・git 凍結は非採用)— ずる申告が Claude の訓練特性ではなく、**kit の規律として通じる**ことの初証拠。
2. **P2(環境抽象)は予測と違う形で決着**: Codex にサブエージェント機構がないことは tooling_gap にならず、**§11 テーラリングの縮退裁定(S規模・1工場・G2 1体)として吸収**された — 権限分離規則が「環境制約の受け皿」としても機能した。代償として検出された新穴 3 件: §11 規模判定の担当者間ばらつき(同クラス題材で M vs S)/ 要求改訂時の oracle 行 supersede 手順の不在 / **凍結(commit/tag)規律の不履行**(ハーネス起因の可能性 — Codex の作業流儀に git が組み込まれていない)。
3. **第3のエリシテーション型 = 決定の外部化**: 年度の区切りを聞かず・埋めず、「fiscalYear は API 入力・開始月に非依存」と**決定自体をシステム外へ出した**。有効だが、運用判断が呼び出し側へ転嫁されたことに顧客は気づきにくい — 質問(fable)・ゲート誘導(sonnet)・外部化(GPT)の 3 様は、いずれも申告なき充填 0 を保ったまま曖昧さを処理する別経路。
4. **指摘への自己改善**: oracle 行の扱いが ECO-001(in-place)→ ECO-002(新規行+個別命名)へ 1 ラウンドで収束 — 破れが恒常でなく学習可能。

## 4. 妥当性の限界

複合軸(ベンダー+ハーネス — git 不履行・S縮退がモデル起因かハーネス起因か分離不能)/ 顧客の一押し(受入確認は t2 同様に顧客起点)/ 遅延シード(付与・前借りの要求シードが GPT のエリシテーション様式により初回で渡らず、納品後投入= 計画外の ECO-001 を誘発 — 結果的に「納品直後の変更要求への即応」という追加観測を得たが、一周の顧客接点数の比較には注意)/ 小型題材・単一ラウンド。
