# Loop 2 総括 — 書き出し(外部プロセスIO)スライス

純粋ロジックでない領域(ffmpeg 連携)で BOM 手法を当て、Split(純粋ロジック)との差を測った。

## 結果スコアカード
| 指標 | 値 |
|---|---|
| Unit A(ExportPlan.Analyze, plan規則) | **15/15 合格**(1回目) |
| Unit B(filter/args 文字列, ffmpeg文法) | 25/30 → 修正後 **26/30** |
| 従来手法の介入 | 1(B08 機能バグ修正) |
| 製造装置のギャップ報告 | Unit A導けない規則=8項目 / Unit B任意様式=12項目 |

## 仮説の検証
- **H4 ✅(ただし条件付き)**: Unit A は規則から再生産でき全green。**しかし** industrialize できたのは BOM が外部/業務知識(コーデック白名簿・VFR非対称・閾値)を運んだから。装置は「VFR注記が無ければ逆に実装した」と明言。→ 外部連携の「設計」=外部仕様の知識転記であって導出でない(CHEAT-007)。
- **H5 ✅✅(予想以上)**: Unit B の失敗5件は3層に分かれた:
  1. **過剰結合の誤検出4件**(ラベル名・引用様式)=製造物は正しいのにオラクルが任意様式に結合して失敗。
  2. **本物の機能バグ1件**(B08: 音声 atrim の時刻基準)=オラクルが正しく捕捉(CHEAT-008)。
  3. **沈黙した意味乖離**(Lossless ffv1→libx264、lanczos→bicubic)=オラクル未被覆で通過。
  → 文字列照合は**過剰結合かつ被覆不足**。外部ツール成果物の忠実な受入は **ffmpeg 実行+メディア比較**(execution)しかない(CHEAT-009)。
- **H6 ✅**: 25/30 が通ったのは、装置が標準 ffmpeg 慣用(trim/atrim/setpts/atempo/asetrate/concat/null/volume/scale/-c copy/-map)を**自前の ffmpeg 知識**で再現したから(M-BOM に無い)。Loop1.5 の「C# 慣習で埋めた」と**同型の C2 現象**。BOM 充足は再び共有暗黙知に支えられていた。

## Split(Loop1/1.5)との対比 — ずるの質が変わった
| 観点 | Split(純粋ロジック) | 書き出し(外部IO) |
|---|---|---|
| BOM 充足を支えるもの | C# 慣習(C2) | ffmpeg 慣用(C2) |
| ずるの主因 | 仕様の矛盾/未指定(AC-12, 丸め) | 外部知識の必要(白名簿)+ 知識の枯渇(seekBase) |
| 知識が尽きた時の結果 | 値の小乖離 | **機能バグ**(音ズレ) |
| 受入の可否 | unit テストで客観判定可 | **文字列照合では不可**(要 execution) |

## 最重要の発見
**外部ツール連携部品は、純粋ロジック部品と同じようには industrialize できない。** 理由は3つの構造:
1. **インターフェース=外部ツールのコマンド文法**。要求から導けず、BOM は外部仕様の知識を大量に運ぶ必要がある(=設計が知識転記化)。
2. **共有暗黙知が「正しさ」を肩代わり**するが、知識が尽きた一点(seekBase の座標系)で**様式差でなく機能バグ**が出る。
3. **受入モダリティが壊れる**。文字列照合は任意様式を誤検出し意味乖離を見逃す。忠実な合否には外部ツール実行+成果物比較が要る。

→ これは構想の分水嶺の一つ「コードを鋳造品とみなせるか」に、領域依存の答えを与える: **純粋ロジックは鋳造に近い。外部ツール連携は、検査(execution)を伴わない限り鋳造の保証が立たない。**
