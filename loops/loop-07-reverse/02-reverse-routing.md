# ② リバース工程(Routing)

原版から BOM を起こす手順。各工程は「未トレースを残さない」ことを受入条件とする。

| 工程 | 内容 | 産物 | 受入 |
|---|---|---|---|
| R10 | 原版証拠の棚卸し(コード/コメント/テスト) | source-inventory | 経路に関わる証拠を列挙 |
| R20 | 各証拠から**観測された振る舞い**を記述 | behavior 列 | 証拠IDと対応 |
| R30 | 振る舞いから **REQ** を導出 | REQ 列 | 各REQが≥1証拠を持つ |
| R40 | **核/表面**判定、E-BOM部品・**K-BOM依存**へ写像 | E-BOM/K-BOM 列 | 表面は外部知識へトレース |
| R50 | **M-BOM AC** + **Control Plan 深さ** + **FMEA 故障モード**を導出 | M-BOM/CP/FMEA 列 | 危険点はFMEA+深い受入へ到達 |
| R60 | **治具**(オラクル/harness)を紐付け | 治具 列 | 受入深さに対応する治具が在る |
| R70 | 双方向トレース行列を組む | trace-matrix | 前方+逆引き両方向 |
| R80 | **カバレッジ監査**: 未トレース原版要素を分類、マルチファクトリ・リバースの差分を弱点として記録 | coverage-audit | 4成功条件を判定 |

## マルチファクトリ・リバース手順(R30 の検証)
同じ証拠(E1-E8 のコード)を複数リーダーに渡し、各自に「seek時の映像/音声の時刻計算・入力構成・根拠・失敗リスク」を抽出させる。
抽出 REQ の一致箇所=自然に読める(強い)/差分箇所=自然に読めない(弱点)。弱点は R80 で `unknown` 候補として明示し、Control Plan/FMEA で補強する。
