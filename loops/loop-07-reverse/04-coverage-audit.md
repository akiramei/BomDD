# ④ カバレッジ監査 — Export seek/audio 経路

リバース工程の出口検査。「全部読めた」でなく「**未トレースを分類でき、危険点が受入に到達したか**」で判定する。

## 未トレース原版要素の分類
| 要素 | 分類 | 扱い |
|---|---|---|
| `-ss` の fast/slow seek 精度(版・コーデック依存) | **unknown** | S-BOM の KB-FFMPEG-SEEK 依存として登録、版更新時に再検査 |
| `s < seekBase` 時の負 trim ガード(3リーダー全員が指摘) | **unknown** | 防御の有無を要確認。負値で空セグメント化のFMEA候補 |
| `plan.SpanStart` と `s` の時間軸一致 | **unknown** | 命名から推測。減算成立の前提として要確認 |
| `F(...)` 数値整形・`AudioTempo` 連鎖 | **out-of-scope** | 別経路(速度/書式)で別途トレース済 |
| エンコーダ引数 | **out-of-scope** | 形式別経路 |
| `-progress`/`-hide_banner`/`-nostats` | **ignored-with-reason** | 進捗・表示のみ。出力内容に不影響 |

## 成功条件の判定
1. **前方トレース可** ✅ 証拠→REQ→E-BOM/K-BOM→M-BOM/CP まで全行で辿れる(03 行列)。
2. **逆引き可** ✅ 各治具/AC/FMEA から証拠へ戻れる。
3. **未トレースを分類可** ✅ out-of-scope / unknown / ignored-with-reason に整理(上表)。unknown は S-BOM/FMEA へ escalate。
4. **危険点が FMEA+L3 に到達** ✅ 座標系(CHEAT-008)が FMEA「座標系取り違え」と CP「L3 音声同期」へ到達。マルチファクトリ・リバースも全員が失敗モード(AV同期ずれ)を抽出。

## マルチファクトリ・リバースの結論
- **強い(自然に読める)**: 振る舞い・座標系・入力構成・失敗モード=3リーダー全員一致。コードが構造的に自己説明的。
- **弱い(自然に読めない)**: **根拠の精度**。全員が「AV同期」と粗く読み、真の「AACサンプル精度・ビット完全一致」を取りこぼした。これは **受入の許容差**を決める情報で、過小読みは粗い受入を招く。
  → リバース工程の弱点は behavior でなく **rationale-precision**。対策: 根拠が受入の深さ/許容差を決める行(REQ-SEEK-4)は、コード読解だけに頼らず **K-BOM(外部知識)で補強**する。
- **unknown の収束**: 3リーダーが同じ unknown 群を挙げた=リバース工程は unknown を**安定して表面化**できる(これ自体が入口品質の証拠)。

## CHEAT-001 の決着
リバース工程は「無い」状態から、**双方向トレース行列 + カバレッジ監査**で入口品質を管理する工程になった。
残るリスクは「根拠の精度の過小読み」で、これは K-BOM 補強 と S-BOM の unknown 登録で運用的に管理する。
