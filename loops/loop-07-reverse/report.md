# Loop 7 総括 — リバース工程の形式化(CHEAT-001 決着)

三層BOMの一周後に残った CHEAT-001=入口保証。Export seek/audio 経路で、リバース工程を
「完全読解」でなく「**双方向トレース + カバレッジ監査**」として形式化し、リバース装置もマルチファクトリ化した。

## 成功条件 4つすべて充足
1. 前方トレース(証拠→REQ→E-BOM/K-BOM→M-BOM/Control Plan)✅
2. 逆引き(治具/AC/FMEA→証拠)✅
3. 未トレースを `out-of-scope / unknown / ignored-with-reason` に分類 ✅
4. 危険点(CHEAT-008=座標系)が FMEA と L3 受入へ到達 ✅

## マルチファクトリ・リバースの発見(Loop4 製造ばらつきのリバース版)
同じ原版コードを3リーダー(opus/sonnet/haiku)に読ませ、抽出REQの差分を測った:
- **振る舞いは全員一致**(映像=相対 `s-seekBase` / 音声=絶対 `s,e`・未シーク入力1 / 失敗モード=AV同期ずれ)。
  → 座標系は**コードから自然に読めた**(構造が自己説明的)。私の事前予想(座標系で割れる)は外れ、より深い発見に至った。
- **根拠の精度で乖離**: 全員が根拠を「AV同期」と粗く読み、原版の真の根拠「**AACサンプル精度・ビット完全一致**」を取りこぼした(伏せた E7)。
  → **リバースの弱点は behavior でなく rationale-precision**。根拠の精度が**受入の許容差**を決めるため、過小読みは粗い受入(粗いL3)を招く。**Loop6 の治具劣化と同根**で接続。
- **unknown は収束**: 3リーダーが同じ未確定点(s<seekBase の負trim ガード等)を挙げた=リバース工程は unknown を安定して表面化できる。

## CHEAT-001 の決着
リバース工程は「無い(完全性証明不能)」から、**双方向トレース行列+カバレッジ監査で入口品質を管理する工程**へ。
残リスク「根拠の精度の過小読み」は、(a)受入の深さ/許容差を決める根拠行は K-BOM(外部知識)で補強、(b)unknown は S-BOM へ登録し版更新で再検査、で運用管理する。
