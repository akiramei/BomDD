# BOM-Driven Development
## AI をソフトウェア製造装置とみなす工業化手法の実証研究

**要旨** — ソフトウェア開発を「人間が都度コードを書く作業」ではなく、設計BOMから AI 製造装置が工程に従って部品を作り、検査し、保守BOMで交換する**工業プロセス**として扱えるか。既存アプリ MoviePad(Avalonia/.NET/C# + LibVLC + ffmpeg の動画編集デスクトップ)を題材に、7ループのアクションリサーチで検証した。中核結論は次の一文に集約される。

> **作業仮説として、ソフトウェア部品は「ロジックの核」と「外部表面」に分けて管理できる。本研究で観測した範囲では、核は BOM から鋳造でき(下記「鋳造」の定義)工業化が成立した。表面(外部ツールのコマンド・画面の見た目)は外部/設計知識への依存ゆえ、知識の管理(K-BOM)・領域に応じた検査(execution/golden の梯子+許容差+承認者)・保守追跡(S-BOM)を伴って初めて鋳造保証が立った。観測した12件の「ずる」はすべて表面側に現れた。BOM 品質は「決定性(工場間ばらつき)× 正しさ(受入)」の二軸で測れる。**

> **用語「鋳造できる(castable)」** — 本研究では「原版非開示の製造装置が BOM/工程のみから成果物を生成し、定義済み受入(unit/L2/L3/golden)を**手修正なしで通過**すること」を指す。比喩でなく、この操作的条件で測る。
> **用語「multi-factory」** — 完全独立ベンダではなく、**同一モデル系列内の複数装置/設定**(opus/sonnet/haiku)を指す(限界 §9)。
> **用語「S-BOM」** — 一般的な SBOM(Software Bill of Materials)ではなく、**Service BOM / 保守部品表**を指す。OSS 依存一覧はその一部に過ぎず、外部知識依存の劣化を逆引きして再検査・交換/再製造判断を導く保守層(§8)。

---

## 1. 問い
ソフトウェアは製造業のように工業化できるか。すなわち、CAD相当の設計モデルから、設計部品表(E-BOM)→製造部品表(M-BOM)→工程→検査→保守部品表(S-BOM)を流す、再現可能な工程として開発を組めるか。

## 2. 仮説
- **AI = 製造装置**(NC工作機械の類比)。重要なのはプロンプトを書くことではなく、装置への投入情報を NCデータ/工程指示として管理すること。
- **BOM = 製造指示**。設計の意図と部品構成、製造単位、受入基準を構造化したもの。
- 失敗は部品表の不足だけでなく、装置が使う**暗黙知・治具・検査・許容差が未管理**であることに起因する(研究途中で得た強化仮説)。

## 3. 方法
MoviePad を BOM 手法で再実装する**アクションリサーチ**。各ループで一機能スライスを ①リバース→②E-BOM→③M-BOM→④工程→⑤AI製造(隔離装置)→⑥合否 に通す。
**真の成果物は動くアプリではなく、「BOM手法がどこで破れ(=ずる)、どう埋めるか」の記録と手法改善である。** ずるは失敗でなく観測データ。
測定器具: 製造装置の隔離(原版非開示の別エージェント)、ずる報告の義務化、マルチファクトリ製造(複数モデル)、execution/golden 受入ハーネス。

## 4. 実験結果(Loop 1〜7)
| Loop | 領域/題材 | 検証 | 主結果 |
|---|---|---|---|
| 1 | 純粋ロジック(区間分割) | 全工程を一巡 | BOMのみでビルド+14/14(従来手法1点介入) |
| 1.5 | 改善効果測定 | 完全性/無矛盾/グラフ化 | 新装置で1回目15/15・介入0・発散2→0 |
| 2 | 外部ツールIO(書き出し) | C2/工程欠落の増加 | 規則部=15/15、ffmpeg文字列部で文字列照合が破綻 |
| 2.5 | execution受入(L2) | 受入モダリティ修正 | 様式差の誤検出消滅+別コーデック沈黙乖離を捕捉 |
| 3 | UI/知覚(タイムライン描画) | 知覚的受入 | 幾何=画素完全一致 / 未規定の色=32.5%乖離・局在 |
| 4 | マルチファクトリ×K-BOM | ばらつきで品質測定 | K-BOMなし=全次元分岐 / あり=3工場完全一致(ばらつき0) |
| 5 | execution L3(信号比較) | 座標系バグ捕捉 | L2同一(尺5.00s)でL3だけが音声位置バグを捕捉 |
| 6 | S-BOM/PLM(保守層) | 表面知識の劣化 | 影響を6部品中1へ絞込・核は再検査不要・L3が必要と導出 |
| 7 | リバース工程(入口) | CHEAT-001 | 双方向トレース+監査で形式化。弱点=根拠の精度 |

合計12件の「ずる」を記録し、**すべてについて 解消・検出可能化・管理対象化 のいずれかの手法的扱いを定めた**(全件が「消えた」のではない。例: CHEAT-003/008 は除去でなく、記録場所や検査深さで**管理可能化**したもの)。分類: C1表現ギャップ/C2暗黙知/C3工程欠落/C4受入不能/C5粒度崩壊/C6手戻り。

## 5. 中核発見 — 核と表面の法則
3つの異なる領域(純粋ロジック / 外部ツール / UI)が**同じ構造**に収束した。

| | 核 | 表面 |
|---|---|---|
| 例 | ドメイン代数・判定規則・幾何 | ffmpegコマンド文法・画面の画素 |
| 正しさの出所 | 要求から導出可 | 外部仕様/設計トークン(知識) |
| BOMから鋳造? | ✅ unitで検査 | △ 知識転記+検査工程が要る |
| 受入 | unit | execution/golden+人間判断 |

- **観測した12件のずるはすべて表面で出た**。核は鋳造に近い(単一題材の観測。一般法則としては §9 で未検証扱い)。
- **C2(共有暗黙知)が成功を偽装する**: BOM が十分に見えるのは、製造者(AI)と設計者が暗黙知(C#慣習・ffmpeg慣用・配色直感)を共有するから。共有知識が尽きた一点で、ずるは様式差でなく**機能バグ**になる(音声座標系=CHEAT-008)。
- 製造業の比喩と一致: 寸法・機構(核)は図面から作れるが、表面処理・嵌合・外観(表面)は加工知識と検査工程を要する。

### 5.1 仕様の正しさは「根拠の精度」に依存する(Loop7 からの昇格命題)
> **仕様の正しさは振る舞いだけでなく、根拠の精度に依存する。根拠の精度が検査深さと許容差を決める。**

原版を複数リーダーに読ませると、振る舞い・座標系は全員一致(コードが自己説明的)だが、**根拠の精度**で乖離した — 全員が根拠を「AV同期」と粗く読み、真の「AACサンプル精度・ビット完全一致」を取りこぼした。根拠を粗く読むと、それに紐づく受入の許容差も粗く設定され、サブmsドリフトを見逃す。よって BOM の入口品質は「振る舞いを写したか」でなく「**受入の深さ・許容差を決める根拠を正しい精度で写したか**」で測る。これは BomDD 独自の主張で、仕様化とは何かに触れる。

## 6. 方法論 — BomDD Method v1
実証済みの規則を束ねた薄い正規版([method/bomdd-method-v1.md])。要点:
- **E-BOM**: 部品を核/表面で型分け。存在理由型(requirement/defect-class/constraint/external-knowledge/design-token)。木でなくグラフ(woven部品に owner/consumers)。**K-BOM**(知識部品表)で外部/設計知識を管理。
- **M-BOM**: 受入基準に〔完全性(全数値/全フィールド)・無矛盾性・Control Plan列〕を必須化。識別子・**座標系**の不変条件を明示。**FMEA先行**。
- **工程**: 製造装置の隔離+ずる報告の義務化、マルチファクトリ、治具を成果物管理。
- **受入**: 梯子 `L0文字列 < L1存在 < L2メタデータ < L3 content/signal < golden+承認者`。核=unit、表面=領域別深さ。L0/pixel-exactは過剰結合のため既定不採用(例外: 正規化シリアライズ/deterministic raster が製品特性の場合)。
- **リバース工程(入口)**: 双方向トレース行列+カバレッジ監査。未トレースを out-of-scope/unknown/ignored-with-reason に分類。

## 7. 品質測定 — 二軸
BOM 品質は2軸で独立に測る。
1. **決定性 = 工場間ばらつき**(multi-factory variance)。同一BOMを複数装置に渡し出力の分散を見る。low ほど BOM が出力を一意化(Loop4: K-BOMで全次元分岐→ばらつき0)。なお本研究の multi-factory は完全独立ベンダでなく**同一モデル系列内の複数装置/設定**(opus/sonnet/haiku)であり、観測したばらつきには「別装置の差」と「同系統のプロンプト差」が混在する(§9)。
2. **正しさ = 受入**(L2/L3/golden)。
**一致 ≠ 正しさ**: 複数工場が揃って同じ誤りを出す「共有暗黙知の罠」がある(Loop4 で観測、Loop5 の L3 で座標系バグを捕捉)。ゆえに両軸を独立に検査する。

## 8. 三層BOM
| 層 | 役割 | 実証 |
|---|---|---|
| **E-BOM** | 設計(核/表面/K-BOM/グラフ) | Loop1-3 |
| **M-BOM + 工程** | 製造(Control Plan/FMEA/隔離/マルチファクトリ) | Loop1.5-5 |
| **S-BOM / PLM** | 保守(外部知識依存の劣化→影響→再検査→交換判断) | Loop6 |

- **表面は核を触らず劣化する**(ffmpeg版・OS・デザイントークン・AIモデルの変化)。S-BOM は依存一覧でなく「外部知識依存×受入深さ」を持ち、劣化イベントを逆引きする。
- **PLM の価値は被覆でなく絞り込み**: Loop6 で再検査を6部品中1部品・1深さ・1シナリオへ絞り、核を一切巻き込まなかった。
- **治具も劣化対象**: 知識更新は再検査の深さ・許容差も上げ得る。
- **位置づけ(概念実証)**: Loop6 は**実 ffmpeg バージョン更新でなく、K-BOM 知識更新イベントの注入**による S-BOM/PLM の概念実証である。「S-BOM は再検査範囲を絞れる」は強く言えるが、「現実のPLM運用で外部仕様の劣化を自動検出できる」は次段(§10)。

## 9. 主張の整理(強い主張 / 未検証)
研究を方法論として外に出すため、言える範囲を明示する。

### 強く言える(本研究で実証)
- 核は BOM から再製造しやすい(3領域で unit 緑)。
- 表面は外部知識と検査工程を要する(ずる12件すべて表面)。
- **L2 では捕まらない正しさがあり、L3(信号比較)が必要**(尺同一で音声位置バグを捕捉)。
- **工場間の一致は正しさではない**(共有暗黙知の罠)。
- S-BOM は再検査範囲を絞れる(被覆でなく絞り込みが PLM の価値)。
- リバースには双方向トレースと**根拠精度の監査**が要る(弱点は behavior でなく rationale-precision)。
- 改善ループは効く(ずるが実測で減る)。ただし完全性は漸近。

### まだ言い切らない(未検証)
- すべてのソフトウェアで核/表面法則が成り立つか(題材は1つ)。
- 表面知識がどの程度のループ数で収束するか(定量化未了)。
- BOM化の最適粒度(表面をどこまで BOM 化すると「設計=実装の転記」に崩壊するか)。
- AIモデル差を長期 PLM でどう扱うか(モデル更新による再製造差分の保守)。
- マルチファクトリの「決定性」と最終品質の相関の一般性(1題材・3モデルの観測)。

### 限界
題材が単一アプリ(デスクトップGUI+外部プロセス)であること。製造装置・リーダーが同一モデル系列であること。execution L3 治具がコース粒度(1秒/100Hz)で、サブms同期は未検証であること。

## 10. 次の研究計画(BomDD v2 research program)
- **別題材での再現性検証**(最優先): Web アプリ / 分散システム / 組込・リアルタイム のいずれかで「核/表面の法則」が再現するか。表面の種類(HTTP/RPC契約・タイミング・ハードウェアI/O)が変わっても二層構造が出るか。→ **Web/API で中間結果を得た(§11)**: 単一ファクトリで漸近収束(2→3→0)、多工場で仕様面が転移・未規定面が分散。
- **収束の定量化**: 表面知識が尽きるループ数を題材サイズで測る。
- **BOM=成果物 崩壊境界の特定**: 表面の最適 BOM 化粒度。外部ツール/デザインシステムを部品として外部化する設計の検証。
- **長期 PLM**: AIモデル更新を S-BOM の劣化イベントとして扱い、再製造差分の管理手順を確立。

## 11. 外部妥当性 — 別ドメインでの再現(v2 中間結果, N=3)
§10 の最優先課題に着手し、別題材(会議室予約 API・.NET/HTTP、別リポジトリ **BomDD-WebApi-Sample**)で核/表面の法則を再検証した。核=予約可否/重複/冪等/キャンセル、表面=HTTP status・OpenAPI 契約・API key 認証・UTC 日時・error schema。手法は本研究と同型(機械可読 BOM、原版非開示の隔離製造、設計者側の固定ブラックボックス HTTP オラクルで 2-way diff)。

- **収束(単一ファクトリ)**: seed 2/16 → 第1次補正 3/16 → 第2次補正 0/16。各補正は fresh device で一致=**修正は BOM に宿る**。完全性は漸近(締めると別の未規定次元が出る)。
- **マルチファクトリ(opus/sonnet/haiku)**: 固定オラクル層 opus 0 / sonnet 1 / haiku 3。差分は ①**BOM未規定残渣**(sonnet: not_found code 名)②**仕様化済み契約の取りこぼし=工場能力**(haiku: 409→400 等)③**探索的未規定面**(ID/日時/応答スキーマの分散)に分類できる。締めた仕様は能力ある工場へ転移し、未規定の表面は工場間で分散した。

**言える範囲**: 核/表面の法則は MoviePad 固有でなく Web/API でも再現した(題材2つ目。下記 §11 末の分散 Saga で3つ目=N=3。一般法則としては引き続き未検証)。BomDD の測定器は、工場間差分を「AI 差」で済ませず **BOM欠落 / 工場能力 / 未規定面** に切り分ける。**用語規律**: 全工場一致を「完全鋳造」と呼ばず「現固定オラクル被覆で未観測差分ゼロ」とする(被覆を広げれば別の未規定次元が出うる)。**Future Work**: 応答スキーマ・ID・日時まで締めると探索層の分散が消えるか(webapi-02.5)。

詳細・証拠は公開リポジトリ [akiramei/BomDD-WebApi-Sample](https://github.com/akiramei/BomDD-WebApi-Sample) の `bomdd/`(機械可読 BOM)・`loops/webapi-0*/`(各ループ報告・metrics・固定オラクル/探索プローブ・As-Built)。コミット列 `35c3022`(seed)→ `7b3a4f2`(webapi-01)→ `c46fe8b`(01.5)→ `4eed25f`(01.6, tag `webapi-02-input-bom`)→ `84338b0`(webapi-02)。

### 11.1 分散 Saga(N=3 — 非同期イベント駆動)
3題材目 **BomDD-DistributedSaga-Sample**(注文フルフィルメント Saga・in-memory synthetic event bus)でも再検証した。核=Saga 状態遷移/在庫・支払い・出荷の判定と補償規則、表面=event schema・at-least-once・冪等 consumer・retry/dead-letter・ordering。HTTP のようなワイヤ境界が無い in-process 題材のため、観測の黒箱境界を**「固定シナリオを流し正規化イベントログ/dispatch trace を emit する観測契約」**で作り、内部 C# 名でなくイベント契約 + Control Plan の語彙で 2-way diff した。

- **単一ファクトリ**: 練られた seed BOM から clean factory が初回で固定オラクル **0/7**(happy 完全イベント列・3 補償フロー・冪等・retry 回復・event 契約)。webapi-01(2/16)と異なり収束を要さず鋳造。
- **多工場(opus/sonnet/haiku)**: **挙動契約の差分 = 0 / 0 / 0**。仕様化済み saga 契約は **3 ティアすべてへ転移**(webapi-02 で 3/16 だった haiku も 0)。探索層(eventId 形式・timestamp 精度・sequence 基点・inbox 区切り・causationId・JSON 形状)は工場間で分散。
- **検査器側の知見(製品差分と分離)**: sonnet の見かけ上の差(リテラルキー採点 1/7)は**製品でなく観測ハーネスの表現差**(dispatch outcome キーを C# 慣習 camelCase で出力。挙動は完全一致)。加えて設計者オラクル自身が events.json の JSON 形状(map/array)に L0 過剰結合していた。いずれも「**オラクルは付随表現でなく契約セマンティクスを見る**」で表現非依存に補正——この規律を**検査器自身**に課すのが v1.2 の追補。**C2(共有暗黙知)は製品だけでなく測定ハーネスの表現にも現れる**。

証拠(公開): [akiramei/BomDD-DistributedSaga-Sample](https://github.com/akiramei/BomDD-DistributedSaga-Sample)。入力固定 tag `saga-02-input-bom`、結果固定 tag/release `v0.1-saga-multifactory`。コミット列 `cef17bf`(seed)→ `5facf25`(設計者オラクル, tag)→ `fbed7e7`(saga-01, 0/7)→ `ce21e52`(saga-02, 0/0/0)。再現手順は [docs/reproduce-saga-v2.md](docs/reproduce-saga-v2.md)。

**N=3 で言える範囲**: 核/表面の法則と品質二軸は、同期 GUI(MoviePad)・同期 HTTP(Web/API)・**非同期分散(Saga)**の3ドメインで再現した(観測した範囲。一般法則は引き続き未検証)。
