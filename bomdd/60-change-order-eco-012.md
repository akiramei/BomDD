### 是正方針案(製造前・凍結前の草案)

1. **厳格ローダーの共通化**: 重複キーで FAIL する SafeLoader 派生(構築時に key 集合を検査・
   重複でエラー位置つき例外)をヘルパ関数化し、C1/C3 の読込を差し替える。
2. **検査対象の拡張**:
   - C1 の走査に `bomdd/` の実台帳 YAML を追加(または C3 側で厳格ロード — 実装時に一本化)。
   - C4 の scaffold 煙試験で、生成された YAML 全件を厳格パース(生成物も保存正本)。
3. **(gate 採否)C3 拡張 — verified の状態契約検査**: `status: verified` のエントリに対し
   「verification が**非空**・**既知プレースホルダでない**(「(製造前」等の既知文言との
   部分一致に依存しない列挙管理)・**diff_audit.head あり**」を一体で検査し、いずれか欠けば FAIL。
   先行助言(2026-07-16 レビュー)= 本文 C(playbook 一般則)への昇格とは分離し、**verified の
   明示的な状態契約の検査= C3 のローカル不変条件としては採用推奨**(経験則の早期一般化には
   あたらない)。gate で正式採否。
4. **変異テスト(陽性対照・playbook §13「検査の新設は陽性対照と対で」)**:
   - 変異 A: 一時コピーへ意図的な重複キーを注入 → C1(厳格化後)が FAIL。

   `_DupKeyError`=行番号つき)+ `strict_yaml_load()` を新設し、C1/C3/C4(lock 読込)の

2. **検査対象の拡張**(案 2): C1 の走査に `bomdd/` の実台帳 YAML を追加(17→18 件)。
   C4 に生成 scaffold の全 YAML(*.yaml/*.yml)の厳格パースを追加(生成物も保存正本)。
3. **C3 = verified の状態契約検査**(案 3・採用): verified 全エントリに対し「diff_audit.head

   `_KNOWN_PLACEHOLDERS` の**列挙管理**(部分一致に依存しない)。

## 教訓(還元候補 — lesson-promote 経由)

- **保存正本の検査は「構文が通る」でなく「情報が保存される」を張る** — 寛容パーサの既定挙動
  (重複キー後勝ち・型強制等)は検査の暗黙前提になりやすい(ECO-079「暗黙前提 3 様態」の
  検査層版・OBS-20260716-01)。
