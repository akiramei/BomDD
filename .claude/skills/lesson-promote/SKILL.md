# /lesson-promote — 教訓の方法論還元

引数: 出典(製品リポ名+ECO/loop 番号群。例: `ViewPrism2 ECO-048〜053`)。

> 本スキル自体が「手順の成熟度ラダー」教訓(ViewPrism2 ECO-048〜053 還元・2026-07-06)の適用例:
> ①暗黙知 → ②散文手順 → **③入口コマンド化(本スキル=順序・停止点の強制)** → ④機械検査 → ⑤自動化。
> 高頻度・逸脱実績のある手順を②に滞留させない。還元は従来「前例コミットの様式を見て倣う」暗黙知だった。

## 絶対規律

- **出典なき還元は禁止**: 教訓は必ず出典(ECO クローズ節・loop 総括・実測)へトレース可能にする。
  転記するのは要約でなく事実(潜伏期間・マスキング・プローブ実測値)。
- **一般化検査**: 製品固有名詞を落としても命題が成立するかを検査する。成立しないものは還元しない
  (製品リポの CP 観点・教訓に留める — 過剰一般化は method を薄める)。
  例外: **不可逆観測データの保存義務**は rule of three を待たない(1 例で本文化可 — 判断基準は
  「遡って回復できるか」。実例: ui-ir-ui-bom.md §12.1 双子出力・2026-07-08 裁定。playbook §13 運用規則)。
- **記帳と織り込みの二段分離**: improvements.md への記帳(本スキルで完了)と、method 本文への
  織り込み(節ごとの diff 案 → **レビュー停止点**)を分ける。本文は方法論の正本 — 無承認改訂禁止。
- **効果測定の宿題**: 改善は「ずるが減ったか」で測る(improvements.md 冒頭の規律)。
  各教訓に次ループ/次 ECO での測定観点を 1 行残す。

## 手順

1. **出典収集**: 製品リポの ECO クローズ節(教訓段落)・register notes・51-cheat-log から、
   教訓と裏付け(潜伏実績・マスキング・read-across)を収集する。
2. **一般化検査**: 各教訓を「製品名を含まない命題」+「検出/防止の機械面(検査・hook・pin テスト・
   スキル)」の対で書けるか検査。対にならない教訓は「観測のみ(様式候補・rule of three 待ち)」と明示。
3. **行き先判定**(複数可):
   - `bomdd-playbook-v1.md` — 工程規律(ECO 規律は §8 系・受入設計は §4.4 等)
   - `silence-checklist.md` — 沈黙次元の掃討観点
   - `control-plan.md`・オラクル系テンプレ — 検査設計(深さの梯子・較正・陽性対照)
   - `cheat-taxonomy.md` — ずる/欠陥様式の分類
   - `FINDINGS.md` — 実証データ(定量・N・限界の明示)
   - `templates/` — 台帳雛形への織り込み
4. **期待効果の棚卸し(回収)**: improvements.md の既存節の「期待する効果」を走査し、今回の出典
   (ECO 種別・触れた面)が測定観点のトリガー条件に一致する **open 項目**(`→ 回収` 未記入)を
   列挙し、一致した各項目へ裁定を出す:
   - **回収**(効いた — 実測つき)/ **再演**(効かなかった — 反証データつき)→ 新節へ
     「効果回収」として記帳し、元の節の「期待する効果」該当行へ `**→ 回収(YYYY-MM-DD)**: <実測>` を
     追記(ECO-066 節の様式)。
   - **機会なし**(今回の出典は適用機会でない)→ 記帳不要。
   - 一致 0 件なら新節に「期待効果の棚卸し: 該当 0 件」と 1 行残す(**実施の証跡** — 未実施と
     区別する。棚卸しは任意でなく毎回実施)。

   **走査の起点は `method/tools/worklist.py` の出力**(worklist / warnings / legacy coverage)。
   スクリプトが厳密解析するのは「追跡項目の記帳スキーマ」(下記)の新書式のみ — **legacy
   (未構造化)節は従来どおり目視走査**する(旧書式を推測して open 扱いしない。legacy 節の
   一覧と件数はスクリプトが表示する — どこまで機械的に保証されているかを常に明示)。
   > 由来: 回収は従来「還元セッションが偶然気づいたときだけ」の任意行為(②滞留)だった —
   > 2026-07-13 裁定で③(本ステップ=毎回強制)へ。④(validator 強制)への昇格条件=回収漏れの
   > 実害 1 例の実測(improvements.md 2026-07-13 節に事前登録)。
5. **improvements.md へ記帳**: `## YYYY-MM <product> 還元 — <title>` 節を既存書式
   (観測=出典つき / 適用した改善=反映先つき / 期待する効果)で追記。手順 4 の回収・再演は
   同節の「効果回収」項として含める。**期待する効果と rule of three 蓄積中の観測は、
   「追跡項目の記帳スキーマ」(下記)の箇条書きで書く**(散文の段落へ流し込まない)。
6. **本文織り込み案の提示** — **停止点: レビュー**。行き先ごとの diff 案(節番号+追補文)を提示し、
   承認された節から織り込みコミットする(節単位で採否可)。
7. **コミット**: `method(<product>): <出典範囲> 還元 — <要約>`。improvements.md 記帳と
   tools/skills の追加は本スキル内で実施してよい(本文織り込みは停止点の後)。

## 追跡項目の記帳スキーマ(v1 — 2026-07-15 レビュー採択)

手順 4/5 が追跡する 2 種(期待する効果= `EXP` / rule of three 蓄積中の観測= `OBS`)は、
散文に埋めず **1 項目 1 箇条書き+状態マーカー+安定 ID** で記帳する:

```markdown
- [open] EXP-20260715-01 — <測定観点とトリガー条件>
  evidence: <出典節・項>
- [watch 2/3] OBS-20260715-02 — <命題>
  source: ECO-086, ECO-087
  evidence: <出典節・項>
- [recovered 2026-07-18 via ECO-092] EXP-20260715-01 — <内容>
```

- **状態語彙(5)**: `open`(実施/判断待ち)/ `watch N/3`(rule of three 蓄積中 — 現在値 N 必須)/
  `recovered`(効果回収済み)/ `withdrawn`(実施せず終了 — 仮説否定・不要化)/
  `superseded`(別項目・別案へ吸収)。`recovered`・`withdrawn`・`superseded` は
  **日付+証拠座標(`via <ECO/loop/commit/レビュー>`)が必須** — 日付だけでは回収判定の
  根拠を再確認できない。open/recovered の 2 値にしないのは、「不要になった」「別案に吸収」
  「仮説が否定された」を回収済みと混同しないため。
  **watch が 3/3 に達したら worklist が actions due(PROMOTION DUE)に出す** — これは
  記帳不整合の警告(W1〜W6)ではなく、正しく蓄積された結果のレビュー要求(別区分。混ぜると
  正常運用でも警告が常時残り、真の異常が埋もれる)。レビューでは 3 例の独立性(同一事象の
  再記述でないか・同じ抽象規則で説明できるか)も含めて判断し、昇格なら
  `recovered ... via <織り込み根拠>`、件数に疑義なら N を戻す、見送りなら
  `withdrawn`/`superseded` へ遷移させる(「3 件たまった」と「昇格判断が済んだ」を区別する)。
- **ID**: `EXP|OBS-<記帳座標>-<連番 2 桁>`(例: `EXP-20260715-01`)。記帳座標=その項目を
  記帳した節の日付(YYYYMMDD。日を持たない旧節は YYYYMM)。**ID は一意な参照座標であり、
  因果帰属を主張しない** — 複数 ECO 合同節でも発生元を ID に埋め込まず、由来は継続行で表す:
  `source:`= 発生元 ECO / `evidence:`= 証拠位置 / `via`= 回収根拠(責務分離)。
  末尾連番は**連続性を要求しない**(欠番は異常でない)。ID は一度割り当てたら**変更・再利用
  しない**。重複は worklist が警告する(採番前の grep を必須手順にしない — 一意性検査は機械の仕事)。
- **bullet の責務は 2 つだけ**: ライフサイクル状態+証拠へのポインタ。証拠散文の再記述は
  しない(証拠の正本=節の散文 / 処理状態の正本= bullet)。表示幅の都合で記帳を変えない
  (全文確認は `worklist.py --full` / `worklist.py show <ID>`)。
- **状態遷移は行内書き換え**: 行を複製せず、該当行のマーカーを書き換える(従来の `→ 回収`
  追記様式の後継。回収の実測詳細は従来どおり新節の「効果回収」項に書く)。
- **期首移行(一回限り・実施済み 2026-07-15)**: 現存していた open/watch 残高は「移行残高」節
  (節タイトルに「移行残高」を含む)へ短い参照として一度だけ移行済み。legacy 本文は書き換えず
  証拠正本として残す。棚卸しの監査境界は**明示マーカー**
  `<!-- worklist-legacy-audit-cutoff: YYYY-MM-DD -->` が正本(位置関係の暗黙判定に依存しない):
  **マーカー以前への legacy 形式の節追加・移動は禁止** — 新規節はマーカー以後に追記し
  新スキーマを使用する(境界以後に legacy 節が現れたら worklist が unaudited=逸脱として列挙)。
  移行節は照合記録(reviewed / last audited section / found / migrated / excluded /
  unclassified)を持つ — unclassified=0 が「判断不能候補を黙って落としていない」ことの証跡。
  移行残高と新規は worklist 上で区分表示される(既定: 移行節内= migrated / それ以外= native。
  継続行 `origin:` で上書き可)— 残高 43 件に新規 open が埋もれないため。移行残高の整理
  (recovered/withdrawn/superseded への遷移)は通常還元と混ぜず別の棚卸し工程で行う。
- **強制化(validator/CI)は運用実測後に判断**: 認知コスト(件数・節数・所要)と品質リスク
  (回収漏れ・3 例目見落とし・ID 重複・状態不整合)の二軸で、数回の運用で基準線を作ってから
  昇格条件を決める(先に恣意的な閾値を置かない)。当面 `worklist.py` は読み取り専用・
  終了コード 0(--selftest のみ失敗時に非 0)。

## 停止点

method 本文(playbook / checklist / taxonomy / FINDINGS / templates)の改訂はレビュー承認後に
節ごと適用。improvements.md 記帳までで一旦停止し、織り込み案を節単位の採否リストで提示する。
